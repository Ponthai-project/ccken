# 2026年度 三菱CC研究会 第３部会 研究の進め方 指南書

本資料は、研究会の運営経験や研究活動の経験が浅いメンバーでも、迷わずに価値のあるアウトプットを創出できるようにするための「研究の実践マニュアル（Markdown版）」です。どのようなステップで研究を進め、どの時期にどのような成果物が求められるのかを定義しています。

---

## 1. 研究の背景と目的

アジャイルの普及と生成AIの進化という、ソフトウェア開発の地殻変動を捉え、次世代の開発・運用モデルを模索します。

```text
【背景】
アジャイル開発の普及により「アウトカム・人間中心」への転換が進む一方、現場では「高度な自律的チーム」の維持が難しく、属人性や組織文化の壁により俊敏性を発揮しきれない「停滞感」が生じています。
この閉塞感を打破すべく、生成AIの進化を背景とした「AI-Ready」化と「AI駆動開発（特に仕様駆動開発）」という新たな潮流を融合させます。
```

### 🎯 研究の目的
アジャイルやウォーターフォールといった特定の手法論に依存せず、生成AIの進化を前提とした**「AI-Ready」化**と**「AI駆動開発」**を軸とした、開発から運用までを一貫して高度化できる**「次世代の開発・運用モデル」**を確立します。
単発のPoCや場当たり的なAIツール導入にとどまらず、**AIOps**を見据えたDevOpsの完成形に向けた、再現性のあるプロセス・成果物・運用指針を策定します。

## 2. 研究活動のコア・プロセス（4つのステップ）

すべての分科会は、以下の4つのステップを参考に研究を具体化・精緻化していきます。

### 🔍 Step 1：先行研究の徹底的な調査（最重要）
すでに世の中で検証済みのことを再検証する無駄を避けるため、活動の初期段階で「既存知見の境界線」を引きます。
特定の組織的・統計的インパクトレポート（DORA等）のみに偏ることを防ぐため、文献（書籍・論文等）を調べる際は、以下の4つの調査観点から情報を収集・整理してください。

#### 📝 文献調査における4つの観点
1. **実務プラクティス・開発者観点**: 開発者がAIツールをどう実務で使いこなすか（プロンプト設計、TDDとの融合、AIエディタでの実践知等）。
2. **アーキテクチャ・品質保証観点**: AIが生成したコードの品質保証、テスト自動化、レガシーコードの刷新、AIの自律動作を支えるシステム設計。
3. **プロダクトマネジメント・価値定義観点**: 「どれだけ早く作るか（量）」から「本当に必要な『何を作るか（価値）』」への移行、AIとの要求定義・仮説検証の共創手法。
4. **組織ガバナンス・プロセス移行観点**: 開発プロセスの移行（アジャイル/ウォーターフォールの補完関係）、セキュリティ規約、AI時代の評価や役割変化。

#### 💡 先行研究調査の具体的なヒント・情報源
*   **国内技術書籍（実務有名どころ）**:
    *   日本国内で出版されている、生成AIを活用したシステム開発、GitHub Copilotの実践解説書、およびアジャイル/スクラム開発とAIの融合に関する解説書籍。
*   **国内外の主要IT論文（技術雑誌や日経コンピューター連載等での紹介例）**:
    *   *『The SPACE of Developer Productivity』*（Microsoft等 / 2021年）: 開発者の生産性を多角的に測定する「SPACE」モデルを提唱した、世界で最も参照されている生産性論文。
    *   *『The Impact of AI on Developer Productivity: Evidence from GitHub Copilot』*（GitHub等 / 2023年）: AIによる開発速度向上（55.8%）をランダム化比較試験で初めて証明したマイルストーン的論文。
    *   *『Cognitive Load in AI-Assisted Programming』*: AI支援によってコード記述が速くなる反面、人間側のコード理解やレビューの「認知負荷（脳の疲労）」に与える影響を分析した研究。
    *   *技術雑誌・データベース*: 日経コンピューターの『世界で最も読まれているIT論文』連載記事、IEEE SoftwareやACMのデータベースなどの重要論文。
*   **過去のCC研研究レポートの確認**: アジャイル部会やクラウド関連部会の過去の成果物を事務局から取り寄せて確認。

#### 📝 先行研究整理マトリクス（テンプレート）
調査した情報は、各分科会で以下の表形式に整理してストックします。

| No | 調査ソース名・著者 | 要点・主要なファクト | その知見が示す限界・未解決の課題 | 我々の研究にどう活かせるか |
| :--- | :--- | :--- | :--- | :--- |
| 1 | 例: DORA Report 2025 | AIは既存組織の強みも弱みも増幅する。テストが弱い組織ではAIがバグを増幅する。 | 個人の生産性は上がるが、組織全体のデリバリーパフォーマンスが落ちる「パラドックス」の具体的解決策が未提示。 | 組織が「AI-Ready」になるための具体的なプロセスと評価指標の策定に活かす。 |
| 2 | （各自追記） | | | |

### 🎯 Step 2：想定される「理想形（To-Be）」の定義
調査を踏まえ、自分たちが対象とするテーマにおける「AI時代の理想の開発・運用のあり方（To-Be）」をモデルとして仮説定義します。

**理想形を定義するための8つの観点:**
1. **価値定義・企画（「何を作るか」の探求）**: AIによるコード生成の高速化に伴い、開発のボトルネックは「どれだけ早く作るか（量・スピード）」から「本当に顧客や組織に必要な『何を作るか（価値）』」へと移行しています。AIへのインプットとなる製品のコンセプト、要求定義、仮説検証のサイクルをどう設計し、AIと人間の共創によって価値を最大化するかの観点。
2. **プロセス（アジャイルとウォーターフォールの補完的フロー）**: AI駆動開発は、繰り返しのイテレーションや高速なリリース（アジャイル的アプローチ）に極めて親和性が高いですが、一方でAIが迷わず正しく動くための明確な構造化ドキュメントやコンテキスト、チェックゲート（ウォーターフォール的アプローチ）も重要です。双方が対立するのではなく、計画性と柔軟性をどのように融合し補完し合うプロセスを構築するかという観点。
3. **成果物・ドキュメント**: AIが正しく読み込み、実装やテストコードを自動生成するために必要十分なドキュメントの記述形式・粒度。
4. **品質保証（QA）**: AIが書いたコードの品質、セキュリティ、保守性を誰がどうやって保証するのか。
5. **組織・スキル**: AIを前提とした場合、開発者やPM、運用担当者の役割・ロールはどう変わるべきか。
6. **開発環境・プラットフォーム**: AIアシスタント/エージェントを安全かつ効率的に動作させるための環境要件（コンテキストリポジトリの整備、プロンプト・モデル管理基盤など）。
7. **ガバナンス・セキュリティ**: 送信データの静的・動的暗号化やプライバシー保護、生成物の著作権・ライセンス管理、およびAI出力に対する監査・ガバナンス体制。
8. **効果測定・継続的改善サイクル**: AI導入による生産性やソフトウェア品質の定量的・定性的な測定指標、およびAI向けルール・ガイドラインを継続的にアップデートするフィードバックループの構築。

### 🧪 Step 3：具体的な検証と課題抽出（PoC ＆ ヒアリング）

> [!NOTE]
> **【重要：分科会の自主的な進め方の尊重】**
> 以下に紹介するPoCの設計案やヒアリング設問例は、**各分科会が活動を具体化する上での「一例（参考案）」に過ぎません。**
> メンバーの自主性とオリジナリティを最も重視するため、本ステップのプロセスや項目に捉われることなく、独自の検証シナリオや調査アプローチを主体的に考案・選択して進めてください。

#### 1. PoC（概念実証・実験的開発）の設計（一例）
実際に生成AIやエージェントツールを使用し、小さな開発テーマで実験します。

```text
【PoC対象ツール】 (例: Claude Code, GitHub Copilot 等)
【検証シナリオ】 (例: 「日本語の基本設計書から、自動でWeb APIと単体テストコードを生成させる」)
【評価指標（定量・定性）】
  1. 生成されたコードのコンパイル・テスト通過率 (定量)
  2. 人間による手戻り・修正にかかった時間 (定量)
  3. AIにコンテキスト（文脈）を正しく伝えるためのドキュメント作成の難易度 (定性)
【検証結果と見つかったボトルネック】
  （例: 例外処理の記載がない場合、AIが不安定なコードを書くため、
   仕様書の『AI向け記述ルール』が必須であることが分かった、など）
```

#### 2. 他社事例ヒアリングの設計（一例）
三菱グループ各社や先進企業の担当者にヒアリングを行い、実務での共通課題（AIツールの適用フェーズ、組織的導入の障壁となるセキュリティ規約や文化、開発・運用プロセス変更の有無など）を整理します。

### 💡 Step 4：CC研としての「独自の視点・提言」の創出
研究報告の価値を決定づけるフェーズです。調査結果や検証データを「ただ並べただけ」の報告書で終わらせず、**「自分たちはどう考えるか」「今後エンタープライズ組織はどう行動すべきか」という独自の解釈や提言**を必ず盛り込みます。

**独自性を出すための具体的な切り口の例:**
* **「課題解決ロードマップや共通フレームワーク」の提案**: AI駆動開発やAIOpsを組織的に導入する際、エンタープライズ企業が段階的に直面する技術的・組織的障壁を乗り難くするための共通の手順やマイルストーンを定義する。
* **実務用「ガイドライン・チェックリスト」の策定**: 他社がすぐに使える実践的な成果物（例: 「AIが迷わずに実装できる設計書の記述チェックリスト」など）を作り、報告書の「別紙（アセットパッケージ）」として公開する。

## 3. 研究テーマの立ち上げプロセスと成果物の定義

### 🚀 研究テーマの立ち上げプロセス（OSTによる発散と収束）
研究内容やテーマは最初から固定しません。6月・7月の活動初期において、以下のステップを踏んでテーマを絞り込みます。
* **OST（Open Space Technology）による発散**: 各メンバーが持ち寄った問題意識や興味領域をもとに、その場で対話型ワークショップ（OST）を行い、アイデアや研究の種を大きく発散（Divergence）させます。
* **対話を通じた収束とテーマ決定**: 発散したアイデアをメンバー全員で議論し、共通の関心事や重要な論点をグループ化して収束（Convergence）させ、分科会テーマとして成立させます。
* **分科会の編成**: 全体として多角的にAI駆動開発の進化を検証するため、最終的に**「4つの分科会（4テーマ）程度」**を編成することを想定して議論を進めます。

### 📦 成果物フォーマット：中間報告書の構成とシナリオ（線）設計
中間報告（9月下旬提出、10/2発表）は、部会全体のビジョンと各分科会の活動計画を共有し、研究会としての軸を確立するための重要なマイルストーンです。
中間報告資料は、単に4つの分科会の進捗を「点」として並べるのではなく、それらが**「一本の線（シナリオ）」**でつながっているストーリーを描く必要があります。

#### 1. 中間報告資料の基本構成（推奨）
*   **① 研究会の目的と背景**: なぜ今、生成AIとアジャイルを融合させた「次世代の開発・運用モデル」が必要なのか、当部会が向き合う共通の課題意識を共有する。
*   **② 4つの分科会の編成と切り口の根拠**: どのような論理的根拠に基づいてテーマを4つに分けたのか、全体モデル（開発ライフサイクル等）との対応関係を示す。
*   **③ 分科会を「線」でつなぐストーリー（全体シナリオ）**: 各分科会の果たす役割が、開発から運用までの大きな流れの中でどのように連携し、価値を循環させていくかを明示する。各分科会の研究を個別の「点」としてバラバラに報告するのではなく、「あるテーマで定義された仕様（インプット）が別のテーマの開発プロセスで処理され、そこで得られたシステムがまた別のテーマのテスト・品質保証に渡り、最終的な運用での成果が再び上流にフィードバックされる」というように、**開発ライフサイクル全体のデータや成果物が一本の線で連動し、お互いの研究価値を高め合っているストーリー（全体最適のシナリオ）**を明確に提示してください。
*   **④ 各分科会の仮説と検証計画（As-Is/To-Be）**: 各分科会が定義した「理想形（To-Be）」の仮説と、それに対する現実（As-Is）の課題、PoCやヒアリングを通じた検証の進め方。**（※ 各分科会ごとに2スライド程度で簡潔にまとめる）**
*   **⑤ 今後の活動スケジュールと深堀り計画（下期の検証ロードマップ）**: 中間報告発表後から、検証（PoC・他社ヒアリング等）をどのように展開し、どう深く研究を展開していくかというロードマップ。および、最終報告書作成に至る具体的なマイルストーン。

> [!NOTE]
> **【中間報告資料の作成・提出フロー】**
> * 上記の構成のうち、**「①目的背景」「②分科会編成の根拠」「③線でつなぐストーリー（全体シナリオ）」「⑤今後のスケジュールと深堀り計画」**は、部会全体で統一された**【共通パート】**となります。
> * 各分科会は**「④各分科会の仮説と検証計画（各チーム2スライド程度）」**の作成に注力し、共通パートに必要な情報や意見のエッセンスを取りまとめて資料の**作成者（取りまとめ担当者）**へ提出するフローとします。

### 📦 成果物フォーマット：最終報告書の構成（二部構成）
最終報告書は、読者のニーズに応じて情報を最適に届けるため、以下の二部構成を基本とします。

* **① 本紙（各分科会ごとで4〜6スライド想定）**
  * 各分科会の結論やコアメッセージを凝縮したサマリースライド。
  * ※ 本紙の具体的なスライド構成やページ構成については、あらかじめ形を固定せず、分科会活動（PoCやヒアリング）の進捗や得られたファクトの重要度に応じて、**分科会活動の中で議論を重ねながら精緻化（ブラッシュアップ）させていくものとします**。
* **② 別紙（制限なし：アセットパッケージ）**
  * 実務で活用したい読者が、詳細なナレッジやデータにアクセスするためのパッケージ。
  * ※ 先行研究の整理シート、PoCで検証した詳細なソースコードやインプット設計書のテンプレート、ヒアリングの生記録、独自策定したチェックリストやガイドライン等の実制作物をすべて内包します。

## 4. 会議運営ルールと研究会マネジメント

部会（毎月第2・第4木曜日 15:00〜17:00）の時間を最大化するための運用ルールです。

### ⏱️ 会議時間（120分）のタイムマネジメント案
連絡事項はTeamsで事前に処理し、会議の場は「本質的な対話と議論」のみに時間を使います。
* **[00分〜05分] 状況共有**: 各自の進捗・今週のトピックスの共有（アイスブレイク）。
* **[05分〜65分] 分科会ディスカッション**: 分科会ごとの議論（先行研究の読み込み、PoC結果の分析、ヒアリング設計など）。
* **[65分〜85分] 分科会間の情報共有**: 他グループへの相談、進捗報告、フィードバック。
* **[85分〜110分] ドキュメンテーション**: 成果物の共同作成・Markdownでのドキュメンテーション作業。
