# CI/CDとAIOpsの整理(具体例つき)

一般的な情報をもとに、CI/CDとAIOpsの基本概念と具体例を整理する。[エンタープライズAI駆動開発_研究ポイント整理と分科会テーマ提案.md](./エンタープライズAI駆動開発_研究ポイント整理と分科会テーマ提案.md)の分科会③(統治構造・品質保証)・分科会④(AIOps・組織変革)を補足する資料。

## CI/CD

**やっていること**: コード変更を「自動でビルド・テスト・デプロイまで持っていく」パイプライン。

**具体例(GitHub Actions + AWSの場合)**:

| ステージ | 内容 | 例 |
|---|---|---|
| CI(継続的インテグレーション) | PRを出すたびに自動実行 | Lint(ESLint)→単体テスト(Jest)→セキュリティスキャン(Snyk/SAST)→ビルド |
| CD(継続的デリバリー) | CI通過後、自動でステージング環境にデプロイ | Dockerイメージをビルド→ステージングにデプロイ→E2Eテスト自動実行 |
| CD(継続的デプロイメント) | 人の承認なしに本番まで自動反映 | Blue/Greenデプロイでカナリアリリース→問題なければ自動で100%切替 |

エンタープライズだと「ステージングまでは全自動、本番は人が最終承認ボタンを押す」という**継続的デリバリー止まり**が多い(完全な継続的デプロイメントはリスクが高い業界だとまだ稀)。

**AI駆動開発との接続**: プロセス③「多層自動検証ゲート」がまさにCI/CDそのもの。AIエージェントが書いたコードをCI/CDパイプラインが機械的に検証する「司法AI」の実行基盤に当たる。

## AIOps

**やっていること**: 運用データ(ログ・メトリクス・トレース)にAI/MLを適用して、異常検知・原因分析・自動対応を行う。

**具体例**:

1. **異常検知**: Datadog/New RelicがAPIレイテンシの異常なスパイクを検知し、閾値超えで自動的にオンコール担当者にPagerDuty経由でアラート
2. **根本原因分析(RCA)**: AWS DevOps Guruが「Lambdaのエラー率上昇」を検知し、関連するCloudWatchメトリクス(DBコネクション枯渇など)を自動相関分析して原因候補を提示
3. **予測的スケーリング**: 過去のトラフィックパターンをMLで学習し、負荷が来る前にAuto Scalingで先回りしてインスタンスを増やす
4. **自律修復まで踏み込む例**: 障害検知→AIがログを分析→修正パッチ案とテストコードを自動生成→PRを自動作成→CI/CDが検証→人間が承認→自動デプロイ、という一連の流れ(この部分がまだ多くの企業で「検知・分析」止まりで、「修正パッチの自動生成」まで到達している組織は少ない)

**AI駆動開発との接続**: 「本番運用監視と自律修復・還流ループ」がこの4番目の例そのもの。分科会④「AIOps・組織変革」のテーマとして、CI/CDが分科会③の検証ゲート、AIOpsがこのループの運用側を担う、という役割分担になる。
