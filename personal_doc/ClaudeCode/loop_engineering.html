1ループとは何か（定義）

Claude Codeチームは、loopを次のように定義している。

"On the Claude Code team, we define loops as agents repeating cycles of work until a stop condition is met."
訳: 「私たちClaude Codeチームは、ループを、停止条件が満たされるまで作業のサイクルを繰り返すエージェントとして定義している」

重要なのは、この定義がタスクの中身（コーディングか調査か等）を一切問わないという点である。「何を、いつ、どうやって止めるか」という制御構造だけで4類型に分類される。記事は次の戒めから始まる。

"Not all tasks require complex loops; start with the simplest solution and use these patterns selectively."
すべてのタスクが複雑なループを必要とするわけではない。まずは最もシンプルな解決策から始め、これらのパターンは選択的に使うこと。

2通常の一問一答 vs ループ

記事はまず、日々のプロンプトのやり取りそのものが「手動のループ」であると説明する。

"Every prompt you send starts a manual loop with you directing each turn. Claude gathers context, takes action, checks its work, repeats if needed, and responds. We call this the agentic loop."
あなたが送るすべてのプロンプトは、各ターンをあなたが指揮する手動ループを開始する。Claudeはコンテキストを集め、行動し、自分の作業を確認し、必要なら繰り返し、応答する。これをエージェント的ループと呼ぶ。

4類型のグラデーションは、この「人間が毎ターン監督する状態」から「人間がループの外に出る状態」への移行として理解すると分かりやすい。

34類型の全体像（階段図）

4つの類型は、それぞれ「トリガー（何が始めるか）」「停止条件（何が止めるか）」「プリミティブ（Claude Codeのどの機能を使うか）」の3軸で整理できる。段を上るごとに人間のリアルタイム関与が減り、エージェントの自律性が上がる。

最適な用途単発の探索・意思決定作業
最適な用途検証可能な終了基準があるタスク
最適な用途反復作業／外部システムとの対話
最適な用途定型化された反復的な作業ストリーム

4手放すものが増えていく

4類型を貫く一本の軸は「人間が何を手放すか」である。Turn-basedでは「検証」を、Goal-basedでは追加で「停止条件」を、Time-basedでは追加で「トリガー」を、Proactiveでは最終的に「プロンプトそのもの」を手放す。

5類型①: Turn-based loop

🔵 Turn-based loop（通常のプロンプト + skills）

Triggered byA user prompt（ユーザーのプロンプト）
Stop criteriaClaude judges it has completed the task or needs additional context.（Claudeが「完了した」あるいは「追加のコンテキストが必要」と自己判断する）
Best used forShorter tasks that are not part of a regular process or schedule.（定型的な反復作業やスケジュールに属さない、短めのタスク）
Managed usage byWrite specific prompts and improve verification using skills to reduce the number of turns.（具体的なプロンプトを書き、skillsで検証の質を上げてターン数を減らす）

普段のClaude Codeとのやり取りそのものが、実は最小単位のループである。

"For example, ask Claude to create a like button. It reads your code, makes the edit, runs the tests, and hands back something it believes works. You then manually check the work, and write the next prompt."
例えば、Claudeに「いいねボタン」を作らせたとする。Claudeはコードを読み、編集を加え、テストを実行し、動作すると信じているものを返す。そこで人間が手動で作業内容を確認し、次のプロンプトを書く。
"You can improve the verification step by encoding your manual steps as a SKILL.md so Claude can check more of its own work, end-to-end. This should include tools or connectors to allow Claude to see, measure or interact with the result. The more quantitative the checks are, the easier it is for Claude to self-verify."
人間が手動で行っていた確認手順をSKILL.mdとして符号化することで、Claudeが自分の作業をより多く、end-to-endで確認できるようになる。これには、Claudeが結果を見る・測る・操作するためのツールやコネクタを含めるべきである。チェックが定量的であるほど、Claudeにとって自己検証がしやすくなる。

記事に掲載されているSKILL.mdの例（フロントエンド変更の検証用）:

---
name: verify-frontend-change
description: Verify any UI change end-to-end before declaring it done.
---

# Verifying frontend changes
Never report a UI change as complete based on a successful edit alone. Verify it the way a human reviewer would:

1. Start the dev server and open the edited page in the browser.

2. Interact with the change directly. For a new control (button, input, toggle): click it, confirm the expected state change, and screenshot before/after.

3. Check the browser console: zero new errors or warnings.

4. Use the Chrome Devtools MCP, run a performance trace and audit Core Web Vitals.

If any step fails, fix the issue and rerun from step 1 — do not hand back partially verified work.

（日本語注釈: 「編集が成功した」だけでUI変更を完了報告しない。①devサーバーを起動して該当ページを開く ②実際に操作し変更前後をスクリーンショット比較 ③コンソールに新規エラー/警告がないか確認 ④Chrome Devtools MCPでパフォーマンス計測とCore Web Vitals監査。いずれかが失敗したら手順1からやり直し、中途半端な状態で作業を返さない。）

6類型②: Goal-based loop

🟢 Goal-based loop（プリミティブ: /goal）

Triggered byA manual prompt in real-time.（リアルタイムでの手動プロンプト、1回）
Stop criteriaGoal achieved OR maximum number of turns reached.（ゴール達成、または最大ターン数への到達）
Best used forTasks that have verifiable exit criteria.（検証可能な終了基準を持つタスク）
Managed usage bySetting a specific completion criteria and explicit turn caps, "stop after 5 tries."（具体的な完了基準の設定と、明示的なターン上限「5回試して止める」など）

Goal-basedループの核心は、「良し悪しの判断」をClaude自身から切り離す点にある。

"When you define the success criteria, Claude doesn't have to make a determination on what is 'good enough' and end the loop early. Each time Claude tries to stop, an evaluator model checks your condition and sends it back to work until the goal is met or a number of turns you define is reached."
成功基準を定義すれば、Claudeは何が「十分に良い」かを自分で判断してループを早々に終わらせる必要がなくなる。Claudeが停止しようとするたびに、評価モデル（evaluator）があなたの条件をチェックし、達成していなければ作業に送り返す。これがゴールを達成するか、指定したターン数に達するまで続く。
"This is why deterministic criteria, such as number of tests passed or clearing a certain score threshold, are so effective."
だからこそ、通過したテスト数や、特定のスコア閾値のクリアといった決定論的な基準が、非常に効果的なのである。
/goal get the homepage Lighthouse score to 90 or above, stop after 5 tries.

（訳: ホームページのLighthouseスコアを90以上にせよ。5回試したら止める。）

📘 公式ドキュメント補足: /goal の内部機構
  • /goalの実体は、session-scopedな prompt-based Stop hook。
  • Claudeが停止しようとするたびに「条件＋会話履歴」が小さく高速なモデル（既定 Haiku）に送られ、yes/noの判定を返す。
  • evaluatorはツールを呼び出さない。会話に既に現れている事実しか判定材料にできないため、条件は「Claudeの出力から証明できる形」で書く必要がある。
  • 条件は最大4,000文字。アクティブなゴールは1セッションにつき1つ。
  • /goal単体で現在の状態確認、/goal clearで解除できる。
  • v2.1.139以降で利用可能。

7類型③: Time-based loop

🟠 Time-based loop（プリミティブ: /loop, /schedule）

Triggered byA specified time interval.（指定された時間間隔）
Stop criteriaYou cancel it, or the work completes (the PR merges, the queue is empty).（人が止める、あるいはPRのマージやキューの空など作業自体の完了）
Best used forFor recurring work, or interfacing with external environments / systems.（反復的な作業、または外部環境・システムとのやり取り）
Managed usage bySet longer intervals or react based on events rather than time.（間隔を長めに設定するか、時間ではなくイベントに応じて反応させる）
"Some agentic work is recurring: the task stays the same and only the inputs change. For example, summarizing Slack messages every morning. Other work depends on external systems, and a simple way to interface with one is to check it on an interval and react to what changed. For example, a PR which may receive code reviews or fail CI."
エージェント的な作業の中には反復的なもの — タスク自体は変わらず入力だけが変わる — がある。例えば毎朝Slackメッセージを要約すること。他方、外部システムに依存する作業もあり、それに対応する簡単な方法は、一定間隔でチェックし、変化に反応することである。例えばコードレビューを受けたりCIが失敗したりするPR。
"/loop runs on your computer, so if you turn it off, it stops. You can move the loop to the cloud by creating a routine with /schedule."
/loopはあなたのコンピュータ上で動くため、電源を切れば止まる。/scheduleでルーチンを作成すれば、ループをクラウドに移行できる。
/loop 5m check my PR, address review comments, and fix failing CI

（訳: 5分ごとに自分のPRを確認し、レビューコメントに対応し、失敗しているCIを修正せよ。）

📘 公式ドキュメント補足: /loop の実務仕様（記事には未記載）
  • 要 Claude Code v2.1.72以降。/loopはbundled skill。
  • タスクはsession-scoped（新しい会話を始めると消える。--resume / --continueで未期限のものは復元される）。
与えるもの例挙動
インターバル + プロンプト/loop 5m check the deploy固定スケジュールで実行
プロンプトのみ/loop check the deployClaudeが毎イテレーションの間隔を自分で決める
インターバルのみ / 何もなし/loop組み込みメンテナンスプロンプト、またはloop.mdがあればそれが走る
  • 単位は s / m / h / d。秒はcronの1分粒度に切り上げ、綺麗なcronステップにならない値は丸められる。
  • slash command併用可: /loop 20m /review-pr 1234（ただし自律起動が許可されたskillのみ実行される。/model等の組み込みコマンドやMCP promptはただのテキストとして届く）。
  • dynamic mode（インターバル省略時）: Claudeが各イテレーション後に1分〜1時間の遅延を自ら選ぶ自己ペーシング。ビルド中やPRがアクティブなら短く、静かなら長く。選んだ遅延と理由が毎回出力される。
  • loop.md: .claude/loop.md（プロジェクト優先）/ ~/.claude/loop.md（ユーザー）で既定プロンプトを差し替え可能。実行中の編集も次イテレーションから反映。
  • 停止: 待機中にEscでpending wakeupをクリア。dynamic modeではClaude自身がScheduleWakeup(stop: true)で終了できる。
  • Jitter: 全セッションが同時にAPIを叩かぬよう、発火時刻を最大30分ずらす。厳密な時刻運用は不可。
  • 7日expiry: 繰り返しタスクは作成から7日で自動失効（「忘れられたループ」の暴走を時間で縛る設計）。
  • 1セッション最大50タスク。CLAUDE_CODE_DISABLE_CRON=1で無効化。

8類型④: Proactive loop

🟣 Proactive loop（合成 + auto mode + dynamic workflows）

Triggered byAn event or schedule, with no human in real time.（イベントまたはスケジュール。リアルタイムでの人間の介在なし）
Stop criteriaEach task exits when its goal is met. The routine itself runs until you turn it off.（各タスクはゴール達成で終了する。ルーチン自体は停止するまで動き続ける）
Best used forRecurring streams of well-defined work: bug reports, issue triage, migrations, dependency upgrades, etc.（バグ報告、issueのトリアージ、マイグレーション、依存関係のアップグレードなど、定義済みの反復作業の連続）
Managed usage byRouting routines to smaller, faster models and using the most capable model for judgment calls.（ルーチンは小さく高速なモデルへ、判断が必要な場面には最も高性能なモデルを使う）
"The primitives above, along with other Claude Code features like auto mode and dynamic workflows (research preview) can be composed into a loop for long-running work."
これまでのプリミティブに加え、auto modeやdynamic workflows（研究プレビュー）といった他のClaude Code機能を組み合わせることで、長時間稼働するループを構成できる。

記事はフィードバック対応を例に、次の4ステップの合成レシピを示す。

  1. /schedule（研究プレビュー）で、新しい報告がないか確認するルーチンを実行
  2. /goalで「完了」の定義を設定し、skillsでその検証方法を文書化
  3. Dynamic workflowsで、各報告をトリアージ・修正・レビューするエージェントをオーケストレーション
  4. Auto modeで、許可を求めて止まることなくルーチンを実行

これらを1本に束ねたのが、記事の目玉プロンプトである。

/schedule every hour: check #project-feedback for bug reports. /goal: don't stop until every report found this run is triaged, actioned, and responded to. When fixing a bug, use a workflow to explore three solutions in parallel worktrees and have a judge adversarially review them.

（訳: 1時間ごとに#project-feedbackチャンネルのバグ報告を確認せよ。今回のスキャンで見つかったすべての報告がトリアージ・対応・返信されるまで止めるな。バグを修正する際は、3つの解決案を並列のworktreeで探索するワークフローを使い、judgeに敵対的レビューをさせよ。）

9品質を支える4本の柱

"The quality of a loop's output depends on the system around it."
ループの出力品質は、その周りのシステムに依存する。
  • Keep the codebase itself clean — Claude follows patterns and conventions that already exist in your codebase.（コードベース自体を綺麗に保つ。Claudeは既存のパターンや規約に従うため。）
  • Give Claude a way to verify its own work — Encode what good looks like for you and your team with skills.（Claudeに自分の作業を検証する手段を与える。あなたやチームにとっての「良い状態」をskillsとして符号化する。）
  • Make docs easy to reach — Frameworks and libraries docs have up-to-date best practices.（ドキュメントに手が届きやすくする。フレームワーク/ライブラリのドキュメントには最新のベストプラクティスがある。）
  • Use a second agent for code reviews — A reviewer with fresh context is less biased and not influenced by the main agent's reasoning.（コードレビューには2つ目のエージェントを使う。フレッシュなコンテキストを持つレビュアーはバイアスが少なく、主エージェントの推論に影響されない。組み込みの/code-reviewskillやGitHub向けCode Reviewが該当。）

この記事で最も思想的に重要な一文:

"When an individual result doesn't meet the standard, don't stop at fixing the individual issue, try to encode it to improve the system for all future iterations."
訳: 個々の結果が基準を満たさなかったとき、その個別の問題を直して終わりにせず、以後のすべてのイテレーションでシステムが改善されるよう、それを符号化せよ。

10トークン使用量の管理

"To manage token usage, loops should have clear boundaries:"
トークン使用量を管理するには、ループに明確な境界を持たせるべきである。
  • Choose the right primitive and model for the job（適切なプリミティブとモデルを選ぶ） — Smaller tasks don't need multiple agents or loops.（小さなタスクに複数エージェントやループは不要。）
  • Define clear success and stop criteria（明確な成功・停止基準を定義する） — Be specific about what done looks like so Claude can arrive at the solution sooner (but not too soon).（「完了」の姿を具体的にし、Claudeが早すぎず・遅すぎず解決に到達できるようにする。）
  • Pilot before a large run（大規模実行の前に試行する） — Dynamic workflows can spawn hundreds of agents. Gauge usage on a smaller slice of the work first.（Dynamic workflowsは数百のエージェントを生成し得る。まず作業の一部で使用量を見積もる。）
  • Use scripts for deterministic work（決定論的な作業にはスクリプトを使う） — Running a script is cheaper than reasoning through the steps.（スクリプトの実行は、手順を推論するより安価。例: PDF skillはフォーム記入スクリプトを同梱し、毎回コードを再導出させない。）
  • Don't run routines more often than you need to（必要以上の頻度でルーチンを回さない） — Match the interval to how often the thing you're watching changes.（監視対象の変化頻度にインターバルを合わせる。）
  • Review usage（使用量を確認する） — /usage（skills/subagents/MCP別の使用量）、/goal（現時点のターン数とトークン使用量）、/workflows（各エージェントのトークン使用量、任意のタイミングで停止可）。
"Your model and effort level choices are among the biggest levers on what a loop costs."
モデルとeffortレベルの選択は、ループのコストを左右する最大級のレバーの一つである。
