## Part 1. 研究すべき概要

### 背景・課題認識

- ADR・Wiki・設計書がいくら「書かれている」状態でも、AIエージェントがそれを参照せず、違反しても止まらないなら、AI駆動開発の土台としては機能しない（ビズリーチ資料：「書かれている」と「効いている」は別物）。
- エンタープライズの現実は「Excel/自然言語の罠」——曖昧な日本語仕様書・ポンチ絵・属人化した暗黙知が常態化しており、AIに与える前提コンテキストが整備されていない（成熟度モデル レベル1の壁）。
- DORA Report 2025が示す通り、AIは既存組織の強みも弱みも増幅する。土台（インプットとプラットフォーム）が弱い組織では、AI導入はむしろ混乱を増幅する。

### 中核の問い

1. **AIが解釈・検証可能な「実行可能仕様（要求ハーネス）」の記述標準**とは何か。Given-When-Then、Markdown構造化、データスキーマ等を、エンタープライズの実務（既存のExcel設計書文化）からどう移行させるか。
2. **コンテキストハーネス（用語集・ドメイン知識・ADR・システム構成）**をどう構造化し、SSOT（人が承認した正典）と派生データ（機械生成物）を分離して管理するか。
3. AIエージェントを安全に動かす**プラットフォーム要件**（モデル・プロンプト管理、MCP等のツール接続、サンドボックス実行環境、コスト按分・利用統制、データ暗号化）は何か。
4. 要求定義→開発→検証→運用監視を手動転記なしで直結する**「一本のデータ・パイプライン」の基盤アーキテクチャ**はどうあるべきか（AIOpsの還流データが流れる土台を含む）。

### 研究スコープ

- **In**: AI可読ドキュメントの記述標準／SSOT設計・Authority Provenance Graphの「立法」側／コンテキストリポジトリの構造／AI開発プラットフォームのリファレンスアーキテクチャ（セキュリティ基盤・コスト管理を含む）／既存ドキュメント資産のAI可読化（リバースドキュメンテーション）
- **Out（他分科会の主担当）**: 仕様を作る工程・フロー設計（→②）、検証ゲート・テストハーネス（→③）、仕様を書ける人材育成（→④）

### 先行研究の起点

- ビズリーチ: 立法AI（ルールのSSOT）、SSOTと派生データの分離原則、Authority/Specification Provenance Graph
- AWS AI-DLC: Steering Rules（AIの文脈維持の仕組み化）、承認ゲートを支えるコンテキスト設計
- クラスメソッド: Devin Knowledge/Cursor Ruleの整備実務、`code2prompt`＋長コンテキストLLMによる既存コードからの設計書逆生成、Bedrockの推論プロファイルによるコスト按分・リージョン統制
- 一般動向: GitHub Spec Kit・AWS Kiro等の仕様管理形式、MCP（Model Context Protocol)によるコンテキスト供給の標準化、コンテキストエンジニアリングに関する各社プラクティス

## Part 2. インセプションデッキ

### 1. 我々はなぜここにいるのか

AI駆動開発の失敗の大半は、AIツールの性能ではなく「AIに読ませるものが無い・汚い・散らばっている」ことに起因する。エンタープライズがAI-Readyになるための**インプット（仕様・コンテキスト）とプラットフォームの整備標準**を明らかにするため。

### 2. エレベーターピッチ

> **曖昧な仕様書と属人知に阻まれてAI導入が進まない** エンタープライズ組織向けの、**「AI-Ready基盤ガイドライン」** という研究成果は、**AIが解釈できる仕様・コンテキスト・実行環境の整備標準** です。これは **既存のExcel/自然言語資産からの現実的な移行パス** を提供でき、**ツールベンダーの個別ドキュメントとは違って**、**特定製品に依存しない「立法（SSOT）設計の原則」** が備わっています。

### 3. パッケージデザイン（最終成果物イメージ）

- **本紙**: AI-Ready基盤の全体像（インプット層・プラットフォーム層のリファレンスモデル）と導入ステップ
- **別紙（アセットパッケージ）**:
  - AI可読仕様書の記述ガイドライン＋テンプレート（Given-When-Then、用語集、データスキーマ）
  - AI-Ready成熟度セルフチェックリスト（「書かれている」→「効いている」の診断）
  - コンテキストリポジトリの構造サンプル（SSOT/派生データ分離の実例）
  - プラットフォーム要件一覧（セキュリティ・コスト統制・モデル管理の考慮点）

### 4. やらないことリスト

- ❌ 開発工程・スプリント運営の設計（→分科会②）
- ❌ CI/CD検証ゲート・テストハーネスの設計と実機検証（→分科会③）
- ❌ 人材育成・ロール定義・研修設計（→分科会④）
- ❌ 特定ベンダー製品の網羅的な機能比較（原則・標準の抽出に留める）

### 5. ご近所さん（連携先）

- **分科会②**: 我々が定義した仕様・コンテキストの記述標準を、②がプロセスに組み込んで使う（最重要のインターフェース）
- **分科会③**: SSOT（ルール）と検証チェックの紐付け方（Authority Provenance Graph）は、立法側＝①、司法側＝③で分担して共同検討
- **分科会④**: 基盤整備の担い手（誰がSSOTを維持するか）の役割定義を④に依頼
- **外部**: 三菱グループ各社へのAI-Ready実態ヒアリング、ビズリーチ・クラスメソッド等の公開情報

### 6. 解決案（研究アプローチ）

1. 先行研究調査: 仕様記述形式（Spec Kit/Kiro/BDD等）とコンテキスト管理手法の比較整理（8〜9月）
2. To-Be定義: インプット層＋プラットフォーム層のリファレンスモデル仮説を策定（9月・中間報告）
3. 検証: サンプル題材で「Excel設計書 → AI可読仕様への変換」を実際に行い、変換コストと効果（AIの生成精度差）を測定するPoC（10〜11月）
4. 提言: 成熟度チェックリストと移行ガイドラインに落とし込む（12月〜）

### 7. 夜も眠れない問題（リスク）

- 「AI可読化」の効果測定が難しく、PoCが定性的な感想で終わるリスク → 分科会③の検証環境を借りて「同じAI・同じタスクでインプット品質だけ変えた比較」を設計する
- 仕様記述標準が乱立・変化しており（Spec Kit等は発展途上）、年度末には推奨が陳腐化するリスク → 特定形式ではなく「満たすべき性質」で標準を記述する
- プラットフォーム要件が各社のセキュリティポリシー依存で一般化しにくいリスク → ヒアリングで共通項を抽出する

### 8. 期間を見極める

全体マイルストーン（7月デッキ確定→9月中間報告→10-11月PoC→3月最終報告）に準拠。PoCは11月末までに完了必須。

### 10. 何がどれだけ必要か

- メンバー: 上流設計・ドキュメント標準に関心がある者＋クラウド基盤に知見がある者の混成が望ましい
- 検証環境: 分科会③と共用のサンプルリポジトリ＋生成AIツール（Claude Code / Copilot 等）
- 活動時間: 部会内ディスカッション＋各自の調査・PoC作業（宿題ベース）
