## Part 1. 研究すべき概要

### 背景・課題認識

- AIのコード生成量増加は「レビューの渋滞」という新たな品質ボトルネックを生む。人間が一行ずつ読む従来型レビューは認知限界に達しており、品質担保は**「人間が頑張って読む」から「機械的に検知・停止できる構造」への転換**が不可避（`AI開発のToBe`1節、品質整理資料）。
- DORA Report 2025の指摘の通り、テストが弱い組織ではAIがバグを増幅する。AI駆動の品質保証は「AIを使う前に安全網（ハーネス）があるか」で成否が決まる。
- ハーネスエンジニアリングは世界的潮流（Mitchell Hashimoto「Engineer the Harness」、OpenAI「Harness engineering」= 3名・5ヶ月・100万行・ゼロ手書きの事例）だが、**エンタープライズの実務でどのハーネス（の組み合わせ）が有効かを比較実証した知見はまだ乏しい**。ここが当分科会の独自性の核になる。

### 中核の問い

1. **AI生成コードの品質・セキュリティ・保守性を、誰が（何が）どうやって保証するのか。** セマンティックチェック（LLMによる意味判定）と決定性チェック（Grep/AST/Linter等の機械判定）の役割分担設計。
2. **多層自動検証ゲート**（①要求整合性BDD/TDD ②セキュリティSAST/DAST ③品質・規約Linter ④インフラIaC検証 ⑤サンドボックス起動テスト）の実務的な構成・しきい値・運用は何が最適か。
3. **既存システム改修の品質防衛**: 仕様が暗黙知化したレガシーの正しい挙動をテストハーネスで囲い込み、デグレを防ぐ現実的な手順は何か。
4. **「書かれている」から「効いている」への転換**: ルール（立法）とチェック（司法）を機械可読に接続し（Authority Provenance Graph）、「立法なき司法」「司法なき立法」を検知する仕組みは実装可能か。
5. **世の中のハーネス実装を実機で試すと、実際どれ（どの組み合わせ）が効くのか。**（→後述の実機検証）

### 研究スコープ

- **In**: 多層検証ゲートの設計／AIコードレビュー・テスト自動生成のベストプラクティス／レガシー囲い込みテストハーネス／品質指標（デグレ数・欠陥検出率・偽陽性率・カバレッジ）／セキュリティ検証ゲート／**ハーネスの実機比較検証**
- **Out（他分科会の主担当）**: 仕様・ルールそのものの記述標準＝立法側（→①）、ゲートをプロセスのどこに置くかの工程設計（→②）、レビュアー・QA人材の役割変化（→④）

### 先行研究の起点

- ビズリーチ: 司法AI（セマンティック＋決定性）、Authority/Specification Provenance Graph、SSOTと派生データの分離
- クラスメソッド: Copilotによるテスト生成→デシジョンテーブル→追加生成のワークフロー、CodeRabbit/PR-Agentの導入実務
- 世の中のベストプラクティス: TDD/BDD、ミューテーションテスト（テストの強度測定）、プロパティベーステスト、LLM-as-a-Judge、SWE-bench等のエージェント評価、シフトレフトセキュリティ（SAST/DAST/SCA）

## Part 2. ハーネス実機検証の計画（当分科会の独自活動）

世の中のベストプラクティスを取り込んだ上で、**実際にハーネス（の組み合わせ）を構築してAIエージェントに開発させ、「この構成が良い」と言える実証データを取る**。

### 検証の基本設計

```
【共通題材】 仕様書つきのサンプルアプリケーション（分科会①②と共通リポジトリ）
【共通タスク】 (a)新機能追加 (b)既存機能の改修（意図的にデグレを誘発しやすい題材）(c)脆弱性を含む修正依頼
【変える変数】 ハーネス構成（下記候補の組み合わせ）
【固定する変数】 AIエージェント・モデル・仕様書の品質（①の標準に準拠）
【測る指標】
  1. デグレ検出率／すり抜け率（仕込んだ欠陥をゲートが止められたか）
  2. 偽陽性率（正しいコードを誤って止めた率）とノイズ量
  3. AIの自己修正ループ完走率（人間介入なしでゲート合格まで到達したか）
  4. フィードバック速度（コミット→検証結果までの時間）
  5. 構築・維持コスト（セットアップ工数、LLM推論コスト）
```

### 検証候補となるハーネス要素（組み合わせて構成する）

| レイヤ | 候補例 |
|---|---|
| 要求整合性（仕様↔テスト） | BDDフレームワーク（Cucumber等）、GitHub Spec Kit、AWS Kiro的な仕様駆動形式 |
| エージェント制御 | Claude Code（hooks・カスタムコマンド・CLAUDE.md）、Cursor Rules、Devin Knowledge、Codex |
| AIコードレビュー | CodeRabbit、PR-Agent（Qodo）、Copilot Code Review |
| 決定性チェック | Linter/Formatter、AST検査、Danger、依存・ライセンスチェック |
| セキュリティ | SAST（Semgrep/CodeQL）、SCA（Snyk/Dependabot）、IaCスキャン（tfsec/cfn-lint） |
| テスト強度 | カバレッジ、ミューテーションテスト（PIT/Stryker）、プロパティベーステスト |
| 実行ハーネス | コンテナサンドボックス起動テスト、E2E自動テスト、エフェメラル環境 |

### ハーネスを考える上で重要な点（設計・選定の評価軸）

1. **多層防御と実行順序**: 安く速い決定性チェック（Linter/AST/ビルド）を前段に、高価で曖昧さを扱えるセマンティックチェック（LLMレビュー）を後段に置く。1つの万能ゲートではなく、層ごとに責務を分ける。
2. **フィードバック速度**: AIの自己修正ループは分単位で回る。ゲートが遅いとAIの速度を殺し、HITLの渋滞が「CIの渋滞」に変わるだけになる。
3. **偽陽性率の管理**: ノイズの多いゲートは人間にもAIにも無視され、形骸化する。「止める基準」の厳選と定期的なチューニングが不可欠。
4. **仕様（立法）とのトレーサビリティ**: 各チェックが「どのルール・仕様に基づくか」を機械可読に紐付ける（Authority Provenance Graph）。根拠のないチェック（立法なき司法）と、検証されないルール（司法なき立法）を検知できる構造にする。
5. **決定性チェック優先の原則**: LLMで判定できることでも、Grep/ASTで機械的に白黒つけられるなら決定性チェックに落とす（安定・高速・安価・監査可能）。LLM判定は「意味理解が本当に必要な検証」に限定する。
6. **サンドボックス隔離と権限設計**: AIに強い権限を与えるほどハーネスの隔離が重要になる（クラスメソッドのPermissions Boundary検証が参考）。本番影響の遮断、ネットワーク・データアクセスの最小化。
7. **観測性・監査ログ**: 何が・なぜ止まった/通ったかの記録は、エンタープライズの監査要件そのもの。AIの思考トレースとゲート判定履歴を残す。
8. **ツール中立性・可搬性**: 特定エージェント製品にロックインされない構成（CI層にゲートを置けばエージェントを差し替えても効く）。harnessは「借りられる」領域だからこそ、乗り換え可能性を担保する。
9. **既存資産への適用容易性**: グリーンフィールドで効くハーネスがレガシーには構築困難なことが多い。「囲い込みテストを後付けするコスト」を評価軸に含める。
10. **ハーネス自体の保守性**: ルール・ゲートは腐る。更新フロー（誰がいつ改訂するか）と、ゲート自体の実効性測定（デグレすり抜け率の継続監視）をセットで設計する。

## Part 3. インセプションデッキ

### 1. 我々はなぜここにいるのか

AI駆動開発の最後の砦は品質保証である。人間の目視レビューが認知限界に達する中、**「機械的に検知・停止できる安全網（ハーネス）」の実務的な最適解を、実機検証の実証データつきで示す**ため。

### 2. エレベーターピッチ

> **AIが生成する大量のコードを品質保証しきれない** エンタープライズQA・開発組織向けの、**「AI駆動品質保証ハーネス実証レポート」** という研究成果は、**多層自動検証ゲートの設計指針と実機比較データ** です。これは **「どのハーネス構成が実際にデグレを止め、どれがノイズになるか」の実証知見** を提供でき、**ツール紹介記事や理論的なフレームワーク論とは違って**、**同一条件での実機比較と評価軸（10の重要点）に基づく推奨構成** が備わっています。

### 3. パッケージデザイン（最終成果物イメージ）

- **本紙**: 多層検証ハーネスの推奨構成（実証データつき）と品質指標の転換（デグレ数を品質担保の実効性メトリクスとする測定体系）
- **別紙（アセットパッケージ）**:
  - ハーネス実機検証レポート（検証シナリオ・測定データ・構成別の評価）
  - ハーネス設計ガイドライン（上記「重要な点」10項目の詳説＋チェックリスト）
  - 検証に使った題材リポジトリ・CI設定・ゲート定義の実物（再現可能なサンプル）
  - レガシー囲い込みテストハーネス構築手順書
  - 品質自己診断チェックリスト（「効いているか」3つの問い）

### 4. やらないことリスト

- ❌ 仕様書・ルールの記述標準そのものの策定（→分科会①。当分科会は①の標準を検証の入力として使う）
- ❌ 開発プロセス全体の工程設計・ゲート配置の決定（→分科会②）
- ❌ QAエンジニア・レビュアーの役割変化、ハーネスエンジニアの育成論（→分科会④）
- ❌ 全ツールの網羅的ベンチマーク（評価軸に基づき代表的な組み合わせ2〜3構成に絞って深く検証する）

### 5. ご近所さん（連携先）

- **分科会①**: 検証の入力となる仕様・コンテキストは①の記述標準に準拠（インプット品質を固定するため）。Authority Provenance Graphは立法側①・司法側③で共同検討
- **分科会②**: 題材リポジトリを共用。②はプロセス効率を、当分科会は品質実効性を測る（指標の棲み分け）
- **分科会④**: ハーネス運用に必要な新ロール（ハーネスエンジニア）の要件を④へインプット
- **外部**: CodeRabbit/PR-Agent等の導入企業事例、OpenAI/Anthropicのharness関連公開情報、グループ各社QA部門ヒアリング

### 6. 解決案（研究アプローチ）

1. 先行研究調査: 世の中の品質保証ベストプラクティス（TDD/BDD・ミューテーションテスト・LLMレビュー等）とハーネス事例の整理、評価軸（重要な点10項目）の精緻化（8〜9月）
2. 検証設計: 題材リポジトリ・欠陥シナリオ・測定指標を確定し、比較する2〜3構成を選定（9月・中間報告に検証計画を含める）
3. 実機検証: 構成別にAIエージェントへ同一タスクを実行させ、デグレ検出率・偽陽性率・完走率・速度・コストを測定（10〜11月）
4. 提言: 実証データに基づく推奨構成＋設計ガイドラインとして集約（12月〜）

### 7. 夜も眠れない問題（リスク）

- 検証環境構築（CI・サンドボックス・複数ツールのセットアップ）に想定以上の工数がかかり、測定まで到達しないリスク → 9月中に最小構成（1レイヤだけ）で試走し、実現可能な検証規模を早期に見極める
- ツールのバージョンアップで検証結果がすぐ陳腐化するリスク → 個別ツールの優劣ではなく「構成パターンと評価軸」を主たる成果として位置づける
- 欠陥シナリオの作り込みが甘く、どの構成でも全部止まる/全部すり抜ける（差が出ない）リスク → 難易度を段階化した欠陥セット（自明なLintレベル〜仕様理解が必要な意味レベル）を設計する
- LLM推論コスト・ライセンス費が各社の予算制約を超えるリスク → 無償枠・トライアルの範囲で収まる検証規模を優先する

### 8. 期間を見極める

全体マイルストーンに準拠。**環境構築は9月中に着手**（他分科会よりリードタイムが長いため前倒し）、実機検証の測定は11月末までに完了。

### 10. 何がどれだけ必要か

- メンバー: CI/CD・テスト自動化の実務経験者を最低1名、セキュリティ検証に土地勘のある者1名を含めたい
- 検証環境: GitHub（Actions）ベースの共通題材リポジトリ、AIエージェント（Claude Code / Copilot等）、AIレビューツール（CodeRabbit等の無償枠）、コンテナ実行環境
- 予算・調達: 各社で利用可能なツールライセンス・API利用枠の棚卸しを8月中に実施
