## 1. コンテキストエンジニアリング

### 問題提起：コンテキストロット（腐敗）

- すべてのLLMは入力長・プロンプト長に応じて性能が劣化する。
- 出典として Chroma Research (2025) の18モデル評価が挙げられ、Claude Sonnet 4、GPT-4.1、Gemini 2.5 Flash、Qwen3-32B を含む全18モデルで劣化を確認したとされる。
- 問題はコンテキストウィンドウの「容量」ではなく「ノイズの蓄積」であり、ウィンドウに余裕があっても早い段階から劣化が始まると説明されている。

### 社内アンケート結果

- AWS Japan の SA ロールを対象に「何トークンから精度の壁を感じるか」をアンケート。
- 結果として、Opus 4.7/4.6（最大コンテキストウィンドウ1Mトークン）でも**400Kトークン程度で劣化を感じる**という体感が示された（ベンチマーク値ではなく体感ベースである旨の注記あり）。

### コーディングエージェントの振る舞い

- コンテキストウィンドウには System Prompt、AGENTS.md、Builtin/MCP Tools、User/Assistant message などが積み上がっていく構造が図解されている。
- ウィンドウ内を「Smart Zone（精度良くコーディングできる領域）」と「Dumb Zone（精度が劣化する領域）」に分け、目安として **400Kトークン**が経験則の境界線とされる。
- ノイズとして取り込まれる要素の例：
  - コードベース調査で読み込んだファイル
  - Lint / Build / Test / Deploy の実行ログ
  - 実装したソフトウェアの起動・実行・操作ログ
  - MCPサーバー経由のツール結果（Web検索など）
- 対策として「セッションのコンパクション（圧縮）」が紹介される。コンテキストウィンドウの限界に近い会話を要約し、実装計画や整理済みの要件を引き継いで新しいセッションを開始する手法（カオスな会話履歴・探索のノイズをフィルタリングし、圧縮されたドキュメント・実装計画書として引き継ぐ）。

### コンテキストエンジニアリングの原則

- コンテキストウィンドウは貴重で有限なリソースであり、多くの情報を詰め込むだけではノイズになる。
- 必要最小限の高品質な情報を厳選することが重要。
- IPA「ソフトウェア開発分析データ集」に基づく概算として、1LOC ≒ 10トークン程度であっても、**1万行規模のコードベースですらコンテキストに収まりきらない**という前提を置くべきとされる（〜1万行：小規模、1万〜10万行：中規模、10万行〜：大規模の目安が示される）。

## 2. 最適なソフトウェアの構造を考える

### コンテキストの凝集

- ソースコード、IaCのソースコード、設定ファイル、設計ドキュメント、エージェントのステアリングファイルなど、コーディングエージェントが効果的に計画・実装するためには、**全てのコンテキストを1つのリポジトリに凝集することが理想的**とされる。

### Monorepo と Polyrepo

- Polyrepo：サービスごとにリポジトリを分割管理。
- Monorepo：インフラ定義、各種バックエンド、共有ライブラリ、テスト、CI/CDまでを1リポジトリで管理。開発初期は特に見通しが良いとされる。
- AIエージェントから見て「近接して読める」ことが重要であり、Polyrepoでも明示的なクロスリポインデックスで代替できるとされる。
- ディレクトリ構成例（TypeScript想定のWebアプリ）：`README.md` / `AGENTS.md`（リポジトリ全体像・ステアリングファイル）、`skills/`（エージェント向けSkills）、`packages/`（frontend / backend / libs / infra など）。

### Monorepoの弱点と対策

- 何も工夫しなければエージェントが大量ファイルを読み込みコンテキストウィンドウを圧迫する。
- 対策として「パッケージの依存関係を最小限にし、細部を見なくても理解できる構造を作る」ことが提案される。

### Deep Module / Shallow Module

- John Ousterhout『A Philosophy of Software Design, 2nd Edition』を引用し、「インターフェースはシンプルに、機能は深く（複雑な処理を内部に隠蔽する）」設計により全体の複雑性を減らすという考え方を紹介。
- 優れたモジュールは薄いインターフェースと深い実装を持つ（Deep Module）。コーディングエージェントがインターフェースだけを読んで意思決定できる状態を目指すべきとする。
- 良い例としてUnix File I/O（ファイルシステムの違いを意識しなくてよい抽象化）を挙げる。

### ドキュメントは腐る

- 実装（動くコード）とコメント／ドキュメント／仕様書は乖離が進み、実装とのズレ・ハルシネーションの原因となる。
- 「ドキュメントだけに頼らないソフトウェア構造の工夫が必要」（ドキュメント管理を完全否定するものではなく、補完的な仕組みが必要と付言）。

### モジュール境界・型表現の強化

- アーキテクチャの不変条件はインターフェース・型定義だけでは強制できないため、リンターや構造テストによる機械的な適用が推奨される（例：Node.jsの `eslint-plugin-boundaries`、Javaの ArchUnit）。
- サービスクラスのインターフェース例を用い、「型レベルで警告が出ない」問題を指摘。TypeScriptにおける型表現の強化（Value Objectなどでただの`string`ではなく検証済み`UserId`であることを型で伝える）を提案。

### Web APIもインターフェースの一種

- Web API = Web Application Programming "Interface"。
- Deep Moduleの例としてAmazon S3のAPI（認証・TLS終端、ルーティング、データ分割・整合性検証、暗号化、複製・耐久性保証などを`s3.put_object()`という単純なインターフェースの裏に隠蔽）を紹介。
- OpenAPI SpecificationによりAPIドキュメント・クライアントコード・バリデーション用サーバーコードを生成可能な点にも言及。

### Gitで管理すべきドキュメントの分類

| 情報の種類 | Source of Truth |
|---|---|
| WHAT（現在の振る舞い・型・API・スキーマ） | インターフェース、型、OpenAPI Spec、テストコード |
| HOW（アルゴリズム・実装の詳細） | コード |
| WHEN, WHO（変更の事実） | Gitのログ |
| WHY, WHY NOT（意思決定の文脈・棄却された選択肢） | コードコメント、ドキュメント（特にアーキテクチャ決定レコード） |

### まとめ：インターフェースは最大のコンパクション

- コンテキストの圧縮を、ドキュメントではなくインターフェースで表現するという考え方。
- 関数・クラス・モジュール・Web APIの各インターフェースは、それぞれ「命令と手続き」「状態と振る舞い」「内部実装と依存」「プロセス・言語・インフラストラクチャ」を隠蔽しており、「認知できて、制御できる」境界であるとする。
- サーバーレスは「AIの認知的な資源＝コンテキストを本質的な設計判断に使わせる」ものと位置付けられる。
- 各パッケージにおける考慮ポイント：
  - "薄い"インターフェースと"深い"実装（適切なモジュール境界、ドキュメントにはWHY/WHY NOTを記す）
  - モジュール単位に機能凝集されたテストコード（過度なモックを避ける）
  - 決定論的な仕掛けでAIにフィードバック（不変条件・依存の向き・型情報の検証）

## 3. 最適なサーバーレスアーキテクチャを考える

### AIエージェントから見た開発者体験の要件

1. 依存関係のあるコード・パッケージが適切に分割されていること
2. ローカル環境での動作確認、テストが素早く実行できること
3. インフラをコードとして管理し、デプロイができること

これらは「AIが自律的にフィードバックループを回せるか」を起点に考えるべきとされる。

### Lambda特有の課題

- 単一のLambda Functionにどれくらいのコード量を載せるべきか、handler関数のテスト方法、外部依存コードのユニットテスト方法が論点として提示される。
- Lambda特有のコード例（クエリパラメータからuserIdを受け取りデータを取得して返す処理）では、外部依存コードのベタ書き、独自のevent構造、handlerエントリポイントといった特性がテストの妨げになる点が指摘される。
- テストコードでは `APIGatewayProxyEvent` を丸ごと手組みする必要がある、関心のあるプロパティに絞りたいのに文字列を再パースする必要がある、DynamoDBクライアントのモックが必要になるといった課題が挙げられる。

### Lambda Web Adapter (LWA)

- Lambda上でWebアプリを実行するためのOSSツール（Rust製のLambda Extension）。2026/03/28にv1.0.0リリース。
- Dockerfileへの1行追加で導入可能。コンテナイメージはLambda以外の環境でもそのまま利用できる。
- 効果：
  - エミュレータ不要のローカル起動、ホットリロード
  - HTTPリクエストベースのテストフレームワーク採用可能（Node.jsなら supertest 等）
  - ローカル起動・リクエスト送信・結果確認のループをエージェントが単独で回せる（イテレーションの自律化）
- 注記：小さなコードや軽量処理、関数単位デプロイを志向する場合、あるいはWebフレームワーク依存を避けたい場合はLWAは非推奨。AWS SAM Localの併用でも同様の効果が得られるとされる（登壇者の私見として「AWS SAMが大好き」との言及あり）。
- HTTP以外のトリガー（SQS、SNS、S3、DynamoDB、Kinesis、Kafka、EventBridgeなど）にも対応。

### Lambda関数の粒度：Lambda-lith / Micro Lambda / Pragmatic Lambda

- **Lambda-lith**：単一Lambda内でルーティングロジックを持ち、複数エンドポイントを1関数で処理。
- **Micro Lambda**：エンドポイントごとに関数を分割。
- **Pragmatic Lambda**：境界付けられたコンテキスト、開発組織の構造、IAMパーミッションのスコープ、共通コードの依存関係、配下リソースとの依存関係、初期化時間とコールドスタート、メモリ割り当てなどを考慮したグループ化（例：User関連・Product関連で分割）。
- Monorepoのpackage単位でデプロイする構成例（`packages/user-service`、`packages/product-service` など）が示され、モノレポの利点（共通処理の参照）とマイクロサービスとしての独立デプロイを両立できるとされる。

### DynamoDBなど外部依存のテスト

- DynamoDB Local（DockerやJVMで起動するローカルエミュレータ）を用い、ロジック検証は高速にローカル/CI環境でテストし、IAM認可など統合テストはクラウド環境のDynamoDBも併用するという使い分けが提案される。

### クラウドを運用するAIエージェントの視点

- 従来の運用自動化（Runbookや監視ツール）を超え、自律的に観測・判断・実行・学習し運用業務を担う運用エージェント像が示される（観測→判断→実行→学習のループ）。
- 「機械的な運用」（クラウドのマネージドサービスによる決定論的・即時反応：オートスケーリング、フェイルオーバー、レート制限等）と「自律的な運用」（運用エージェントによる推論的判断、テレメトリ観測から設定変更のPull Requestを起票するなど、人間の介在を求める領域）を分けて捉える考え方。
- 運用エージェントが必要とするコンテキストとして、ランタイムテレメトリ（ログ・メトリクス・トレース・アラート）、アーキテクチャ情報（サービス境界・依存関係・IaC）、設計意図（SLO・アラート閾値・リトライ戦略）、運用の記憶（過去のインシデント履歴・ランブック）の4種が挙げられる。
- 「現実世界のサーバーレスアーキテクチャ」の例では、分散されたログがトレースできない、サービス境界が不明瞭、全テレメトリ調査でコンテキストウィンドウが不足する、Lambdaの言語ごとにログ構造がバラバラ、といった課題が図示されている（詳細は図版のため一部不明）。

### 可観測性の勘所

- Logs / Metrics / Traces をトレースIDでつなぐ。
- Metrics をビジネス指標に：CloudWatch標準メトリクスに加え、EMF（Embedded Metric Format）でビジネスKPIも出力。
- Traces でボトルネックを可視化：コールドスタート・スロットリング・ダウンストリームのレイテンシ等をCloudWatch Application Signals等で特定。
- Logs を構造化して相関IDでつなぐ：サービス境界ごとにCloudWatch Logグループを分離し、JSON形式でトレースID・ユーザーIDを含める。
- Lambda Powertoolsによるメトリクス収集（構造化ログ、AWS X-Ray統合トレース、コンテキスト情報付きカスタムメトリクス）でサービス境界を明示する。

## まとめ（Key Takeaway）

「AIが開発／運用しやすいクラウドの設計原則」として以下の4点：

- **コンテキストロット**：ノイズの蓄積により精度は落ちる前提を受け入れる
- **Pragmatic Lambda / Architecture**：PackageやLambdaを実用的でAIが管理可能な粒度で分割する
- **インターフェース・コンパクション**：サービス間の境界をインターフェースだけを読めばわかる状態を設計する
- **エージェント・オブザーバビリティ**：運用エージェントの目線でログ・メトリクスのサービス境界を設計する

> Key Takeaway：コンテキストの圧縮を、ドキュメントだけでなく、アーキテクチャで実現する
