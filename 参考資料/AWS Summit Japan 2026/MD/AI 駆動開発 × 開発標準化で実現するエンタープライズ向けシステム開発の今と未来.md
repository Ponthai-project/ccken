## Waterfall Boost：AI利用の開発環境

Vibesのコーディングから進化した、Vibesのノリを取り除いて企業ユースに耐えられるAIを利用したシステム開発環境として「Waterfall Boost」が紹介される。

### 3つのAgent群による構成

1. **Procedural Agent**（AIが開発をリード）
2. **QA Agent**（AIが品質を担保／ドキュメントを評価）
3. **Coding Agent**（AIが開発標準化を推進／生成するソースコードを制御）

### 開発フェーズ（Waterfall Boost全体像）

Phase1 要件定義 → Phase2 基本設計 → Phase3 詳細設計 → Phase4 反復開発・実装 → Phase5 結合テスト → Phase6 本番リリース → Phase7 保守運用

各Phaseで、要件定義書／基本設計書・ADR／詳細設計書・設定ファイル／実装コード・Pull Request／テスト結果・品質レポート／リリースノート・デプロイ記録／改善PR・運用ドキュメントが中間成果物として生成される。QA Agentによる「七賢者のドキュメントレビュー」、Coding Agentによる「自社開発ポリシーの反映」が各所に組み込まれる構成。

### Agent Knowledge Base

- 各Agentは自身の作業に必要な情報や次の手順確認のため「Agent Knowledge Base」にアクセスし、決められたルールに従って作業を進行する。
- Knowledge Baseにはこれまでの開発業務で培ったルール・規約・知財が登録されており、ルールがある限りAgentはエンジニアの指示よりKnowledge Baseの情報に必ず従う。
- AI開発IDE＋AI Coding Agentの例：Kiro、VSCode、Eclipse、IntelliJ IDEA等 ＋ Claude Code、GitHub Copilot等。
- AWSプロフェッショナルサービスの支援範囲：
  - Cognito×GitHub認証によるGitHub Enterprise利用エンジニアへのMCPサーバアクセス限定
  - Amazon Bedrock × Amazon S3 Vectors・Index構成のナレッジベース参照
  - Amazon Bedrock AgentCore Gatewayを介したAmazon S3上のナレッジベース（開発ナレッジ・標準開発規約）参照、GitHub内の標準コンポーネント情報の参照
  - エンジニアが過去の知財・事例を意識的に検索することなく、MCP経由でナレッジベースから開発に必要なナレッジを参照可能

## Procedural Agent（AIが開発をリード）

- 「開発プロジェクトをリードして人の怠けを許さない」Procedural Agentが、各Phaseで設定された実行タスクをリードし、人と会話しながら開発を進行するとされる。
- 各Phaseにおける主なタスク例：
  - Phase1要件定義：ビジネス要件、機能要件、非機能要件定義
  - Phase2基本設計：アーキテクチャ設計、技術スタック選定、ADR作成
  - Phase3詳細設計：API/データモデル/インフラの設計、プロジェクト本体のセットアップ
  - Phase4反復開発・実装：Epic/User StoryのIssue作成→TDD→実装→コードレビュー→CI/CD・マージ
  - Phase5結合テスト：統合テスト、E2Eテスト、負荷テスト
  - Phase6本番リリース：リリース作成、承認フロー、本番デプロイ
  - Phase7保守運用：監視・アラート、バグ修正、機能改善、ドキュメント保守
- AI（Agent）がやること／人がやることの役割分担が各Phaseごとに整理されている（設計書草案生成、要件定義草案作成、Issue自動生成・登録、テストコード生成、移行スクリプト生成などはAI側。設計方針決定、質問への回答、レビュー、整合性確認、承認、最終意思決定などは人間側）。
  - **注記**：本項の詳細な役割分担表（p13相当）は、抽出テキストの一部が図版・レイアウトの都合で文字化けしており、対応関係の細部までは判読できない箇所がある（図版のため詳細不明）。
- 人が要件定義書のレビューをして承認した後に次Phaseに進行するというゲート構造が明示される。

## QA Agent（AIが品質を担保）

- AIが生成した成果物を「7人のペルソナAI」がチェックする仕組み（「七賢者のドキュメントレビュー」）。
- ウォーターフォール型開発ではドキュメントの精度が悪いと成果物の品質も劣化するという課題に対し、マルチLLMで7人のペルソナが7つの観点からチェックし、課題があれば改善点をフィードバックする。
- 既存開発手法でもAIを活用したシステムドキュメントの品質管理を実施：レビューと工程管理による品質管理に加え、開発工程ごとにAIが成果物をチェックし、そのチェックに基づくテストで合格判定がされない限りリリースできない仕組み（AIが成果物をチェック／AIがテスト計画書を自動生成／AIがテスト結果を評価／AIが実装・コードレビュー）。

## Coding Agent（AIが開発標準化を推進）

- AIによる実装をコントロールし、リスクのないコードを生成する役割。
- AIが生成したソースコードがブラックボックス化して著作権や脆弱性の課題が発生するのを防ぎ、自社ポリシーに則ったコーディングや既存知財の再利用を促進。
- トランスコスモス社内に蓄積された開発資産（コーディングルール・ドキュメントテンプレート、UIコンポーネント集/プロジェクトノウハウ）へアクセスし、標準化された設計書に基づくドキュメント作成や、開発済みのUI・コンポーネントの実装を行い、「エンジニアに意識をさせない開発標準化」を実現するとされる。

### コンポーザブルアーキテクチャ

- AI駆動開発によりコンポーネントの大量開発が可能となり、さらなる開発工数の圧縮・開発効率の最大化が可能とされる。
- レイヤー構成：Platform Layer（Infrastructure）／Business Capability Layer（コンポーネントライブラリ：直接組み込みコンポーネント、API利用コンポーネント）／Application Layer（独自UI/ロジック開発、オーケストレーションプラットフォームによる汎用UI/ロジック提供）。
- コンポーネント情報の共有→問い合わせ→利用した実装、という流れでAI駆動開発コンポーネントが活用される（図解、詳細は図版のため一部不明）。

## Forward Deployed Engineer (FDE) による展開

- AI駆動開発環境を必要とするクライアントの内製開発チームやSIerに対し、「AI駆動開発環境の提供＋FDEの常駐」という形で展開。
- クライアント側：Component Knowledge / UI Template / Waterfall Boost 環境の提供を受け、現場の改善要望をトランスコスモスへ持ち帰る。
- トランスコスモス側：①次世代AI駆動開発PFの開発、②要望を機能アップデートに反映、③現場知をプロダクト価値に転換、というサイクルで自社プロダクトをアップデート。
- FDEの役割：①AI駆動開発環境の導入支援、②AI活用の啓蒙・利用促進、③導入効果の計測と改善、④現場の改善要望の回収。

## 完全自律型システム開発AIエージェントの実現に向けて（今後の挑戦）

3フェーズで構成される将来構想：

- **PHASE1 要件抽出/タスク化**：①人間がクライアントとMTGした録音データを音声のままAIに投入→議事録化、②AIが議事録からシステム開発に必要な機能要件・作業をタスクに自動分解して整理、③整理済みタスクをAIが人間に提示→人間が確認
- **PHASE2 仕様化/QAループ**：④開発に必要な既存仕様書・設計書等をAIが作成、⑤AIが仕様や解釈の不明点を自らQA化して人間に質問、⑥QAに対する人間の回答でAIは理解度を深め設計書等の齟齬を再確認
- **PHASE3 開発/モック/デプロイ**：⑦最終設計書をもとにAIが開発作業を開始、⑧開発環境でモックを生成→人間が確認、⑨問題なければAIが本番環境でシステムを完成させリリース

> これからのエンジニア（人間）は、クライアントが作りたいシステムの情報を聞いてくる係。過不足があればQA担当のAIエージェントが人間に質問してくれるので、必要に応じてクライアントに再質問して情報を補足する、という将来像が示される。
