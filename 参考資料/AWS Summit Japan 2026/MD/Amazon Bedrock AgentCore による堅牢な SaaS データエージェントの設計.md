## 1. データエージェントとその課題

### データ分析手段の変遷

定型レポート依頼 → ダッシュボード構築依頼 → セルフサービスBI → セマンティックレイヤー整備・自然言語クエリ（NLQ）→ AI エージェントが自律的に分析を遂行する「データエージェント」へと進化してきた、という変遷が示された。

### データエージェントと BI の違い

- **BI ツール**: データエンジニアが事前に検証済みの SQL 文・ダッシュボードを設計・構築し、ユーザーはそれを利用するのみ。
- **データエージェント**: LLM がユーザーの指示に従って動的に SQL 等を生成し、DWH からデータを取得・解析する。事前検証されたクエリではなく、その場で生成される点が本質的な違い。

### 脆弱なエージェントのデモ（4パターン）

素朴な実装のデータエージェントが抱える脆弱性を4つのデモで例示。

| デモ | 脆弱性 |
|---|---|
| デモ1 | PII（個人情報）漏洩 |
| デモ2 | テナント越境（異なるテナントのデータへのアクセス） |
| デモ3 | 業務外の話題への応答 |
| デモ4 | LIMIT のない非効率なクエリの実行 |

### 実装上の課題

- AI が非効率なクエリを書いたらどうするか
- DB 権限の制御が不十分だったらどうするか
- 偽の `tenant_id` を注入されたらどうするか
- プロンプトインジェクションへの対策は何か

これらは以下3点に整理される。

- **行動統制の不足**: LLM が生成する SQL を制御する仕組みがない
- **認可レイヤーの不在**: ツール呼び出しの認可が LLM 任せになっている
- **テナント越境**: LLM への WHERE 句指示のみではテナント分離は保証されない

## 2. ４つのレイヤーで考える堅牢なデータエージェントの設計方針

### OWASP Top 10 for Agentic Applications（参照フレームワーク）

本セッションは OWASP の Agentic Applications 向けリスク分類（ASI01〜ASI10）を防御設計の基準として用いている。

| コード | カテゴリ | 内容 |
|---|---|---|
| ASI01 | Agent Goal Hijack | プロンプト操作で目的を改竄 |
| ASI02 | Tool Misuse | 正規ツールを意図しない方法で悪用 |
| ASI03 | Identity & Privilege Abuse | 権限委譲・認証情報の悪用 |
| ASI04 | Agentic Supply Chain | サードパーティモデル・プラグインの汚染 |
| ASI05 | Unexpected Code Execution | 生成コード・外部コードの意図しない実行 |
| ASI06 | Memory & Context Poisoning | メモリー・RAG の汚染 |
| ASI07 | Insecure Inter-Agent Communication | エージェント間通信の改竄 |
| ASI08 | Cascading Failures | 障害の伝播・増幅 |
| ASI09 | Human-Agent Trust Exploitation | エージェント出力への過信 |
| ASI10 | Rogue Agents | 認可範囲を逸脱した行動 |

（出典: https://genai.owasp.org/resource/owasp-top-10-for-agentic-applications-for-2026/ ）

### エージェントの動作モデルとフロー

エージェントは Perceive（観察）→ Reason（推論）→ Act（行動）のループで動作する。データエージェントの文脈では、ユーザーの問いを LLM が受け取り（Perceive/Reason）、SQL 構築・実行ツールを呼び出し（Act）、DWH から結果を得て次のターンへ、という流れになる。この動作フロー上の各ポイントに OWASP の ASI 分類が対応づけられ、防御すべきポイントが可視化される。

### ４つのレイヤー

上記の対応関係から、防御点を以下の4レイヤーに整理する。

| レイヤー | 名称 | 制御対象 | 主な OWASP 分類 |
|---|---|---|---|
| レイヤー1 | LLM 層 | LLM への入出力（プロンプト・コンテキスト上の脅威） | ASI01 |
| レイヤー2 | 呼び出し層 | ツール呼び出しの可否検証 | ASI02, ASI03 |
| レイヤー3 | ツール層 | ツール実装内部の安全性（パラメータ検証等） | ASI05, ASI03 |
| レイヤー4 | リソース層 | DB・ナレッジベース・メモリーへのデータアクセス制御 | ASI06 |

対策の原則は、LLM 層は「入出力の制限と適切なコンテキストの注入」、呼び出し層は「ツール呼び出し時の引数・認可情報の検証」、ツール層は「ツール内部の安全性をコードで保証」、リソース層は「最終的なリソースへのアクセス制御の実装」である。

## 3. 各レイヤーの実装

### レイヤー1: LLM 層の対策

- **ガードレールの整備**: Amazon Bedrock Guardrails により、業務範囲外の会話やプロンプトインジェクションをブロックする実装例が紹介された。
- **システムプロンプトの整備**: Strands Agents でシステムプロンプトを設定し、禁止事項や行動ルールを記述して振る舞いを抑制する例が示された。

### レイヤー2: 呼び出し層の対策

Amazon Bedrock AgentCore gateway を用いると、ツール（Lambda 関数など）を実行する前に各種検証を差し込むことができる。AgentCore gateway は API Gateway Target、OpenAPI Target、AWS Lambda Target、MCP Server Target といった複数のターゲット種別をサポートし、Gateway Policy（宣言的・静的な判定）と Gateway Interceptor（手続き的・動的な判定）の2種類の検証機構を持つ。

- **静的検証**: Cedar Policy によって、エージェントがツールを呼び出す権限を持つかを宣言的に検証する（危険ツールの明示的拒否、テナントID保持時のみ利用許可、使用可能ツールの明示的許可、など）。
- **動的検証**: Gateway interceptor を利用し、カスタム Lambda 関数で引数の妥当性確認、許可リストとの照合、呼び出し主体の認可といった動的判定を実装する。

### レイヤー3: ツール層の対策

データアクセスには2つの戦略があり、トレードオフが存在する。

- **SQL自動生成**: LLM が SQL 全体を生成する方式。柔軟性は高いが堅牢性は低い。
- **関数型ツール**: LLM はパラメータのみを生成し、SQL 自体はコードが構築する方式。堅牢性が高い。

SQL自動生成を採る場合でも、入力安全化・スコープ制限（読み取り専用ロール、クエリ時間制限、実行前構文検証、許可外テーブルの拒否など）を段階的に強化することで堅牢性を高められる。関数型ツールでは、テナントIDを実行コンテキストから取得してクエリに埋め込むことで、プロンプトインジェクションを防止しつつ JOIN/WHERE/カラム選択を正確に制御できる。

ツール内部で実装すべき統制は以下の4種に整理される。

| 統制 | 内容 | 具体策の例 |
|---|---|---|
| 入力境界 | ツールに入ってくるものを制限 | SQL識別子の許可リスト、引数検証、JWT claim からのテナントID埋め込み |
| 実行制約 | 実行時のリソースと権限を制限 | 読み取り専用ロール、LIMIT、タイムアウト、パラメータバインド |
| 出力境界 | ツールから出ていくものを制限 | PII列除外、エラーマスキング、出力結果サニタイズ |
| 横断 | 全操作を記録 | 監査ログ（tool / tenant / user / operation / row_count） |

### レイヤー4: リソース層の対策

マルチテナント SaaS では、ユーザーの JWT から取り出した `tenant_id` を、LLM の推論を挟むことなく機械的にリソースまで伝播させることが要点となる（Backend Server → エージェントランタイム → SQL構築・実行ツール → DWH という経路）。

- **テナットのアクセス制御（DB）**: プールモデルの SaaS では PostgreSQL の RLS（Row Level Security）やテナントごとの View を用いる。どんな SQL が来ても指定テナント以外は見えない構成とする（ただし RLS をネイティブサポートしないDBも存在する点に留意）。
- **メモリーの分離**: Bedrock AgentCore memory の namespace と IAM により、テナントを厳密に分離する（複合キーによるテナント分離の実装例、IAM Policy によるテナントID付与の強制）。
- **ナレッジベースの分離**: Amazon S3 にテナントごとのフォルダを構成し、metadata に `tenant_id` を挿入して Amazon Bedrock Knowledge Bases の Retrieve でフィルタリングする。

## 4. リファレンスアーキテクチャ

AWS Cloud 上で、Amazon Cognito による認証、AgentCore runtime / memory / gateway、Amazon Bedrock Knowledge Bases、Amazon S3、Amazon RDS (PostgreSQL)、AWS Lambda を組み合わせた構成が示された。ユーザーの JWT から抽出したテナントIDを、LLM の推論を経由せずにリソース（RDS・S3・AgentCore memory）まで機械的に伝播させる点、およびツール呼び出し検証（AgentCore gateway）とリソース側の RLS・IAM Policy による強制という「多重防御」を組み合わせる点が強調された。

### レイヤーと攻撃の対応（まとめ表）

4つのレイヤーの防御が、プロンプトによる攻撃・テナント越境・PII漏洩・非効率なクエリのそれぞれに対し「停止」または「部分軽減」の効果を持つことが一覧化された（具体的な対応関係はスライド図表のため詳細不明な部分を含む）。

## 5. デモ

- プロンプトインジェクション防止: ガードレールで検出し、回答を抑制するデモ
- 業務外の話題の禁止: ガードレールで検出し、回答を抑制するデモ
- 非効率クエリの防止: 検証済みクエリの実行により非効率クエリの生成を防止するデモ

## まとめ

- データエージェントは分析体験を大きく変える一方、従来のBIとは異なる設計課題を持つ
- マルチテナントSaaS環境では、エージェントの柔軟性とテナント分離・権限制御を両立する必要がある
- 堅牢性はLLM層・呼び出し層・ツール層・リソース層の各層で設計する
- LLMは自然な対話と推論を担い、最終的な制御は決定的な仕組みに委ねる
