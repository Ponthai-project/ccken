## セッションテーマ・スコープ

**テーマ**: AI駆動開発に必要なソフトウェアの叡智を、個人がどうやって獲得し、価値に転換するか

**対象**: ソフトウェアエンジニア、ITリーダー向け（L300）

**このセッションで扱わないこと（明示的に除外）**:

- AIコーディングエージェントの機能詳細
- 誰でも同じ結果が得られるプロンプトや技法
- **AI-Driven Development Lifecycle (AI-DLC)**

> 本セッションは冒頭で「AI-DLCは話さない」と明言しており、AI-DLCそのものを解説する資料ではない点に注意。

## 導入: AI駆動開発の現在地と課題

- AIコーディングエージェントは十分な品質で開発タスクの一部を完遂できるようになっており、AIエージェントによる自律的なコード探索・編集・テストは実用化済み。長時間の自動運転・人間の意図の補完も進んでいる。
- コードを書く主体はAIになりつつあり、エンジニアはコードに直接注意を払うことに加え、意図の伝達や設計への関心をシフトしつつある。
- 開発の自律化が進む中で「仕様駆動開発」「テスト自動化」「ブラウザ操作」「DB/ログの確認」が鍵となる要素として提示された。

### 「ソフトウェアはいつから本番か？」という問い

- ユーザーが生まれたとき／機密データが入力されたとき／他のソフトウェアに依存されたとき、という3つの契機が提示される。
- Amazon CTO Dr. Werner Vogels の言葉（AWS re:Invent 2025）: **"You build it, you own it."**
- ソフトウェアを所有することで発生する責任として「正しく動くこと」「安全であること」「変化に応え続けられること」の3つが挙げられる。

### 自律化の進んだAI駆動開発における課題（3つの問い）

1. チームやプロセスのブロッカーを解除するには？
2. AIが構築したソフトウェアを評価し所有責任を負うには？
3. ソフトウェア開発の能力を自分の価値に転換するには？

本セッションは、この3つの問いに沿って以下の各セクションで議論を展開する構成になっている。

## Design for Evaluation（本セッションの中核概念）

**定義**: AIにタスクを引き渡す前に「評価可能か？」を問い、評価のための設計を完了する「型」。

### 「型」という概念について

大庭良介『「型」の再考: 科学から総合学へ』（京都大学学術出版会, 2021）を参考に、叡智の保存・伝達方法を2種類に整理する。

| 種別 | 説明 | 例 |
|---|---|---|
| **外在化された叡智** | 自分の外にある叡智を選んで適用する。対象を分析して選択・適用。誰が使っても同じ推論が期待される | デザイン／アーキテクチャのパターン、プロンプトテンプレートやスキル、ライブラリ、共通基盤 |
| **内在化された叡智（型）** | 自分の内にある叡智が自ずと現れる。主体を通して自ずと現れ、使う人や状況によって異なる推論になる | Design for Evaluation、Everything as Code、DevOps、アジャイル |

Design for Evaluation は、評価可能性の設計という叡智を個人に**内在化**させるための「型」として位置づけられる。

### 評価可能にするための3要素

- 評価基準
- 観測対象
- 評価手段

この3つが揃った状態を「評価可能」と呼ぶ。

### 型の修得プロセス：守破離

- **守**: 型の想定・動作・姿勢を崩さず、型を徹底的に守ることで叡智の獲得を目指す
- **破**: 新たな型や、より自分に合った型を模索する
- **離**: 既存の型に囚われることなく離れて自在となり、達人を目指す

（AI にタスクを引き渡す前に「評価可能か？」と問うこと自体が「守」の実践として位置づけられる）

以降、本編は「正しく動くこと」「安全であること」「チームやプロセスのブロッカー解除」「価値への転換」の各テーマに沿って、守破離の各段階と対応づけながら展開される。

（他セッション紹介: 変更容易性と設計については CNS449「AIが開発／運用しやすいクラウド サーバーレスの視点から考える設計原則」（Day2 14:00-14:40、淡路大輔氏）が案内されている）

## テーマ1: 正しく動くこと

### AI駆動開発における要求とのずれ

AIが十分な品質でコードを書けても、以下のようなずれがあると価値を生まない。

- ソフトウェアの振る舞いがビジネスの要求とずれている（Biz側）
- ソフトウェアの構造や特性がエンジニアの期待とずれている（Dev側）

解決の方向性として「意図を評価可能にしてAIと人間の双方が合意すること」が重要と説く。自然言語プロンプトの曖昧さ（AMBIGUOUS）から、精密な仕様（PRECISE SPEC）への変換が必要という趣旨の図解が示された（詳細な図の文言は一部不明瞭）。

### 仕様駆動開発で正しさを合意する（Kiro）

Kiro が人間の指示を仕様に変換する3段階のプロセスとして紹介される。

1. **Requirements（要求定義、EARS記法）** → **Design（ソフトウェア設計）** → **Tasks（実装タスク）**
2. 仕様を介して人間とAIが合意形成する（目的は仕様書を作ることではない）
3. AIが仕様をもとに自律的に実装する。仕様は「スーパープロンプト」として機能する

**Requirements（振る舞いを評価可能にする）**

| 主体 | 評価基準 | 評価手段 | 観測対象 |
|---|---|---|---|
| Dev / AI 合意 | 要求定義の受入基準 | ブラウザテストツール | 開発環境のソフトウェア |
| Biz 合意 | ドメインエキスパートの知見 | 文書やデモの読み合わせ | requirements.md |

**Design（構造と特性を評価可能にする）**

| 主体 | 評価基準 | 評価手段 | 観測対象 |
|---|---|---|---|
| Dev / AI 合意 | 構造の制約やテストの要件 | Linterやツール、自動テスト | ソースコード |
| Dev 合意 | シニアエンジニアの知見 | 文書や図のレビュー | design.md |

### 仕様はいつ、どのように作るか

- 毎回・すべて作る必要はない。「次のタスクを担当する人が自由に決められないことを事前に合意する」観点で判断する
- 作成タイミング: ソフトウェアの外部的な振る舞いが変わるとき（Requirements）／内部的な構造や特性が変わるとき（Design）
- 永続化の方針: 機能ベース・時点的な合意のための仕様から、永続化すべき情報を選択的に反映する（`README.md`, `docs/migration.md`, `design/<domain>/user-story.md`, `design/<domain>/adr/...` 等）

### 評価基準をエンジニアの中で育てる

評価基準（評価可能の3要素の1つ）は、エキスパートとの対話やソフトウェアの古典・名著からエンジニア自身が学び、経験を圧縮することで育つものであり、これにアルゴリズムはない。Amazon CEO Andy Jassy の言葉が引用されている（具体的な引用文はスライド画像内のため詳細不明）。

### ツールより個人の叡智

Kiro IDE・Kiro CLI、あるいは他の好みのツールや技術を使ってもよいが、「型にはまる」のではなく型を内在化することが重要。良いツールは叡智を含むが、それでもツール自体より個人の叡智獲得が重要である、という立場が示された。

## テーマ2: 安全であること

AWSの標語 "SECURE BY DESIGN, AT EVERY LAYER, AT EVERY STAGE" ("At AWS, everything starts with security") を引用。

### ソフトウェアの安全性を評価可能にする

評価可能性（評価基準・観測対象・評価手段）＝フィードバック可能であることの前提として、コード・実行環境・依存ライブラリ・CI/CDパイプラインなど、インフラからアプリまでソフトウェア全体が観測可能である必要がある。

### Everything as Code

Infrastructure as Code、Configuration as Code、Policy as Code、Security as Code、Documentation as Code、Operation as Code、Observability as Code、Database as Code 等、あらゆる構成部品をコードで管理する考え方。

- ソフトウェアのあらゆる構成部品をコードで管理し、変更の追跡やテストを可能にする
- 環境間での差異をなくす
- 人間しか知らない設定やルールをなくす

「ソフトウェアのすべての部分をコードとして管理することは、AI駆動開発とセキュリティの両方の基礎」と位置づけられる（参考: AWS Well-Architected DevOps Guidance の Everything as Code）。

**AWS CDK**: 使い慣れたプログラミング言語（TypeScript, JavaScript, Python, Java, .NET, Go）でクラウドリソースを定義できるOSSフレームワーク。"Meeting developers where they are" という思想のもと、型チェック・入力補完、デプロイ前のセキュリティ検査、テストの自動化など、アプリと同じ言語での一貫したデプロイを実現する。

**『Gitリポジトリを見ればすべてわかる』**: アプリ（apps/agents, api, web）からインフラ（infra, AWS CDK）、運用まで、変更の経緯やポリシー（Policy as Code, Documentation as Code, Infrastructure as Code, Security as Code）も含め、AIが全体を把握して変更できる状態を作る、というリポジトリ構成の考え方。

### Everything as Code で評価の面を広げる

コード化によって、SAST（静的アプリケーションセキュリティテスト）やSCA（ソフトウェア構成解析）、ポリシー検査をインフラからアプリまで適用できる。

- SAST/SCA: Amazon Inspector
- ポリシー検査: CloudFormation Guard、CDK Nag
- コードレビュー: Kiro ヘッドレス実行、AWS Security Agent

Kiroヘッドレス実行やAWS Security Agentにより、CI/CDパイプライン上でのコードレビュー範囲を拡張できる。

### AIエージェントが安全に探索するための分離

評価基準と手段が与えられていれば、AIエージェントは自律的に実装を評価できるという前提のもと、環境とアクセス権を分離する実例が示された。

- Webブラウザ: Playwright CLIと専用テストユーザーで操作
- データベース: Amazon RDS Data APIでテスト環境をクエリ
- ログやイベント: CloudWatch MCP ServerやAWS CLIで読み取り

Infrastructure as Code で構築した環境において、IAM Policy Autopilot（`awslabs/iam-policy-autopilot`）を使うと、アプリの関連リソースにのみアクセス可能なIAM Policyを定義できる。

### セキュリティ評価は標準から始め、継続する

広く合意された標準やセキュリティサービス（Amazon Inspector、AWS Security Hub CSPM、AWS Security Agent）の評価結果を観察して何が問題かを学ぶことから始め、一度きりでなく継続的に評価する、という方針が示された。

### 「評価可能か？」をすべてのレイヤーで問う（テーマ1・2のまとめ）

- 仕様駆動開発: 振る舞いと内部構造を、人間とAIの両方が評価できるか（正確性）
- Everything as Code: ソフトウェアの全体を観測して安全性を評価できるか（安全性）

この問いをきっかけに評価可能性を設計する、という形で両テーマが統合される。

## テーマ3: チームやプロセスのブロッカーを解除するには？（守破離）

### 評価可能性が『待ち』を解消する

「UIのデザインをかっこよくして」「コードに問題がないかレビューして」のような、言語化はされているが評価基準と手法がない依頼は、人間の評価待ちがボトルネックになる。評価可能性（AIが評価可能かという問い）を通してAIへのフィードバックループを閉じることで、AIの自律性を強化できる。

### 評価手段の例（5パターン）

1. **ソフトウェア構造の評価と制約**: モジュール間の依存グラフ等を評価し構造を制約する（例: モノレポでの `apps` インポート禁止、循環参照の禁止、リポジトリ層を迂回したDBクライアント使用の禁止）
2. **テスト容易性の設計**: ポートとアダプターパターン（ヘキサゴナルアーキテクチャ）でストレージ操作等を抽象化し、AIエージェント自身が素早くテストを実行できる構造をあらかじめ設計する（参考: AWS Prescriptive Guidance "Building hexagonal architectures on AWS"）
3. **Generator-Verifierパターン**: 自然言語の出力や構造をAIエージェントを分離して評価する。コード・デザインのレビューやAIエージェントの出力評価に適用。メリットは自然言語での意味評価が可能で適用範囲が広いこと、課題は評価基準の構築と評価基準自体の評価に専門性が必要なこと（参考: Anthropic "Multi-agent coordination patterns: Five approaches and when to use them"）
4. **セキュリティ評価の自動化（AWS Security Agent）**: 従来は数週間を要した侵入テストをマルチエージェント構成で自動化。セキュリティ専門家が記述した自然言語アサーションを組み込んだ検証エージェントが検出結果を評価する（参考: AWSブログ「AWS Security Agent 徹底解説」）
5. **形式検証による制約条件の評価**: ネットワークの到達可能性を、設定情報のみでパケットを実際に送らず数学的に評価する（AWSの実例: VPC Reachability Analyzer）。メリットは非決定論的な出力の論理的検証、課題は制約条件の定式化に形式検証とドメイン両方の専門性が必要なこと

### Design for Evaluation で『待ち』を解消する（Plan-Design-Code-Test-Release-Operateの各段階）

- Plan/Design: 要求やUXをBizが仕様駆動開発で評価
- Code: コードの正確性をAIエージェントが実装中に評価
- Test: セキュリティや品質を専門エージェントが評価
- Release: コードレビューをAIエージェントが補助

## テーマ4: ソフトウェア開発の能力を自分の価値に転換するには？（守・破離）

### AIにコードを「書かせない」選択肢

AIがすぐに実装してくれても、無駄な機能を作れば負債になる。効率的な課題解決のエキスパートとしての熟練エンジニアの叡智が「作らない」判断を可能にする。

### Working Backwards

Amazon のイノベーションメカニズムである Working Backwards（Press Release / FAQ / Visuals / Customer を起点に、どのように顧客と検証し何を成功指標とするか）には、Design for Evaluation が組み込まれているという位置づけが示された。

引用（佐藤賢太氏, Solutions Architect, AWS Japan）: 「自分にできることが小さいと感じるなら、社会をシステムとして捉え、問いを立て続けよう」

### システム思考とWorking Backwardsの往還

- 顧客と同じ言葉で話し、抱えている課題と力学をシステム思考で分析し、領域を絞り込む
- Working Backwards とAI駆動開発でプロトタイプを何度も作り、仮説を洗練させる
- 両者を意識的に切り替える

### ソフトウェアの叡智を師範として伝える

学習者にとって叡智がどう使われてほしいかに応じ、伝え方を使い分ける。

| 想定される受け手のあり方 | 伝え方 |
|---|---|
| 学習者の外にある叡智として、対象を分析して選択・適用してほしい | **外在化**した叡智として切り出す（パターン、ライブラリ、テンプレート） |
| 学習者の内にある叡智として、主体を通して自ずと現れてほしい | **内在化**するための叡智として伝える（型） |

### 「型」の習得を組織で支える

- **価値ある型を探索する**: 外在化と内在化の使い分けを認識する
- **内在化を支援する**: 内在化のための修練と、個人ごとに異なる発露を前提とする共通認識の醸成・場の提供
- **個人による発露を広める**: フォーマルな事例・文書・共通化よりも個人の生の活動から叡智を得る。失敗が許容され多様性を受け入れる実践者コミュニティを作る

### 事例: AWS Japan のAI駆動開発コミュニティ

- 2025年3月にAWS Japan社内エンジニア対象のAI駆動開発コミュニティを設立
- 多様な考えの共有と「遊び」による体験をベースとする、AI時代の組織における学びを奨励
- 1,000名以上が参加し、ライブコーディングやLT大会などの参加型イベントを定期的に開催
- Our Tenets: Play with Tech / Every Voice Matters / Learn Through Your Hands / At Your Pace

引用（巨勢泰宏氏, アマゾンウェブサービスジャパン合同会社 常務執行役員技術統括本部長）:
「生成AIの非決定論的な振る舞いと急速な進化は、ビルダーに好奇心と不安の両方をもたらします。心理的安全性の高いコミュニティでの実験と学びの文化こそが、この変化を組織の競争優位性に転換するメカニズムなのです。」

## まとめ

- 評価可能性がAIの自律性を支える（仕様駆動開発、テスト自動化、ブラウザ操作、DB/ログの確認）
- **Design for Evaluation**: AIにタスクを引き渡す前に「評価可能か？」を問う。評価基準・観測対象・評価手段の3要素で評価可能にする。評価可能性の設計という叡智を個人に内在化させるための「型」である。
