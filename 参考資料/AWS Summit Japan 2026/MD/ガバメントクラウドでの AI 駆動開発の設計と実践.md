## 1. AIコーディングエージェント導入による期待

### 開発生産性の向上とボトルネックの移動

- 開発速度は劇的に向上しており、開発のボトルネックは「実装」から「判断」へ移動している
- 判断待ち時間を減らすため、AIエージェントの環境整備が重要
- 実装がボトルネックとならないことで、体制全般（従来の受発注、ユーザー企業とSIer、POと開発者という体制）の変革が必要になる
- 人海戦術による品質確保からDevOps、テストファースト、SHIFT LEFTへの変化に続き、AIの登場でボトルネックがさらに移動し、品質確保の方法論も変化している

デジタル庁ガバメントクラウド班では、AIエージェントに適切な役割と制約を与えながら活用する設計手法を検討中としている。

### AIコーディングエージェント導入による効果（期待される4分野）

| 分野 | 内容 |
|---|---|
| 生産性/開発速度の向上 | AIコーディングエージェントが基本実装を進めることで生産量の増加が見込める |
| スキルギャップの補完 | AIに業務・作業の一部をスキルとして定義し、熟練の技を誰でも使いやすい状態に移す |
| 引き継ぎコストの低下 | 異動・転職が激しい業界で引き継ぎが不可欠であり、AIがスキルトランスファーの要になる |
| 品質/セキュリティの向上 | AIによるレビューでバグや脆弱性を早期発見し、根本原因への対処速度を上げる |

これらを通じて「高品質なサービス提供」を目指す。まずは小規模でステークホルダーの少ないシステムでの実験検証から着手する方針。

### 効果検証の実験

- **検証内容**: プロダクトオーナー（PO）がAIコーディングエージェントで小規模機能の実装を進める
- **条件**: エンジニア出身のPO、小規模・限定ドメインで既存チームとの見積り比較
- **体制**: プロダクトオーナー兼エンジニア＋AIコーディングエージェント
- **結果**: 従来見積り（PO＋エンジニア）720H前後（576〜864H）に対し、実験結果は120H前後＋LLM利用料約20万円
- **わかったこと**: 仕様調整と実装への反映が早くなることで生産性は大きく向上する一方、AIコーディングエージェントの導入により見積りはより困難になっている
- **再現に必要な条件**: 自身で仕様調整やコード修正ができるエンジニアがいる場合に再現可能

### 検証の再現性を高めるために

1. 仕様をその場で判断できる開発体制への移行
2. AIエージェントの挙動を熟知し、コードやアプリの動作から修正指示できるエンジニアの存在
3. AIエージェントを効率的に使える環境の整備

必要なのは「スーパーPO」ではなく、誰でも再現できる仕組みづくりであり、そのためにまず3（環境整備）の土台づくりから始める、という方針が示された。

## 2. ガバメントクラウドとは

### 定義

**ガバメントクラウド＝「技術要件を満たした民間クラウド」＋「モダンなITガバナンス機能」**

デジタル庁が契約したクラウドサービス（Amazon Web Services (AWS)、Google Cloud、Microsoft Azure、Oracle Cloud Infrastructure、さくらのクラウド）を公共情報システムのための共通クラウドサービス利用環境として提供し、迅速・柔軟・セキュア・コスト効率の高いシステムの実現を目指す。

- 利用者は調達することなく、既成のクラウドサービスを選んでそのまま使うことができる
- デジタル庁はテンプレート等のガバナンスに必要な機能のみを利用者に提供し、クラウド最適でモダンなシステム構成の推進を方針とする

### 法的根拠・位置づけ

- 「情報通信技術を活用した行政の推進等に関する法律」に基づく「クラウド第一（クラウド・バイ・デフォルト）原則」（2024年12月法改正、国の行政機関等はクラウド利用の検討・整備が義務、それ以外は努力義務）
- 「政府情報システムにおけるクラウドサービスの適切な利用に係る基本方針」
- デジタル社会推進標準ガイドラインの1つ、デジタル庁重点計画の「デジタル社会の実現に向けた理念・原則」の1つとして位置づけ

### ガバメントクラウドが実現してきた「制約からの解放」

| 従来 | 現在 |
|---|---|
| 厳格な統制 | 発見的統制へ（中央からの利用制限による厳格な管理から、より自立的なセキュリティ対応が可能に） |
| 個別制限 | ガードレールへ（機能・サービスの個別利用制限から、評価・監査・国内制限等の枠組みによるモダンなガバナンスへ） |
| 手動処理 | 自動化へ（利用開始の手作業＝サービスチケット管理から、利用開始作業の自動化による大量処理対応へ） |
| 静的評価 | 動的評価へ（見積り・設計時のコスト/セキュリティ前提の運用から、動的・リアルタイムの状況把握へ） |
| 独自実装 | CSP機能の利用へ（マルチクラウドの統合管理を独自画面・APIで行うのではなく、既存クラウドの画面・APIをそのまま利用可能に） |

### 構成の概要

デジタル庁管理領域（GCAS: ガバメントクラウド支援サービス。本人確認、認証、環境申請、ガイド、ヘルプデスク、ダッシュボード管理、プログラム管理）と、利用システム領域（各CSP上の独立した環境・拠点システム群）で構成される。利用システムの管理者はGCASを使って環境申請が可能で、ネットワーク的にも管理系も独立した環境とその管理権限が提供される。

## 3. ガバメントクラウドでのAIエージェント活用の制約とその解決策

ガバメントクラウドでAIエージェントを利用する際の制約として、以下5点が挙げられている。

1. データの国内完結と管理（ガバメントクラウドでは基本的に国内サービス利用が必須、国外への重要情報流出防止が必要）
2. 利用状況/コストの把握（基本従量課金のため想定外課金を防ぐ仕組みが必須）
3. DevSecOpsの普及（官公庁でのDevSecOpsは調達・契約・人材・文化など複数の問題があり、未だ先進事例扱い。非決定論的なAIエージェントを扱うためのデプロイサイクル整備も必要）
4. 既存認証連携（ガバメントクラウドは外部IdPとSCIMを参考にしたユーザープロビジョニングを独自実装済みであり、既存管理プログラムとの連携が必要）

（データの取り扱いも別項目として扱われている）

### 3-1. データの国内完結と管理

Amazon Bedrock はクロスリージョン推論プロファイルが既定となっており、国内に閉じた利用に完結できない場合がある。対策として、国内専用の推論プロファイルを指定し、AWS Organizations の SCP（Service Control Policy）を併用して国外リージョンでのLLM利用を制限する構成が示された。

**データの取り扱い**: Amazon Bedrock は、サービスページ上でAIサービス（Bedrock）への入力データを学習に利用しないことが明示されている（参考: 「Amazon Bedrock のセキュリティ、プライバシー、責任あるAI」）。その他のAIサービスもオプトアウト設定により入力データの学習利用を防止できる（参考: 「AI services opt-out policies」、「AWS Service Terms 50.1」）。ガバメントクラウドでは、他の機械学習サービスでの学習データ利用をさせないため、Organizationsの設定でAIサービスの学習データ利用をオプトアウトし、新規サービスがある場合も学習データへの利用を制限している。

### 3-2. 既存認証連携

ガバメントクラウドでは AWS IAM Identity Center と AWS IAM ロールを併用し、長期アクセスキーなしで一時クレデンシャルの自動ローテートの仕組みを整備している（参考: 「デジタル庁GCASガイド ユーザー管理方法(AWS編)」）。認証認可情報の取得には、許可セットから別のIAMロールへのスイッチロールが必要となる。

**具体的な実装（Claude Code の例）**:

- ローカルPC上の Claude Code / `settings.json` / `.aws/config` に、許可セット（上位）とスイッチ先IAMロールを別途記載
- Claude Code起動時に自動でAWSアクセスポータルがブラウザで開かれ、ユーザーがログイン操作を行う（一定時間は自動ローテーション）
- 一時トークンを自動取得し、永続的なトークンは使用しない
- 推論リクエスト時は `bedrock-inf` 権限を持つIAM Roleを経由してAmazon Bedrockへアクセス

### 3-3. DevSecOpsの普及

デジタル庁は「政府情報システムにおけるセキュリティ・バイ・デザインガイドライン」（参考: 「デジタル社会推進標準ガイドライン」）でDevSecOpsの取り組みを推奨しているが、公共分野での実現は未だ先進事例として扱われている。開発サイクル全体（Code / Build / Test / Release / Operate / Monitor、Development・Security・Operationsの各観点）のうち、コード作成はあくまで一工程に過ぎず、承認・テスト・リリースを含むプロセス全体が最適化されなければ機能がユーザーに届く速度は変わらない。AIエージェントによる素早い開発サイクルを生かすには、DevSecOpsの実現が不可欠であるとしている。

**GCASファミリー**: ガバメントクラウドの利用ライフサイクル（登録申込・開発・運用）にわたる利便性・セキュリティ向上のための各種サービス群。GCAS AuthN (w/ ZeroTrust)、GCAS Directory (w/ Review)、GCAS DevStack (git service)、GCAS Connect for multi cloud、GCAS Auditcollect、GCAS Dashboard、GCAS Billinghub などで構成される。このうち **GCAS DevStack** が、開発フェーズでの利便性・セキュリティ実現に役立つサービス（ソースコード管理・CI/CD等を提供予定）として紹介された。

#### ハーネスエンジニアリングの参照とユーザーハーネスの整備

開発運用フェーズでのAIエージェント活用を効率化するための方法を検討中であり、以下の「ハーネスエンジニアリング」の考え方を参考に整理している。

> 参考文献: 「Harness engineering for coding agent users」https://martinfowler.com/articles/harness-engineering.html

このハーネスエンジニアリングの考え方は、以下2つの機構に整理される。

- **フィードフォワード**: エージェントの行動を方向づける機構
- **フィードバック**: エージェントの動作後に状況を監視し、自己修正を支援する機構

これを踏まえ、「ユーザーハーネス」という観点で品質向上の仕組みの整備を開始している。整備できたユーザーハーネスをGCAS DevStackから利用可能に整備し、全体の品質向上を目指す方針。ユーザーハーネスの構成要素として以下が挙げられている（LLM = Claude Opus/Sonnet with Bedrock、AIコーディングエージェントor SDK = Claude Code / Claude Agent SDK / Kiro / その他 の上に位置づけられる）。

- プロジェクトごとの Memory / Skills / MCP
- CSPやSaaS側提供のSkills / MCP活用
- Linter / Formatter / Feature Test / Security Test
- 実行環境、外部環境へのアクセス権限

（なお、AIエージェントのセキュリティについては、別セッション「政府の生成AI基盤『源内』－ガバメントクラウド上でのAI実装とAgentCoreによるエージェントAIへの進化」を参照するよう案内されている。）

**品質強化のための静的解析・セキュリティテスト**:

- Linter / Formatter: インデントやコードが満たすべき構造、階層ごとのコード間依存関係の逸脱防止などを決定論的に定義
- Unit Test / Integ Test: 機能面でサービスが満たすべき仕様を定義。AIエージェント任せの場合は不要なテストが増大するため、効果的なテストかどうかやシンプル化を並行検討
- Security Test: 生み出したコードが脆弱性を生まないよう定期的にチェック。AIエージェントによる検査と決定論的チェックが可能なツールを併用し、今後は「Claude Mythos」など高度自立型AIによるチェックも想定

これらの仕組みをローカルとCI/CD環境の両方に実装し、常に定期的に検査が走る状態を維持する。CI/CDを組んでいない場合は、最低限のガードレールとしてAIエージェントの **Hook機能** で検査を共通化する。

**CSPやSaaS側提供のSkills / MCPの活用**:

| ツール | 内容 |
|---|---|
| Agent Toolkit for AWS（`github.com/aws/agent-toolkit-for-aws`） | AIコーディングエージェントがAWSサービスを利用するために必要なツール・知識・安全対策を提供するスキル。プロンプトメインで導入しやすい |
| AWS Knowledge MCP Server | AWSドキュメント・APIリファレンス・トラブルシューティングガイドラインなどへリアルタイムアクセスできるMCPサーバー。最新情報取得によりハルシネーションを抑制 |
| claude-code-guide（サブエージェント） | Claude Code自体の仕様をドキュメントから確認するエージェント。AIエージェント自身についての質問時のハルシネーションを抑制 |

**プロジェクトごとの Memory / Skills / MCP**:

- Memory: AIコーディングエージェントへ常時／状況に応じて与えるプロンプトを管理。ユーザー定義部分とエージェント側自動設定部分がある
- Agent Skills: 独自知識となるプロンプトやプログラム等をまとめて実行。設計書生成、ログからの障害調査などで型化可能。`skill-creator` スキルで簡単に作成・テスト可能
- MCP Server: AIエージェントと外部情報を接続するプロトコル。クラウド自体やSaaS等への接続が可能で、自律的な調査・外部操作を期待する場合に有用

プロジェクトごとに必要なものは異なるためプロジェクトに委ねる方針だが、利用可能なスキル・MCPの制御は管理が別途必要なため検証中としている。

**ユーザーハーネスに関する今後の展望**: ユーザーハーネスという観点での環境整備を進行中であり、ハーネスだけでなくGCAS DevStackとしての開発用環境も整備中。開発と合わせて、運用保守へのAIエージェント活用も推進予定。

**運用視点でのAIエージェント活用**: ローカルに権限を与えるのが難しい場合は AWS DevOps Agent が有効。MCP経由で外部リソース（ソースコード管理サービス、Remote MCP Servers、Amazon Bedrock AgentCore Gateway経由の任意のAWSリソース、その他サービス）と接続し、AIエージェントへフィードバックが可能。アクセス権限の付与が難しい環境ではこちらを選択する。

### 3-4. 利用状況/コストの把握

Bedrockの請求はモデル単位で確認可能だが、個人ごとにAWSアカウントを分離すると請求分離はできる一方でアカウント数が膨大になり、コスト把握の負担が増す課題がある。

**アプリケーション推論プロファイルによる分離**: 従量課金が前提のため、予算確保や異常な高額利用者の定期検査のために請求の詳細な分離が必須。1つのAWSアカウント内で利用者単位に請求を分離するため、アプリケーション推論プロファイルとコスト配分タグを活用し、利用者/利用LLM単位で課金額を分離する（参考: 「Amazon Bedrockを使用した生成AIのコストと使用状況の追跡、配分、管理」）。Amazon CloudWatch Dashboardによる可視化も検討したが、App推論プロファイル名とユーザー名の動的な紐付けに実装が必要なため今回は除外している。

**最新の課金状況把握の方法**: データエクスポートでCUR 2.0形式かつ「Include caller identity (IAM principal) allocation data」をONにするだけで、利用者単位での分離が可能になる。これによりユーザー追加ごとのApp推論プロファイル設定・タグ付けが不要になる（参考: 「IAMプリンシパルベースのコスト配分でAmazon Bedrockのコストを呼び出し元ごとに追跡する」）。ただし現状、AWS Cost Explorer上ではIdCユーザーのIAM principal単位の可視化はできない点に注意が必要、との補足があった。

## まとめ

デジタル庁はガバメントクラウドだけでなく、AIエージェントを活用し、迅速・柔軟・セキュア・コスト効率の高い公共情報システムの構築を進めるための環境整備を進めていく。（AIエージェント活用環境整備 → 開発/運用スピードおよび品質/セキュリティ改善 → 高品質なサービス提供、という流れで締めくくられる）
