# ハーネスエンジニアリング入門 〜モデル単体ではできないことを、周辺で動かす設計〜

- 登壇: 西岡 賢一郎（DataInformed CEO / D-stats CTO）
- 場: AI社会実装勉強会 第58回（2026/04/30）
- 元ファイル: `20260428_ハーネスエンジニアリング入門.pdf`（29p）

## つかみ：毎回同じ注意をしていないか

「commit前にlint走らせて」「rm -rf 絶対やめて」「このチケット見てから着手して」——Claudeに毎回言っている注意を、**言わなくても効く仕組みに変える**のがハーネスの発想。

- **単一エージェント**（ハーネスなし）: $9・20分 → 動かないプロトタイプ
- **3エージェントハーネス**（Planner/Generator/Evaluator＋Playwright MCPで実機検証）: $200・6時間 → 遊べるゲーム
- ※この差は「ハーネス」というより「マルチエージェント設計」の効果だが、いずれも"外側で支える"思想の表れ（出典: Anthropic「Harness design for long-running application development」2026/3）。

## ハーネスの定義

- harness ＝ **「Claudeを呼ぶループ＋ツール呼び出しを実インフラに繋ぐ仕組み」**（Managed Agents, 2026/4）。
- 本質は**「モデル単独ではできないことの"仮定"を符号化したもの」**（"every component in a harness encodes an assumption about what the model can't do on its own"）。
- イメージは馬具の harness——モデルを支え、暴走させずに走らせる周辺装備。

## AI実装手法の進化アーク（発表者の整理）

| 時期 | 手法 | 一言でいうと |
|---|---|---|
| 〜2024 | Prompt Engineering | どう書けば期待通り動くか（Few-shot / CoT / Role prompting）。会話のドライバーは人間 |
| 2025/9 | Context Engineering | 何を入れて何を入れないかを設計する。コンテキストは限界効用逓減の有限資源（コンパクション/context rot/サブエージェント） |
| 2025/11 | Harness 設計 | セッションが切れても仕事を継続できる仕組み（★メイン） |
| 2026/4 | Managed Agents (meta-harness) | ハーネスを載せるホステッドサービス（インフラ層） |

※ この段階整理は発表者独自のもの。Anthropic自身は段階進化として打ち出していない。

### ハーネスなしの素のClaudeが陥る3失敗

1. **一発書き**: アプリ全体を一発で書こうとしてコンテキスト切れで止まる
2. **E2E未確認**: 書き換えるがend-to-endで動くか確認しない
3. **早期完了申告**: テストを1つ通したら「完了です」と申告

（比喩: シフト制で働くエンジニア。新人が来るたびに前シフトの記憶がゼロ）

## 今日の主役：Claude Code hooks（ハーネス思想の軽量実践）

- プロンプトは"お願い"、hooksは"ルール"（決定論的に強制できる）。
- 28イベント / 5ハンドラ種（command / http / mcp_tool / prompt / agent）。今回触るのは PreToolUse と SessionStart。
- 設定の4階層で個人→チーム→組織へ昇格できる経路が組み込まれている:
  - `~/.claude/settings.json`（個人専用）
  - `.claude/settings.json`（チーム共有・git commit）
  - `.claude/settings.local.json`（ローカル上書き・.gitignore推奨）
  - managed policy（組織で強制）

### 実装例

- **PreToolUse で危険コマンドを止める**: `rm -rf` や `DROP TABLE` を検知して `exit 2` でブロック（matcher=Bash、jqでstdinのJSONを読む）。
- **PostToolUse で lint 自動実行**: Edit/Write直後に編集ファイルへ `eslint --fix`（prettier/gofmt/black等でも可）。
- **SessionStart で動的コンテキスト注入**: 起動のたびに現在ブランチ・未コミット数・最終コミットを流し込む（CLAUDE.mdの静的情報と対で使う）。

## 重い vs 軽い（同じ思想の両端）

| | 重い: autonomous-coding | 軽い: Claude Code hooks |
|---|---|---|
| 規模 | Python約700行＋Markdown約1000行 | settings.json 数行 |
| 効果 | 200+ feature を無人実装 | rm -rf を1個防ぐ |
| コスト | 数十時間・数百ドル | 0円・数秒 |
| モード | 無人で動く | 人と並走 |

- 共通項は「モデル単独ではできないことを"周辺で補う"」。規模の差は予算の差、思想は同じ。まずは軽い方から。
- autonomous-codingは**Python＝骨格 / Markdown＝知能**の二段構造（Session1=Initializer、Session2以降=毎回fresh contextで復帰するCoding agent）。feature_listをJSONにするのは「Markdownだと勝手に編集するから。passesフィールドの更新だけ許可」＝プロンプト設計が信頼の設計になっている。

## ハーネス設計の4原則

1. **最小権限**: 危険なツール呼び出しを事前に止める（本番DB事故を技術的にゼロに）。正規表現マッチは入門用、本番はif/MCP構造化判定/LLM judge。
2. **可観測**: 何が起きたか後から再現できる（PostToolUseで全Bash実行をaudit.logに追記）。金融・医療・公共では監査要件そのもの。
3. **失敗前提**: 1つ落ちても全体は死なない（cattle-not-pets）。Stop hookでテストを走らせ、緑でなければ終われないよう強制（①一発書き・③早期完了申告を同時に潰す）。
4. **足し算ではなく引き算（剥がす運用）**: ハーネスはモデル進化で陳腐化する（"Harnesses encode assumptions that go stale as models improve."）。例: Opus 4.6リリースでsprint構造を削除しても性能維持。半年ごとにhooksを読み直し不要なものを消す。

### ただし思想は陳腐化しない

個別のhookは陳腐化するが、「外側で支える」設計スキル自体は陳腐化しない。むしろモデルが賢くなるほど、より複雑なタスクに挑むハーネスを作る余地が広がる。**「剥がす目」を持ち続けることが一番のメッセージ**。

## 行動喚起（3段ロードマップ）

- **今すぐ5分**: rm -rf ブロックを1個コピペ、`claude --debug`で確認
- **今週中**: SessionStartでgit状態注入、動いたhookをチームに共有
- **今月中**: `.claude/settings.json`に移してgit commit、個人hook→チームhookに昇格

役割別: エンジニア個人＝自分の事故を可視化、チームリード＝チーム標準をgitで可視化、事業責任者＝取り組み状況を聞く。

## 結び

ハーネス＝モデルへの「信頼の前提条件」。AIが暴走しない仕組みを人が用意する。プロンプトでお願いするのではなく、ルールとして決定論的に保証する。
