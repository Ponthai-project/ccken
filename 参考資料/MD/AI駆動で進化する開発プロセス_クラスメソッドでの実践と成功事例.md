## 発表の背景

クラスメソッドは「自社開発」と「伴走型開発（顧客システムを共に開発）」の両方に大規模に取り組んでおり、両方の視点で得た技法を紹介する構成。

- 会社文化として、知見や経験をオープンに公開する「DevelopersIO」（約40,000本の技術記事、最低限のルールで自由に発信）と、技術情報共有コミュニティ「Zenn」を運営。生成AI関連記事も2023年の話題時に月100本以上、MCP関連も60件以上公開。

## コード生成AI／AIコーディングエージェントの整理

| ツール | 特徴 | 導入状況 |
|---|---|---|
| GitHub Copilot | VS Code等に搭載するAIコーディングアシスタント。Tab補完・Agentモード | 全社導入 |
| Cursor | VS Codeフォーク。コード全体をインデックス化し優れた推論結果を生成 | 全社導入 |
| Devin | 自立型ソフトウェアエンジニアAI。Slack/UIから自然言語で指示、非同期で処理 | 一部導入 |
| Windsurf | VS Codeフォーク。コード間の関連性を抽象構文木として理解 | 一部導入 |
| Cline | VS Code拡張機能。Bedrock/Claude/Gemini/OpenAI等を選んで実行可能 | 一部導入 |

**AI駆動開発とは**: 提案・要件定義・設計・実装・テスト・運用というソフトウェア開発の全工程に生成AIを積極的に組み込み、スピード・品質・効率を飛躍的に高める開発手法（まだ厳密な定義はなく各社模索中）。

## 自社開発の例

- **Zennでの活用**: Zennの実装にDevinを活用。「pit of death（抜け出せない穴）」に落ちないようタスクのスコープを狭め、Knowledge（プロンプト）でLint・自動テスト・開発ガイドを整備。一部機能の7〜8割はDevinが実装し、実装の合間に既存コードのテストも生成。
- **自社テンプレート作成**: ICASUアーキテクチャ（AWS CDKテンプレート）作成でDevinを一部活用。小さめのタスクで効果を発揮し、IaCのセキュリティ対応の指摘作成・微修正、スナップショットテストの自動更新、DeepWiki/Searchでのノウハウ共有に利用。

## 伴走型開発の例（工程別）

- **仮説提案**: 顧客ヒアリングをMeetで録音・文字起こし → DeepResearchで競合調査 → ChatGPT/Claude・Cursor・Notion MCPで要件整理 → Cursor＋プロンプトでプロトタイプ作成・デモ環境へのデプロイ
- **設計／実装**:
  - 全工程にCursorを組み込み、要件のOpenAPI Specification化、機能命名の案出し、ドキュメント/コードレビュー依頼、Tabによるコード/ドキュメント生成（体感約30〜50%生産性向上）
  - 一部PJでやらなくなったこと: シーケンス図等のMarkdown手動生成、Cursor RuleベースでAgentにコードを書かせること（Rule作成とのトレードオフ、適切なタスク分割が必要）
  - OSS調査にDeepResearch/Perplexity、DeepWikiでOSSコードをWiki化・Chatbotに質問しシーケンス図生成で処理概要を把握。クローズドソースもDevin Wiki/Search(Beta)で解析可能
- **テスト**: GitHub Copilotでテストコードを1つ実装 → デシジョンテーブル整理 → それに合うテストコードを追加生成。CI/CDにはCodeRabbitやPR-Agent（生成AIレビュー）を導入。ブログレビューにも活用
- **運用／保守**: `code2prompt`でリポジトリをプロンプト化し、最大入力トークン数約100万のGeminiに渡して設計書生成・質問（大容量コードベースにも対応）

### Rule作成のトレードオフ

Rule/Knowledge整備とタスク分解にかかる時間 vs. 定型実装・運用・レビューに費やす時間の天秤。後者が長いプロジェクトほど前者への投資価値が高い（恒常タスクや実装例が豊富な大規模PJほど有利）。

### 変わらない本質

Rule作成の営みは「プログラマの三大美徳（怠惰・短気・傲慢）」的な自動化の延伸に過ぎず、エンジニアの本質的な作業（影響範囲を絞り、繰り返しの自明な作業を減らす）自体は変化しない。

## 従来像 → 開発の将来像

- **従来**: ビジネス側が仕様を確認したい場合、開発者がドキュメント/ソースを都度確認して回答。大規模PJでは長年在籍するエキスパートしか正しい判断ができず、情報収集にラグが生じる。
- **将来像**: 設計ドキュメント・コード・PJ管理ツールという1つのデータソースに情報を集約し、Devin Wiki/SearchやNotebookLMのような「検索/生成エージェント」を介して、ビジネス・QA・開発者が直接情報を参照・生成できるようにする。

## 実験的な取り組み

- **全社導入時の課題**: Bedrock利用時にユーザごとの利用料按分ができない、プロンプトキャッシュ未使用だと高額、海外リージョン制限時の統制が必要
  - 解決策: ユーザ単位のアプリ推論プロファイル作成、プロンプトキャッシュの一括有効化、リージョン制限を考慮したAPI提供
- **Clineを大規模利用する場合のアーキテクチャ**: AWS IAMでユーザごとにIAM User/AccessKeyを発行し、Bedrockのアプリ推論プロファイル＋CostTagで按分。AWS CUR/Cost Explorer/CloudWatchでコスト・利用状況を監視
- **AIエージェントへの強い権限付与の検証**: AWS CDKでDeploymentActionRole → CloudFormationExecutionRole（Permissions Boundary付き）→ AWS Resourcesという権限委譲チェーンと、Read専用ロールを分離し、将来的なクラウド操作の自律化に向けたガードレールを検証中

## Vibe Codingについて

- **定義**: OpenAI共同創業者Andrej Karpathyが提唱。LLMに自然言語でコーディングを依頼し、動作するコードを生成・実行するスタイル。バイブス任せでエラー解決もLLMに委ねる。Wikipediaにも項目化。
- **The 70% problem**: 非エンジニアがAIでコーディングすると70%までは驚くほど速く到達できるが、残り30%は労力をかけても進まなくなるという指摘。
- **「居眠り運転」現象**: ハーバード大学とBCGの調査で、AIが解決しにくいタスクにおいて高精度AIを使う群の成績が低精度AI使用群より19%低下した（過信によるチェックの手抜き）。コーディングでも同様のリスクがあると指摘。
- **難しさの具体例**: AWS CDKで`readonlyRootFilesystem: true`と1行書くだけでも、「なぜ必要か」「ECS Execとの整合性」「ミドルウェア/OSパッケージへの影響」「検知・解消方法」等の理解が必要 → **Vibe Codingでコード理解が不要になるわけではなく、ソフトウェアエンジニアリングを深く理解する必要性は変わらない**。
- **適正利用の方向性**: アイデア出し（試作・提案資料用プロトタイプ）、実装（未知技術の仮実装で学習後に作り直す）、レビュー（実装コードをAIエージェントに投げて指摘をもらう）という用途分けが有効。

## 研修の必要性（宣伝パート）

- MS/Accentureの5,000人規模のGitHub Copilot導入実験で、研修を行った群（Treated）は導入率69.4%、導入のみの群（Control）は24.4%。ツール導入だけでは実際に使われない可能性がある。
- クラスメソッドはAI駆動開発支援サービス（AI駆動PoC/MVPサービス、AI駆動開発実践プログラム／研修・ワークショップ）を提供中。
