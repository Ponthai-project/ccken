## エンジニアリングの4層

Prompt Eng.（指示を作り込む）→ Context Eng.（見せる情報を管理）→ **Harness Eng.（モデルを動かす仕組み全体・本日のテーマ）** → Loop Eng.（ループ自体を書く）。

## ハーネスとは（起点となった定義）

Anthropic公式動画（2025/9/2, "Building the future of agentic coding with Claude Code"）でのBoris Cherny（Claude Code開発者）の言葉:

> モデルを直接使うことはできない。ハーネスを使う必要がある。Claudeは馬で、鞍のようなものが要る。

ハーネス＝渡すツールからコンテキスト管理までモデルの周りぜんぶ。つまり**「Claude Code自体がハーネス」**。

## ハーネスの概念図（3層）

| 層 | 内容 | 自分で設計できるか |
|---|---|---|
| ① モデル | 推論エンジン。コンテキストから出力を決める（Opus / Sonnet / Fable） | 選択のみ |
| ② 内部ハーネス | Claude Codeのエージェントループ・ツール・サンドボックス・コンテキスト管理。モデルを「エージェント」に変える層 | 提供物を使う |
| ③ 外部ハーネス | CLAUDE.md / スキル / hooks / スラッシュコマンド / MCP / サブエージェント。内部ハーネスの上に搭載 | **設計・改善できる主戦場** |

- 同じ「Claude Fable 5」でも Cursor / Devin と Claude Code では別物。**ハーネスが違えば挙動が変わる**。
- モデルと内部ハーネスの進化は速い。かつて自作のスラッシュコマンド・hookで補っていた機能が、公式機能としてリリースされることがある（例: Artifacts）。

## 万能なハーネスはない

競走馬のハーネス（検証ゲート＋専用スキル＋hooks＋MCP）と、乗馬クラブの鞍（最小限の設定）は別物。**人のハーネスが自分に合うとは限らない**。

## ハーネスで意識すること

1. **小さく保つ**: 足すほど不具合の切り分けが難しくなる。効いていない設定は消し、定期的に棚卸し。
2. **ポータブルに**: 特定ツール専用の書き方に依存させない。Markdownのスキル・手順書なら内部ハーネスが変わっても持ち運べる。
3. **まず使い倒す**: 内部ハーネス＋モデルの素の実力で足りることは多い。足りないと分かった分だけ外部ハーネスを足す。

## まとめ

1. 自分が作っているのが「どの層」かを意識する
2. まず「内部ハーネス＋モデル」を使い倒す
3. 外部ハーネスは「小さく、ポータブルに」
