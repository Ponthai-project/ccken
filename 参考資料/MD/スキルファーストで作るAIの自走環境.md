## 背景：経理AIエージェント「TOKIUM」

AIとプロスタッフが連携し経理作業を丸ごと代行。「人が操作するSaaS」から「AIエージェントが実行する業務」へ——SaaSはAIエージェントが参照・操作するためのデータベースになり、基本的な判断はAIが行い、人は例外対応・承認だけを行う未来を目指す。

## 業務での実践事例と、そこで生まれた苦しさ

新規PJで運用中の代表的なハーネス:

- **jira-to-pr**: チケット→実装→Pull Request作成まで自動化。3人チームで週30〜40 PRを捌く。
- **self-review**: AI生成コードの品質磨き込み。
- **code-review**: マージ前ガード＋変更意図のチーム共有知化。

コーディングやレビューの自動化率は高いが、**タスクの整理や細かな指示出しがまだ人手のまま**。承認ポイントが各所に残りAIに任せきれず、結果として開発者の手は常に動いている状態。「なんかAIの奴隷みたいで苦しい」→先輩の助言「**スキルファースト**で作ればいい」。

## 個人プロジェクトでの検証：COLOR M@STER

アイドルの個人カラーを表示しスマホをペンライト化するWebアプリ（Kotlin Multiplatform、丸1年放置）。ライブラリ更新・Firebase→Cloudflare/Cloud Run移行・JSビルド→Wasmビルド置換などをやりたい。

### 進め方（最初に課した制約）

1. やりたいことを思いつくだけ全部書き出す。
2. やりたいことが全て**スキル経由**で実現されるようハーネスの構想を練る。
3. 課した制約: ①「スキルの組み合わせ」で全タスクが実行できるようにする ②いま解決不要な不明点・意思決定は記録して後回し ③Pull Request 1件ごとに必ず振り返りスキルを実行し、スキル・規約を自己改善させる。

### 結果

- 雑多な要求は最終的に**1,753行のMarkdown**に。ロードマップは3ステップ（初期化→ハーネス構築→実装）。
- Claude Codeに作業させると、**200+のハーネス関連Markdown＆約70件のPR**が、ほぼ指示なく追加＋レビュー＋マージされるように。真面目にレビューしたのは数PRのみ。

## 業務PJ（ルール・規約ファースト）と個人PJ（スキルファースト）の違い

| | 業務（これまで） | 個人（今回） |
|---|---|---|
| 何から作る | ルール・規約ファースト | スキルファースト |
| 初期の解像度 | 高（理想を頭で描いてから） | 低（まずやってみる） |
| 暗黙知→形式知 | 人間が言語化しAIに伝える | AIが提案し人間が承認 |
| 改善サイクル | 後付けになりがち | 最初から組み込み |
| 手戻りコスト | 大（理想が間違っていると痛い） | 小（小さく試して直す） |

- **枯れた技術が相手なら「ルール・規約ファースト」が有効**。
- **日々進化し非定型入力もカバーできるAIなら「スキルファースト」が有効**（AIがより多くの知識にアクセス可能な状態を最短で作れる）。

## まとめ（※まだコードは書かせておらず仮説段階）

- AIの自走環境は、AI自身に作らせよう。AIはマイクロマネジメントせず人間の部下のように育てる。
- 伝えるのは**ゴール・方針・意図の3つ**。Howは細かく口出しせず評価に留める。
- 今決める必要のない意思決定は記録して後回し。
- 成果物はAIにこまめに振り返りさせ、スキルを自己改善。スキルの自己改善の仕組み＋スキル経由の指示徹底で、暗黙知が形式知に変わり続ける仕組みが作れる（はず）。
