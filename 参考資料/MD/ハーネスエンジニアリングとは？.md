# ハーネスエンジニアリングとは？ 〜5つの流派：実装が先、命名は後〜

- 登壇: KINOPEE（Kinopee）
- 日付: 2026/4/24
- 元ファイル: `ハーネスエンジニアリングとは？.pdf`（22p）
- テーマ: 同じ「ハーネス」を語っても、論者が原器にしている実装で重心が違う。定義の揺れを現場感覚で読み解く。

## ハーネスエンジニアリング 5つの流派

| 流派 | 代表者 | 分類 | 中核主張 |
|---|---|---|---|
| **Chase派** | Harrison Chase (LangChain) | 広義・分類論 | モデル以外のすべてがハーネス |
| **Hashimoto派** | Mitchell Hashimoto (HashiCorp) | 運用哲学 | 失敗したら環境を再設計せよ |
| **Fowler派** | Martin Fowler (Thoughtworks) | 制御理論 | コンテキスト設計の特殊形として定義 |
| **Chawla派** | Avi Chawla (教育系) | 同心円モデル | LLMのOS層、三重の入れ子 |
| **Codex派** | Ryan Lopopolo (OpenAI Codex) | スケール駆動 | スケールで壊れる問題の答え |

### 1. Chase派（広義ハーネス / 分類論）

- 出典: 2025年10月ブログ「Agent Frameworks, Runtimes, and Harnesses」。原器はDeepAgents (LangChain)。
- framework/runtime/harnessの三分類を提示し、ツール実行・メモリ・サブエージェント・ガードレール・オーケストレーションまで全部ハーネスに畳み込む。ハーネスは「コンテキストを生成する外側の層」で、コンテキスト設計はその一構成要素。
- 特徴的フレーズ: **"If you're not the model, you're the harness."**

### 2. Hashimoto派（運用哲学 / 失敗駆動）

- 出典: 2026年2月の自身のブログ。原器はAGENTS.md / CLAUDE.md / .cursorrules。
- エージェントが失敗したら物理的に同じミスができないよう環境を再設計せよ。AGENTS.mdに一行ずつ失敗防止ルールを追加していく「運用の累積」がハーネスエンジニアリング。設計物ではなく育てていくプロセス。日本の.cursorrules運用者と地続きで現場感覚に最も近い。
- ループ: 実行→失敗→AGENTS.mdに1行追加→再発不能。**失敗のたびに環境が強くなる**。

### 3. Fowler派（狭義ハーネス / 制御理論）

- 出典: 2026年 Thoughtworks記事「Harness engineering for coding agent users」。原器はコーディングエージェント全般。
- **ハーネスをコンテキストの内側（特殊形）に位置づける唯一の派**。guides（feedforward制御＝不要な出力を事前に防ぐ）とsensors（feedback制御＝逸脱を検知して自己修正させる）の両輪でエージェントを信頼に足るものにする。制御工学の語彙に接続する理論志向。

### 4. Chawla派（同心円モデル / OS比喩）

- 出典: 2026年前半「The Anatomy of an Agent Harness」ほか。概念モデル。
- プロンプト ⊂ コンテキスト ⊂ ハーネスの三重同心円。Beren Millidge (2023)のOS比喩を継承——**LLMはCPU、コンテキストはRAM、ツールはデバドラ、ハーネスはOS**。初学者への説明しやすさを優先。

### 5. Codex派（スケール駆動 / プロダクション答え）

- 出典: 2026年2月 OpenAIの事例（3人で百万行、95%AI生成）。原器はOpenAI Codex。
- プロンプトでもコンテキストでも解けない、多段ステップ・自律実行・並列特有の故障モードを解くのがハーネス、というスケール駆動の動機づけ。「なぜ今この概念が要請されたか」の導入に使いやすい。
- 特徴的フレーズ: **"Agents aren't hard; the Harness is hard."**

## 対立の構造（包含関係の逆転）

流派間の最も劇的な噛み合わなさ:

- **Chase派 / Chawla派**: ハーネス ⊃ コンテキスト（総合フレーム設計の視点）
- **Fowler派**: コンテキスト ⊃ ハーネス（制御理論 feedforward/feedback の視点）

→ 日本に紹介する際は、どちらの意味で語られているかを毎回明示する必要がある。

## 補論：ハーネスとガードレール（役割の違う2つの仕組み）

| ハーネス（＝馬具） | ガードレール（＝道を外れない柵） |
|---|---|
| 走り出す前に「どう走ってほしいか」を伝える事前設計の層 | 走った後に「逸脱していないか」を機械で自動検証する事後検証の層 |
| カスタムインストラクション/プラン、Skills、サブエージェント | 静的解析(lint/型)、ビルド、テスト |
| 主観 | 客観 |

- **機械的な判定が強いほど、エージェントにハンドルを預けられる**（自律性を上げる鍵）。
- 馬具と柵だけでは目的地に着けない→**フィードバック・ループ**（走らせる→逸脱を観察→軌道を戻す。テストが落ちたら直させる、lintが出たら修正、型が通らなければやり直し）。
- 主張: 日本の現場は名前がつく前から既にやっていた。**実装が先、命名は後**。

## 結論

言葉の定義は重要ではない。大切なのは期待した結果を得ること。今使える手段（馬具も柵もループも）を総動員し、適切な場所で適切に使って良い結果を得る。
