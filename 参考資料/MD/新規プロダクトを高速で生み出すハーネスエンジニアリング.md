## 1. 2025年：AIでコードはすぐ書けるように

- 意図を伝えればAIが設計・実装・テストまで書く。人間は仕様確認と品質判断に集中。Layermateのリリースまでの実工数は**約1人月**。
- しかし——細かい指示が必要、AIが過去の文脈を忘れる、同じバグを何度も埋め込む、権限・セキュリティ・DB変更は入念なレビューが必要。**最初作るだけなら速いが、低工数で運用し続けるには環境整備が必要**。

## 2. ハーネスエンジニアリング：スムーズなAI駆動開発の土台作り

**AIエージェント = Model（賢さ）+ Harness（環境）**。設計するのはHarness側。

| 設計軸 | 問い | 例 |
|---|---|---|
| コンテキスト | 何を読ませるか | CLAUDE.md、設計ログ、アーキテクチャ文書 |
| アクション | 何を許可するか | 権限、ツール、サンドボックス |
| フィードバック | どう失敗に気づかせるか | test、lint、typecheck、CI、drift detection |
| 運用 | どう継続的に改善するか | rules、PRのWhy、設計ログ |

**ハーネスの効用**: 壊さない（権限分離/サンドボックス）、間違いに気付ける（実DBテスト/CI/設定ズレ検出）、レビュー疲れしない（rules/CodeRabbitが先に弾く）、大雑把な指示で自走（CLAUDE.md/階層化文脈）。→人間レビューなし＋大雑把な指示でAIが自走し、0→1サイクルが爆速化。

## 3. 実践：整備したハーネス

技術スタック: Next.js / Hono / MobX / PostgreSQL + Drizzle / Firebase Auth / Google Cloud

### AIのミスを食い止める防御機構（多層防御）

| 層 | 防御方法 | 止まる例 |
|---|---|---|
| 事前に書かせない | CLAUDE.md / rules | AIが最初から正しい書き方を選ぶ |
| そもそも書きにくくする | 権限分離 / ミドルウェア / 型 | セキュアでない実装が書きにくい設計 |
| 動かして検出 | 実DBを使った権限テスト | 権限漏れを実行時に検出 |
| マージ前に止める | CI（型/テスト/設定ズレ） | 問題があればmergeできない |
| botに1次レビュー | CodeRabbit | 機械が先にチェック |
| 人間レビュアー | 判断が必要な箇所のみ | 人間は最後の判断だけ |

### 7つの実践

1. **機能ごとにコンテキストを分割し段階的開示**: フロントとAPIを同じfeatureフォルダに同居させCLAUDE.mdもfeature単位。`.claude/rules/*.md`をファイル種別ごとに自動ロード。全部読ませず必要な範囲だけ渡す。
2. **同じ指摘を2度繰り返さない**: AIは一度指摘したことを別セッションで忘れる→レビュー指摘を随時rulesに移すskillを運用（SQLの書き方・API呼び出し・状態管理・テストの書き方・自動生成ファイルの手編集禁止など）。
3. **RLSでDBレベルのアクセス制御**: Row-Level Securityでワークスペース単位のテナント分離（別テナントのデータはクエリしても0行）。権限はSQLスキーマで宣言的に管理し1ファイルに集約。
4. **テストは「トロフィー型」で**: E2E(Playwright)は遅く不安定→結合テスト（MobX store + Hono testClient + 実DB）をメインに最も厚く。MobXのstoreは描画せずテストできるためE2Eがほぼ要らない。
5. **スキーマは宣言的SQLで管理**: `schema/*.sql`を編集（あるべき姿を宣言）→migrationを差分から自動生成→TypeScript型も自動再生成→CIで「宣言vs実DB」のdriftを検出（単一のsource of truth）。
6. **AIがPR作成→レビュー対応→知見保存するskill**: ドキュメント反映→PR作成（人間向け説明＋AI向け詳細）→CodeRabbit・人間・CIを並行監視→指摘の振り分け→修正＋返信＋学びをrulesに書き戻す。「PR作って」だけであとは全自動。
7. **GitHubで全員がドキュメント管理**: デザイナー/PdM/エンジニアが企画・仕様・UIモック（Next.js）を同じリポジトリにpush→それが実装の起点になる。

### 未解決の課題

- CLAUDE.md/rulesの肥大化対策（重複・矛盾の検出、常時ロードをどこまで絞りskill/子CLAUDE.mdに逃がすか＝**メタハーネスの構築**）
- AIの並行実装（Git worktreeのPORT/DB/env分離が煩雑、何並列までが人間の指示の限界か）

### AIへの指示の変化

- 〜2025: 「ここはこう書く」と細かく指示、指示コスト自体が重い
- 2026〜（ハーネス整備後）: 「この機能を追加して」で文脈読込・実装・権限・テスト・PRまで自走。人間は最初と最後だけ。

## 4. ハーネス自体が、プロダクトになる?

**HaaS = Harness as a Service**: モデルそのものではなく、モデルを取り巻く環境（ハーネス）をプロダクトとして売る。モデルは汎用LLMを利用し、価値の源泉はその周りのコンテキスト/ツール/UI/安全性。

- 実例: コーディング（Cursor / Copilot / Devin）、コードレビュー（CodeRabbit）、デザイン/プロト（v0 / Bolt.new / Lovable）、業界特化（Harvey〈法律〉/ Hebbia〈金融〉）、CS/業務自動化（Sierra / Salesforce Agentforce）。どれもモデル自体は提供せず、その周りの環境を売っている。
- これからのAIプロダクトの価値: 精度・使いやすさ・安全性・継続性（使うほどrules/skillsが育つ）・連携性。**これからのAIプロダクトの正体は、ハーネス（かも?）**。

## まとめ

1. AIに機能を作らせる前に、AIが適切に作業できる環境（ハーネス）を作る
2. 少ない人間のレビュー負担でAIコードを通せる多層防御を組む
3. AIにプルリク作成→レビュー対応までやらせ、学びはrulesに書き戻す
