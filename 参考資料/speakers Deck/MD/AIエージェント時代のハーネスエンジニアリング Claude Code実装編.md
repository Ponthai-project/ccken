## 前回の3つの問いに、今回は5層の実装で答える

前作（概念編）で提示した3つの問いに、本作は5層アーキテクチャの実装ファイルで回答する構成になっている。

| 前回の問い | 今回の対応 |
|---|---|
| 問1: AIに何を・どう渡すか（コンテキスト設計） | 第1層ルール層（CLAUDE.md）でルールを固定／第2層コンテキスト層（MEMORY.md）で文脈を継続 |
| 問2: AIの暴走をどう防ぐか（制約・ガードレール） | 第1層ルール層（settings.json）で権限制御／第3層フック層（hooks/）で実行前後をガード |
| 問3: 品質をどう担保するか（評価・フィードバック） | 第3層フック層（PostToolUse）で出力検証／第5層エージェント・MCP層でレビュー担当を分離 |

概念を型（ファイル）に落とすことがエンジニアの仕事、という立場が本編の主張。

## ハーネスは"ファイル"で定義され、対話で生成できる

- ハーネスエンジニアリングの成果物はファイルである。CLAUDE.mdがAIのルールを、`hooks/*.py`がAIの行動制御を、`skills/*.md`がAIの仕事の型を、`agents/*.md`がAIのチーム設計を、それぞれ定義する。
- これらのファイルはClaudeとの対話で生成でき、エンジニアでなくてもコードを書かずに「何をしてほしいか」を伝えるだけで作れる。
- 対話→生成の実例3つ:
  - **CLAUDE.md**: 「`/init`でCLAUDE.mdの雛形を生成して。ロールはキャリアコンサルタント、出力は日本語、git pushは承認なしに実行しない、と追記して」→ `/init`実行→CLAUDE.md生成→追記完了
  - **hooks**: 「Bashコマンド実行前に`rm -rf`や`--force`が含まれていたら実行をブロックするフックを作って。`/hooks`で設定も更新して」→ `check_dangerous.py`生成＋`settings.json`自動更新
  - **skills**: 「note記事の執筆フローをスキルにしたい。リサーチ→執筆→校正→投稿の順で、各ステップで私の承認を入れて。`/draft`で呼び出せるようにして」→ `.claude/skills/draft/SKILL.md`生成

"設計する人"になるのにコーディング経験は不要で、必要なのは「何をしてほしいか」を言語化する力、とまとめられている。

## 実装の全体像：5層アーキテクチャ

5層は「制御の深度」で積み上がる構造であり、上位層ほど人間の意図に近い。

| 層 | 層名 | 役割 | 設置場所（対話で生成可能） |
|---|---|---|---|
| 第1層 | ルール層 | AIの行動原則・ロール定義＋権限設定 | CLAUDE.md / .claude/settings.json |
| 第2層 | コンテキスト層 | セッション横断の記憶・エージェント別局所文脈 | memory/MEMORY.md / agents/*.md |
| 第3層 | フック層 | ツール実行前後のガードレール実装 | settings.json > hooks / check_*.py |
| 第4層 | コマンド・スキル層 | 繰り返し作業の型化（基本: commands／発展: skills） | .claude/commands/*.md / .claude/skills/*/SKILL.md |
| 第5層 | エージェント・MCP層 | 役割分担AIと外部ツール連携の統合管理 | agents/*.md / settings.json > mcpServers |

まず第1層（ルール層）から始め、深度を上げながら実装していくのが推奨される進め方。

## 実装例① 第1層ルール層（CLAUDE.md + settings.json）

第1層ルール層は"AIへの憲法"であり、構造化すれば行動が安定する。

- **Before（構造なし）**: 「あなたはキャリアコンサルタントです。日本語で答えてください。丁寧に対応してください。」→ ロール・制約・禁止事項がバラバラで、セッションごとに挙動がブレ、AIが何を守れば良いか判断できない。
- **After（構造化CLAUDE.md）**: 「## ロール」「## 中核命題」「## 禁止事項」「## 出力言語」の見出しで整理して記述する。
- 構造化の4要素:
  - **ロール定義**: AIの「役割」を明示。複数ロールを番号付きで列挙する。
  - **中核命題**: すべての行動の拠り所となる1文。行動基準の源泉。
  - **禁止事項**: やってはいけないことを明示することで「暴走」を防ぐ。
  - **出力言語**: 言語仕様を固定することで一貫性が生まれる。
- 作り方: ① `/init`を実行してCLAUDE.mdの雛形を生成 ② ロール・禁止事項・出力言語を対話で追記 ③ 「npmコマンドを許可して」と話しかけてsettings.jsonの権限設定を追加。

## 実装例② 第2層コンテキスト層（MEMORY.md + agents/*.md）

第2層コンテキスト層は"3層で分離"する——全体・セッション・エージェントを分けることで爆発しない。

- 第1層CLAUDE.md＝**全体文脈**（全セッション共通の原則・ロール・禁止事項）
- 第2層MEMORY.md＝**セッション文脈**（セッション横断の記憶・プロジェクト状態・途中経過）
- 第2層agents/*.md＝**局所文脈**（各エージェント専用の役割・制約を最小限に）
- **Before（全情報一括渡し）**: 全ユーザー情報＋全プロジェクト情報＋全記事草稿＋全メモをそのまま渡す→トークン爆発でコスト増大、無関係情報でエラー・幻覚が増加。
- **After（分離）**: CLAUDE.md＝全エージェント共通原則のみ、MEMORY.md＝今セッションの状態のみ、agents/sage.md＝Sage専用の役割のみ→各層が最小限の情報だけを持ち、コスト削減＋精度向上。
- 作り方: ① CLAUDE.mdに「全体文脈」を記述（`/init`で雛形後に追記） ② 「プロジェクトの状態をMEMORY.mdに記録して」と話しかけてMEMORY.mdを生成 ③ 「○○エージェントの役割定義ファイルを作って」と話しかけてagents/*.mdを生成。
- 生成指示例: 「Sageというリサーチエージェントのファイルを作って。担当はリサーチのみ」→ `Sage — Research Agent`（Role/Contextを持ち、全体原則はCLAUDE.mdを参照、セッション状態はMEMORY.mdを参照、Sage固有情報のみ記載）が生成される。

## 実装例③ 第3層フック層（PreToolUse / PostToolUse / Stop）

第3層フック層は"AIのガードレール"である——実行の前後に検査を挟むことで暴走を防ぐ。

- フロー: ツール呼び出し → PreToolUse検査 → ツール実行 → PostToolUse検査 → Stopフック → 完了／Block。
- **Before（フックなし）**: AIが自由にツールを実行→危険なコマンドも即時実行→`git push --force`が承認なしで走る→本番ファイルが意図せず削除される。
- **After（第3層実装済み）**: PreToolUseでコマンドを事前検査→危険パターン検出でブロック＆警告→PostToolUseで出力の品質チェック→Stopフックで最終確認・ログ記録。
- 作り方: ①「〜するフックを作って」と話しかける（例:「危険コマンドをブロックするフックを作って」）②ClaudeがPythonスクリプトを生成＋settings.jsonのhooks設定を自動更新 ③ `/hooks`コマンドで設定内容を確認・調整。
- 生成指示例: 「Bashコマンド実行前に`rm -rf`や`--force`をブロックするフックを作って」→ `check_dangerous.py`（PreToolUseで危険パターンを検出しblockを返す）とsettings.jsonのhooks設定が自動生成される。

## 実装例④ 第4層コマンド・スキル層（commands + skills）

第4層コマンド・スキル層は意図を型化する＝Translation Bridgeの実装。

| 比較軸 | Commands（基本） | Skills（発展） |
|---|---|---|
| 位置づけ | 一般的なClaude Code機能 | 発展的な使い方（Commandsの上位版） |
| 複雑度 | シンプル・1ファイル完結 | 多ステップ・承認ルール付き |
| ファイル形式 | .claude/commands/<name>.md | .claude/skills/<name>/SKILL.md |
| 呼び出し方 | /commit, /review など | /bizslide, /sage など |
| 典型例 | git commitの型化、定型レポート生成 | スライド生成ワークフロー、エージェント協調タスク |

- **Before**: 毎回自然言語で「前回と同じ形式でコミットして。メッセージは○○で。Co-Authored-Byも忘れずに。emojiなしで。」と繰り返し指示し、品質がブレる。
- **After**: `.claude/commands/commit.md`（git diff --staged確認→Conventional Commits形式生成→Co-Authored-By付与／emojiなし、といった手順）を対話で生成しておけば、`/commit`と打つだけで品質が一定になる。

## 実装例⑤ 第5層エージェント・MCP層（agents + mcp-configs）

第5層エージェント・MCP層を設計する＝エージェントアーキテクトの仕事。

- 構成イメージ: ユーザーの一言指示 → Claude Code（Orchestrator、全体を指揮・タスクを振り分ける）→ Researcher（リサーチ担当）／Writer（執筆担当）／Publisher（投稿担当）に委譲 → 各エージェントがMCP経由でWeb検索・Notion（ページ作成）・Slack（チャンネル投稿）等の外部ツールを操作。
- エージェントの作り方: 「○○という役割のエージェントを作って。担当は〜のみ」と話しかける→ `.claude/agents/○○.md`が生成される。
- MCPの接続方法: ①CLIコマンド `claude mcp add`で対話形式によりMCPサーバーを追加 ②対話（「GitHubのMCPサーバーを追加して。リポジトリ読み書きとPR操作を許可して」）でsettings.jsonのmcpServersに自動追加される。

## 実装例⑥ 品質評価・フィードバックループ（第3層＋第5層の連携）

品質評価は"ファイルを1つ書く"ことから始まる——そのファイルも対話で生成できる。

フィードバックループ: ①AIが出力を生成 → ②第3層PostToolUse hookが実行（check_output.py） → ③品質スコアが閾値以下なら自動リトライ → ④第5層レビューエージェントが最終確認 → ⑤結果を第2層MEMORY.mdに蓄積（不合格ならリトライへ戻る）。

| 層 | ファイル | 書くこと |
|---|---|---|
| 第3層 出力検証 | .claude/hooks/check_output.py | PostToolUseで品質チェック |
| 第1層 リトライ制御 | settings.json | hooks.PostToolUseにcheck_output.py登録 |
| 第5層 レビュー担当 | .claude/agents/reviewer.md | 検証専用エージェントの定義 |
| 第2層 学習蓄積 | memory/feedback_log.md | 検証結果・改善点の記録 |

作り方: ①「出力が空のときブロックするフックを作って」→ check_output.py生成 ②「出力を検証するレビューエージェントを作って」→ agents/reviewer.md生成 ③「検証結果をMEMORY.mdに記録して」→ 継続的な改善ループが完成。
生成指示例: 「出力が空のときブロックし、禁止ワードを検出したら警告するフックを作って」→ `check_output.py`（空出力を検出しblockを返す）が生成される。

## まとめ：5層を実装すれば、AIは「ツール」から「チーム」に変わる

1. **第1層ルール層**: CLAUDE.mdを構造化する（ロール・中核命題・禁止事項・出力言語の4要素で記述）
2. **第2層コンテキスト層**: 3層に分離する（CLAUDE.md／MEMORY.md／agents/*.mdの役割を明確に）
3. **第3層フック層**: ガードレールを実装する（PreToolUse＋PostToolUse＋check_*.pyで検査）
4. **第4層コマンド・スキル層**: 意図を型化する（.claude/commands/＋.claude/skills/にワークフローを蓄積）
5. **第5層エージェント・MCP層**: チームを設計する（役割分担＋MCP連携）
6. **第3層＋第5層**: 品質評価ループを自律化する（PostToolUse＋エージェントレビュー＋第2層MEMORY蓄積）

前回3つの問いとの最終対応: 問1（コンテキスト設計）→第1層ルール＋第2層コンテキスト、問2（制約・ガードレール）→第3層フック、問3（品質評価・フィードバック）→第3層PostToolUse＋第5層エージェントレビュー。ハーネスエンジニアリングは5層で実装できる、というのが結論。

## 要点まとめ

- ハーネスの成果物は"ファイル"であり、コードを書けなくても対話でCLAUDE.md・hooks・skills・agentsを生成できる。
- 5層アーキテクチャ（ルール／コンテキスト／フック／コマンド・スキル／エージェント・MCP）は「制御の深度」で積み上がり、上位ほど人間の意図に近い。
- 前作の3つの問い（コンテキスト設計・制約とガードレール・品質評価）は、それぞれ第1〜第3層・第5層の実装ファイルに対応づけられる。
- 第1層CLAUDE.mdの構造化は「ロール定義・中核命題・禁止事項・出力言語」の4要素が核心。
- フィードバックループは第3層フック（自動リトライ）と第5層エージェント（最終レビュー）を組み合わせ、第2層MEMORY.mdに学習を蓄積することで自律的に回る。
- まず第1層から始め、必要に応じて深度を上げていくのが実装の推奨順序。
