## 問題意識：AIがルールを破らない世界は可能か

Claude Code Hooksはルールベースなので人間が考慮できれば抜け漏れは避けられそうだが、**正解自体をAIに書かせようとすると大体ズルをする**——型チェックが通らないと`any`で強引に回避、テストを通すためのテストを書く、等。

## Claude Codeとは

- "Code"の名に騙されてはいけない、ほぼ何にでも使える汎用CLIエージェント。自走力が高く人間の介入が少なくてもタスクを完了できる。
- エンジニアは**「何を作るか」を決め、「どう作るか」は自律型AIに任せる**時代に。

## チーム開発にAIを組み込む時の課題

個人開発・プロトタイプでは対話で決められた「なぜ・何を作るか」も、チームでは多数のステークホルダーが関わり一人では決められない。加えて——人間は要求・要件を言語化できない、レビュー地獄、セキュリティ脆弱性。**人間が関わる箇所がボトルネックになり生産性が根本的に上がらない**。

## そこに現れた「Kiro」

2025/7にAWSが発表した、プロトタイプから本番までを持っていくためのAI駆動IDE。背景にある問題意識は SDLC の3課題（意思決定の複雑化・要件の曖昧性・品質管理）で、これらを解く**AI-DLC（AI駆動開発ライフサイクル）**を実現する。

### AI-DLCの3フェーズとメリット

1. **開始フェーズ**: ビジネス要望を具体的な要件・ユーザーストーリーに変換
2. **構築フェーズ**: AIがアーキテクチャ・コード・テストを提案、技術判断は人間
3. **運用フェーズ**: AIが蓄積コンテキストでインフラ・デプロイを人間監視下で管理

メリット: 意思決定の高速化（スプリント→ボルト、週単位から時間・日単位へ）、要件の明確化、品質管理の緩和。

## Kiroの3つの特徴

1. **Spec-Driven Development（仕様駆動開発）**: プロンプトから要件・システム設計・実装タスクを自動生成
2. **Agent Hooks**: ファイル保存等のイベントでAIをトリガーし、ドキュメント生成・テスト作成・最適化をバックグラウンド実行
3. **Steering**: 規約・ベストプラクティスをルール化し一貫性を担保

### 仕様駆動開発の流れとドキュメント構成

```
初期化 → Steeringドキュメント生成(product/tech/structure)
requirements.md（要求定義）→ 人間レビュー
design.md（詳細設計）→ 人間レビュー
tasks.md（タスク分解）→ 人間レビュー
→ 実装フェーズ（要求変更時は各mdを修正して還流）
```

```
.kiro/
├── steering/          # プロジェクト全体のメモリ
│   ├── product.md / tech.md / structure.md
└── specs/[feature-name]/
    ├── requirements.md / design.md / tasks.md
```

- **requirements.md**: ユーザーストーリー（As a [role], I want [function], so that [business value]）＋ **EARS記法**（構文6種のみで学習曲線が低く要件定義の品質を維持）。テスト可能な受け入れ基準を記述。
- **design.md**: 最大の特徴は「技術調査フェーズ」（Steering確認→既存コードベース調査→依存関係・インフラ調査）。アーキテクチャ・データフロー・IF・データモデル・エラーハンドリング・テスト戦略・ドメインモデルを文書化。
- **tasks.md**: 作業を追跡可能な個別タスクに分割し、各タスクが要件を参照。段階的実装でAIの暴走・重複実装を防ぐ。Markdownチェックリスト記法で書く。

## Kiroを試して分かったこと

- ✅ フローを固定化でき人間の認知負荷を下げるガードレールになる。AWSが考えるAI-DLCに則ったドキュメント管理でオレオレドキュメント問題を緩和。
- ⚠️ システムプロンプトの制約を守れるほどモデルが賢くなく、仕様出力にばらつき。新技術調査がデフォルトでは難しい。自由度が低い（個人開発・開発外タスクはCLIエージェントの方が良い）。
- Kiroのデメリット vs Claude Codeの優位性: Kiroは最強モデル（o3, Opus 4）が使えない・最新情報を反映できない・実装が遅い ⇔ Claude CodeはOpus 4が使え、WebSearch/WebFetch標準搭載、実装が速い。

## KiroをClaude Codeで再現できるか

| 特徴 | Kiro | Claude Codeでの再現 |
|---|---|---|
| 仕様駆動開発 | Specs | Plan Mode（オレオレ構成になりがち） |
| 自動化 | Agent Hooks | Claude Code Hooks（圧倒的自由度だがAgent Hooksの方が簡単） |
| 知識管理 | Steering | CLAUDE.md（1ファイル管理では限界） |

→ Specs/Steeringをしっかり再現できればKiroの仕様駆動開発を実現できる。発表者は **`claude-code-spec`**（Kiroの仕様駆動開発をSlash Commandsで再現し、出力・ドキュメント構成を忠実に再現）を公開。

## Claude Code Hooks

- LLMが現行アーキテクチャである以上「指示を忘れる」はいつか起こる。ルールベースでガードし、仕様駆動で作ったタスクを安全に実装する。
- 早い段階でフィードバックを回せる。小さく頻繁に実行すると開発速度が上がる。AIも人間も忘れるので、いつも実行するものは設定しておく。
- できること: 完了通知（音・Slack/デスクトップ）、編集時テスト実行、Lint/Format（失敗なら完了ブロック）、`rm -rf`拒否、tasks.md更新強制、commit前lint/format/test、ログ。
- Python個人利用例: 完了通知(Stop)、pip→uv変換(PreToolUse)、.env機密ファイル保護(PreToolUse)、Ruff Lint/Format、mypy型チェック。
- 作り方のコツ: 公式ドキュメントを取得するCustom Slash Commandを作り、`/docs:claude-code 〇〇のhooks作って ultrathink`で概ね生成できる。参考リポジトリ: CCPlugins、awesome-claude-code、claude-code-hooks-mastery。

## まとめ

- 仕様駆動開発はAI-DLC実現の第一歩。エンジニアの価値は「実装スキル→ビジネス価値創出＋組織内でのAIの不確実性最小化」へシフト。
- AIを主体に人間の承認を最小化する開発プロセスが求められており、仕様駆動開発とHooks（ルールベースでの指示忘れ防止）はその一環。
