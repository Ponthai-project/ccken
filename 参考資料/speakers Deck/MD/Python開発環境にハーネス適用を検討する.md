## 「ハーネスエンジニアリング」の登場と混乱

- 2026年2月11日、OpenAIのRyan Lopopoloが公開した記事（openai.com/index/harness-engineering/）が起点。「開発パフォーマンスが大幅に向上した」という紹介のみで、具体的に何をしたかは書かれていなかった。「初期の進捗は予想より遅れましたが、それはAIに能力がなかったからではなく、環境が十分に定義されていなかったためです」という一節が引用されている。
- その後バズワードとして度々語られるようになったが、今も定義がバラバラ。「ハーネスとは何か」を各自が解釈し、記事・登壇・OSSで独自の定義を発信し続けている状態。

## 「ハーネス」と聞いて最初に浮かぶイメージと実際

- 最初のイメージは「制御する・束縛する・抑える」（mainブランチへの直接プッシュを止める、`rm -rf`の実行を防ぐ、意図しないファイル削除を防止する、等）。
- しかし実は「能力を最大限に引き出すための装具」という側面が大きい。
- 定義が人によって様々なのは、AIを「使う人と作る人」「扱うスコープの広さ」によって見えているものが違うため（出典: nrs（成瀬允宣）「実践ハーネスエンジニアリング：TAKTで実現するAIエージェント制御」）。
  - コンテキスト設計の話として語られる
  - 出力検査・ガードレールの話として語られる
  - ドキュメントやコードの劣化対策として語られる
  - 「なんとなく便利な仕組み」の総称になりがち

## Computational vs Inferential（決定論的 vs 確率論的）

出典: Birgitta Böckeler「Harness engineering for coding agent users」（martinfowler.com）

| 観点 | Computational（決定論的） | Inferential（確率論的） |
|---|---|---|
| 実行主体 | CPU・高速・信頼性高い | GPU/NPU・低速・非決定論的 |
| 特徴 | 安価に繰り返し実行可能、結果が確実 | 意味的判断が可能、非決定論的 |
| フィードフォワード例 | LSP・型チェッカー、スクリプト | AGENTS.md・Skills、コーディング規約 |
| フィードバック例 | テスト・linter・ruff、構造解析・型チェック | AIコードレビュー、LLM as judge |

## ハーネスの構造：フィードフォワードとフィードバック

出典: nrs「実践ハーネスエンジニアリング」／Birgitta Böckeler「Harness engineering for coding agent users」

- **フィードフォワード**（動く前に方向づける）: 役割と目的を明示する／背景知識をあらかじめ渡す／完了条件を事前に合意する／使える手段を必要な範囲に絞る → エージェント実行へ。
- **フィードバック**（動いた後に自己修正を促す）: 出力を検査しやすい形で受ける／異なる観点から複数回チェックする／不備があれば修正を促す／停滞したら別の手立てへ切り替える。

## ステアリングループ

- 人間の役割は「書く」から「操縦する」へ。フィードフォワード（事前の方向づけ）→エージェント実行→フィードバック（事後の検査）→結果→ハーネスを調整、というループを、FFとFBの両方を継続的に改善しながら回す。
- AIが生成した結果を見て、GuidesとSensorsを継続的に改善する。コードを「書く人」から「エージェントを操縦する人」へ。ハーネスを育て続けることが新しいエンジニアリング、と位置づけられる。

## 私が考えるPythonでの開発におけるハーネス

- 私が考えるハーネス＝AIが能力を発揮できる環境づくり。スコープはモデル以外の全要素（広義）、立場はAIツールを使う人、本質は抑制ではなく能力発揮の環境整備。

### フィードフォワード：クリーンな土台の構築

AIが迷わず安全に動ける「クリーンな土台」を構築する3要素。

1. **ガイドファイル**（AGENTS.md／CLAUDE.md、Kiro steering files）: Python規約・禁止パターンを明記、日本語での出力・コメントを指定、Permissionsのallow/denyで操作範囲を制限。
2. **環境の完全隔離**（`uv venv`）: 仮想環境を強制しグローバルと完全に分離、AIが勝手にグローバルを変更できなくなる、`uv sync`で環境を瞬時に再現。
3. **依存関係の固定**（`uv.lock`）: バージョンを厳格に固定、AIによる未承認パッケージの追加を防止、PyPIサプライチェーン攻撃対策。

型ヒント・docstringは、AIへのData Context層として機能する。書くほどエージェントの推論精度が上がる。

### フィードバック：ツールが瞬時に弾き返す

AIのミスを人間が指摘するのではなく、ツールが瞬時に弾き返す構成。

- `ruff`（Computational）: `T201`（print検出）、`BLE001`（例外握りつぶし）、`E722`（bare except）等を検出。
- `pyright`／`mypy`（Computational）: 型チェック、型ヒントなしの検出。
- `pytest`（Computational）: 振る舞い検証（仕様どおりに動くか）。
- AI Code Review（Inferential）: 抽象的Exception、意味的アンチパターンの検出。

`pyproject.toml`で上記すべてのツール設定を一元管理し、ハーネスの設定散逸を防ぐ。デッドコード検出には`vulture`の併用も推奨。

## 明日からできること

ハーネスは「開発者の経験」の外部化、として3つの具体アクションが示される。

1. **`uv`で環境を整える**（`uv venv` / `uv sync`）: 仮想環境の分離から。グローバル汚染を防ぎ、AIが安全に動ける土台を作る。
2. **`ruff`を`pyproject.toml`に設定する**: linterとformatterを一本化。print・例外握りつぶしをComputationalに検出する（`[tool.ruff.lint] select = ["T201","BLE001","E722"]`）。
3. **`AGENTS.md`にプロジェクトの文脈を書き出す**: Python規約・禁止パターン・日本語指定を明記。エージェントへのフィードフォワードの起点とする。

「AIと並走しよう！」というメッセージで結ばれている。

## 要点まとめ

- 「ハーネスエンジニアリング」は2026年2月のOpenAI記事を起点にバズワード化したが、提唱者ごとに定義がバラバラで、Böckelerの「Guides/Sensors」枠組みが最も体系的で入門に適する。
- ハーネスは「制御・抑制」のイメージが先行しがちだが、実際は「AIの能力を最大限引き出すための装具」である。
- ハーネスはComputational（決定論的・高速・確実）とInferential（確率論的・意味的判断可能）の2種の要素を組み合わせて設計する。
- ハーネスの基本構造は、動く前の「フィードフォワード」と動いた後の「フィードバック」のループであり、人間の役割は「書く人」から「操縦する人」へ移行する。
- Python開発への具体適用として、フィードフォワード側は`uv venv`／`uv.lock`による環境隔離・サプライチェーン対策、フィードバック側は`ruff`／`pyright`／`pytest`／AIコードレビューの組み合わせを提案。
- 自プロジェクトの棚卸しでは、ガイド（Specs／Steering／CLAUDE.md）は整備済みだが、Permissionsのallow/denyやSkills/Subagentsなどのセンサー側が今後の優先課題。
