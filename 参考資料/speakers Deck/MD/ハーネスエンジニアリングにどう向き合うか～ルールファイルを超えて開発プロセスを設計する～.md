## 前提：1人開発の中で探求したAIネイティブな開発プロセス

創業からシード調達までの半年弱は1人でプロダクト開発を担っており、その中でAIネイティブな開発プロセスを探求し試行錯誤してきた内容が本発表のベース。「今の時代にゼロから立ち上げる開発組織・プロセスはどうあるべきか」という問いで試行錯誤した結果であるため、内容には世間一般のハーネスエンジニアリングに該当するものもあれば、ハーネスより上位レイヤーに分類すべき取り組みも含まれる。組織やプロセス全体で「エージェントによる開発」をどう行うかが発表の中心軸。

## ハーネスエンジニアリングの3つの難しさ

昨今特有の曖昧さによる捉えどころのなさ、寿命・有効範囲の見極めの難しさが指摘される。

| 難しさ | 内容 |
|---|---|
| 何をすべきかわからない | LangChainは「モデルでなければハーネス」と呼び、Anthropicは「ツールコールをルーティングする制御ループ」と言う。ベンダーと利用者で定義やスコープも異なる |
| 効いてるかわからない | 「このハーネスは効いている」と言い切るための物差しがないことが多く、ルールファイルでも起きた定量的・実験的・継続的な改善活動を回すのが手探りな状態 |
| いつ消えるかわからない | モデル／モデルプロバイダーの進化に影響を受ける。進化により不要な制御・処理になる可能性に加え、Claude Managed Agentsのような登場も今後起こりうる |

## ハーネスは何で構成され、何を達成したいか

- 利用者側であるなら、記憶と文脈、品質と安全がメインのスコープ。system prompts、CLAUDE.md、skills、tools、orchestration、hooks、observabilityなどが中心となる。
- ハーネスエンジニアリングで達成したいのは「エージェントが動く環境・システムの設計」。事前の予防（フィードフォワード）と事後の検証（フィードバック）を設計し、失敗からの学習まで含む。テストは事後の検出にとどまるが、ハーネスは事前の予防〜事後検証〜学習までを含む、より広い概念。
  - **行動を制約する**（何をどう作るかを定め解空間を絞る）: CLAUDE.md、スキル定義、ワークフロー、spec.md
  - **正しいか確かめる**（検証し、問題があれば差し戻す）: 静的チェック、ブラウザテスト、探索的テスト、AIレビュー
  - **次をもっと良くする**（学びを蓄積する仕組み）: Self-Refinement、メモリ、実行ログ分析、ハーネスの自動修正、パターンDB

## 「ルールファイル整備＝AIネイティブ開発」への疑問

- Claude Code/Codexを配りルールファイルを整備すれば、それでAIネイティブな開発と言えるか？「個人がどれだけうまく使えるか」に閉じていないか、という問いを提示。
- ハーネスエンジニアリングにおいても、ルール／スキル整備と個人の感性による「俺の考える最強のX」に留まっていないかという懸念。
- ルールファイル・スキルの整備は「一部」に過ぎず、評価・Evalこそが長期的な改善を導く。「その修正は何にどう／どの程度Hitするのか」をわかることの重要性が増している。ハーネスを育てるにも実行/思考履歴と評価基準が必要であり、これはソフトウェアエンジニアリングとして向き合える領域。
- 一番好きな捉え方は「モデルでなければハーネス」。ワークフロー、パイプライン、評価、組織、顧客接点まで、モデルの外側は全部ハーネスの範囲として発想を広げられる（広すぎて全てがハーネスになる面もある）。
- 少なくともプロダクト組織・プロセス全体で見ればNO。コーディングエージェントで開発するだけでなく、コーディングエージェントが機能する環境や開発プロセスを設計することが重要であり、ハーネスエンジニアリングもあればその上位に位置しそうな取り組みもある。環境や開発プロセスの捉え方を広げれば、いわゆるユーザーハーネスの範疇を超えて取り組めることは多い。

## 実践：比較的類似例の少ない3つの取り組み

### 1. 決定論と確率論のハイブリッドオーケストレーション

- 「行動を制約し、正しいか確かめ、次をもっと良くする」を1つのパイプラインに。Claude Agent SDK＋TypeScriptで構築し、Claude Code／Codexで人間が行っている開発プロセスをそのまま型化する。
- specからブラウザテスト・スクショ付きのPR作成までスクリプト化。CodexによるCross Model ReviewやOpus 4.6によるAdvisor等も組み込む。
- 決定論的ステップ（例: lint、ブランチpush）とエージェント的自由（実装、CI修正）を交互に配置するハイブリッドオーケストレーション。スキル・開発用ツール/スクリプトも共通利用しているため、スキルを育てることがパイプラインの進化にも直結する。
- npxコマンドで起動するため、GitHub ActionsでIssueからのキックも容易。Issue→PRが一定のクオリティで担保された上で実現できる。
- Claude Agent SDK＋TypeScriptで開発フローをコード化する良さ: 開発フロー全体がコードのため、クラウド実行・ログ/トレース取得が容易でソフトウェアと同様に扱える。
  - **品質保証の組み込み**: ブラウザテスト・証跡スクショ・動画まで完了した状態でPRが出る。不良specを早期に弾き、Codexクロスモデルレビューで検証。
  - **計測と改善の基盤**: スクリプト内で全実行のログが取れる。設計判断・インシデント履歴をDBに永続化させるなど、「効いてるかわからない」を解消する土台を組み込みやすい。
  - **再現性と属人性の排除**: 誰が起動しても同じプロセスが走り、個人のAI開発への熟練度に依存しなくなる。今はデザインハーネスの組み込みに着手中。
- Braintrustにログを送り、分析と改善を回せる仕組みを構築。定量的な評価やダッシュボードの本格化に着手しており、どの類のタスクがどの程度の成功率／人間の追加介入なしでマージされているか等を追跡する。後述のself improveの文脈では、分析・改善提案から修正まで自動化することを目指す。

### 2. 開発をキックするトリガーの拡張

- 開発パイプラインが作れたら、次はトリガーを拡張する。エンジニア以外でも、ターミナル以外からも開発をトリガー可能にし、開発を起動する入口を増やす。
- 画面から開発をトリガーするUI Annotator（Chrome拡張）を開発。画面上でのピン留めやテキスト入力が前述のパイプラインに流れ、PRが作成される。一般的なハーネスエンジニアリングに含めるかは議論があるが、「エージェントが動く環境の設計」として取り組んでいる。
- 変わり種として、自社プロダクトでAIがヒアリングした結果をそのまま開発に回す仕組みもある（基本ドッグフーディング目的）。

### 3. Self-improveへの挑戦

- ルールファイルの整合性確認・修正等のフィードバックループを自動化。同様にClaude Agent SDKでツールとしてスクリプトを書いて自動実行する。直近はエラー履歴のメモリ化に着手。
- せっかく自動ブラウザテストをしているので、「Xを編集したら、Yでエラーが出た」などの履歴を蓄積する。ローカルのSQLiteにエージェントから格納しているが、実際にworkするかは定かではないとしている。
- 実際に動かしているのはClaude Agent SDK製の自動修正系の仕組み。開発パイプライン自体の改善は基本、人間 with エージェントで今は頑張っている段階。
- やがてはハーネスの自動生成へ。Meta-Harness（Stanford）やAutoHarness（Google DeepMind）では、LLM自身が制約を発見し、ハーネスを生成・改善、コード化するという方向性が紹介されている。

## 今後、ハーネスエンジニアリング（的なもの）にどう向き合うか

- モデルやマネージドサービスの進化によって消えづらいレイヤーに投資する。LLMプロバイダーを切り替えても同様に利用可能な仕組みを中心に据える。
- 人間が介在するタイミングやタッチポイント設計も大事（Human-in-the-loop）。一方、ループの上でハーネス（仕様、品質チェック、ワークフロー）を設計するのがこれからのエンジニアの役割。

## まとめ

- （ここ数年こんな感じだが）定義はさておき、ハーネスエンジニアリングの「モデルの外側の仕組み」を設計する営みは重要。
- ルールファイルやスキル整備だけでは足りず、評価・計測・ガードレール・プロセス全体まで射程を広げる必要がある。
- AI時代のエンジニアは、コードを書くことに加えて、コードを書く仕組みを設計し育てる役割を担う。
- エージェントのログを読み、失敗を分析し、仕組みに還元すること自体が、重要なエンジニアリングスキル。

## 要点まとめ

- ハーネスエンジニアリングは「何をすべきか／効いてるか／いつ消えるか」がわかりにくいという3つの難しさを持つ。
- ハーネスの本質は「行動を制約する・正しいか確かめる・次をもっと良くする」という予防〜検証〜学習のサイクル設計であり、テスト（事後検出）より広い概念。
- ルールファイル・スキル整備だけに閉じた「個人のうまい使い方」に留まらず、評価・計測基盤（Braintrust等）まで含めたプロセス設計が必要。
- 決定論的ステップとエージェント的自由を組み合わせたハイブリッドオーケストレーション（Claude Agent SDK＋TypeScript）で、Issue起票からPR作成までを型化・自動化する実践例を紹介。
- 開発トリガーの拡張（UI Annotator等）やSelf-improve（エラー履歴のメモリ化）にも挑戦しており、将来的にはLLM自身がハーネスを生成・改善する方向性（Meta-Harness／AutoHarness）も視野にある。
- AI時代のエンジニアの役割は、コードを書く仕組みを設計・育成し、ログ分析から仕組みへ還元し続けること。
