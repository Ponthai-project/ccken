| 文献・事例 | ハーネスの位置付け | キーワード |
|---|---|---|

→ **ハーネスエンジニアリング＝「LLMを効果的に動作させるための仕組み全般（ハーネス）」を設計・開発する行為**（暫定定義）。

## スコープとキーワード

対象領域は広く要素は多岐:

- **Where**: Workspace / Sandbox / Runtime / Interface / Data Sources / Permissions
- **When**: Trigger / Event / Schedule / Queue / Approval Points
- **How**: Principles / Verification / Tools / Skills / Orchestration

構成要素の一部は、これまでのソフトウェア開発の蓄積の上に成り立つ（Verificationの設定＝Linter・テストコード・CIチェック、Principlesの設定＝ADR・開発標準〈TERASOLUNA等〉）。

## ハーネスの影響範囲

SDLC/V字モデル（要件定義・設計・開発・テスト・デプロイ・運用）において、ハーネスは大きな影響範囲を持つが、**既存のハーネスは開発・テスト領域に集中**しており、要件定義・運用は一般的なハーネスのスコープ外になりがち。

## 設計思想：進め方の一例

注力指標を考慮しつつ、**システム開発の自由度を下げる施策から始める**と、難易度・影響範囲の観点で進めやすい。

| ハーネス要素 | 自由度 | 難易度 | 主な効果指標 |
|---|---|---|---|
| Principlesの設定 | 下げる | 相対的に低い | 保守性・変更容易性(Q) |
| Verificationの追加 | 下げる | 相対的に低い | 要件達成度(Q) |
| Hooksの設定 | 下げる | 相対的に低い | 保守性(Q) |
| Skillsの設定 | - | 相対的に低い | 自律性向上 |
| パラメータチューニング | 上がる | 相対的に高い | 開発自由度 |

## ハーネスエンジニアリングで十分か？（不足分）

1. **Four Keys・ボトルネック**: デプロイ頻度・変更リードタイム・変更失敗率・復旧時間。**開発・テストが速くなっても、デプロイ頻度を上げられなければユーザーへの価値提供は加速しない**。
2. **人の働き方（SPACE指標）**: 満足度・活動量・成果・コミュニケーション/協働・効率/フロー。ハーネスはあくまでLLMが働きやすい仕組みづくりであり、**人が働きやすい環境を作ることも中長期で必要**。

→ 設計ポイント: LLMのためのハーネスを作ると同時に、人・チームのための仕組みも整備する。

## 事例紹介（Algomatic）

- **並列開発によるdiscovery業務高度化**: 投機的な開発＋テストを並列化。
- **Agent Loop機構**: 人の介入を模したAgent Loopで開発を自律化。
- **デザインシステム**: ユーザー体験原則を含むデザインシステム＋事業KPIを含むプロダクトPrincipleを用いた確度の高い開発。
- **Remote Test Runtime**: テスト前倒しによる観点潰し込み。クラウド環境でのテスト自動化で負荷軽減、テスト動画をLinearへ添付。
- **Quality Assurance**: 独自の品質管理基準による、開発機能種別に応じたレビュー。
