## 前提とスコープ

本セッションで扱う「ハーネス」は、コーディングエージェントの外側にある**ユーザーハーネス**の話。オーケストレーションの話はしない。

## 登壇者について

Gota（@gota_bara）。Agentic AIエンジニア＆アナリティクスエンジニア。AIエージェント開発／小売向けデータプロダクト／データ整備に従事。好きなエージェントは"pi"（超シンプルなTerminal型エージェント。システムプロンプトを人間が容易に読める）。自身が開発する`cc-sdd`（v3.0、2026/4更新）では、境界のファーストクラス化、specのTDD実装↔レビューのClosed Loopによる長時間の自律実装、Slash CommandからAgent Skillsへの変更を進めている。

## 現場でよく聞く話：段階は違うが根っこは一つ

判断基準が明文化されていないことが、あらゆる段階で問題化する。

| 段階 | 症状 |
|---|---|
| ハーネスがない | 一人一人バラバラに指示し、Agentが出すコードの品質が安定しない |
| 作り始めた | ルールが増えすぎ、Skillに切り出しても読まれない／効かない。CLAUDE.mdが1000行超え、指示の半分が無視される |
| 運用中 | モデルが変わっても旧モデル向けの指示が剥がせない。半年前の設定を消していいか／更新していいか分からない。オーケストレーションがドリフトする |

足せても削れず、ハーネスは肥大化し、やがて変更できなくなる。判断基準がないまま太ると、トークンを増やしてコストが増加するのに加え、指示の内容が直接精度を下げる。

## ハーネスを分解する

HOW（手段）はたくさんあるが、業務目的と状況で最適な選択肢は変わる。→ 業務目的から逆算する必要がある（出典: "Harness engineering for coding agent users"）。

## ハーネスを足す前に考えるべき4ステップ

| Step | 内容 |
|---|---|
| 1. 業務目的を言語化 | 何を守る／守らない／人が介入するか＋前提確認 |
| 2. 構造で守れないか問う | 設計／型／依存方向 ＞ scaffolding ＞ test ＞ 指示、の優先順位で検討 |
| 3. 指示で補う | 残りをCLAUDE.md／skill／review agentで補う（agentは最小限） |
| 4. 削れる状態で運用 | 小さく始める／失敗駆動／目的で削る／ログを残す |

「ハーネスの前」＝エージェントの推論に載せずに済む可能性を潰してから、残ったものを載せる、という順序。ゴールはハーネスの変更容易性を高く保つこと。

## Step1: 業務目的を言語化する

目的が曖昧だと、制御は「設計」ではなく「不安への反応」になってしまう。前提として「Agentに任せられる業務か」「開発環境はハーネスが効くか」を満たす必要がある。

例（重い集計処理の軽量化: DWH直叩き→マート層参照）:

| 問い | 答え |
|---|---|
| 何を守るか？ | 結果の数値一致／既存利用側（ダッシュボード・API）契約／マート層スキーマ（複数あり互いに衝突しうる） |
| 何を守らないか？ | 元クエリの可読性、最初からの最適設計、ドキュメントの同期（スコープ外） |
| どこで人が介入するか？ | 軽量化対象の優先順位、コスト/性能のトレードオフ、アーキ変更採否（エージェントに任せない判断） |

## Step2: エージェントではなく構造で守る

違反が書けない形にできるなら最強。構造（設計・型・依存方向）で守れるなら、指示は要らない。

## Step3: 指示を足すのは簡単……判断基準は削るためにある

判断基準の本丸は「削るため」にある。ただしguideは評価しにくく、放っておくと溜まる。

## Step4: 業務目的に照らして既存ハーネスを棚卸しする

既存のハーネスを「判定」で動かす。実務では次の3パターンが多い。

| 既存要素 | 判定後 | なぜ |
|---|---|---|
| ① 観点を絞る | 可読性＋セキュリティ＋数値妥当性のreview agent | review agentは残す、可読性観点だけ落とす（要素を消すのではなくスコープを絞る） |
| ② 層を移す | CLAUDE.md「データ品質を担保せよ」（指示）→ rule → review agentの観点に移す | 具体的チェック（数値一致・スキーマ）の方が強い |
| ③ 目的次第 | 「対象クエリをAgentが自動選定」→ 軽量化プロジェクトでは不採用 | 業務目的が変われば採用（例: ボトルネック自動発見） |

## 小さく始める、失敗駆動で育てる

理想の順序は「業務目的を言語化→構造で守る→指示で補う→削れる状態で運用」。新規プロジェクトで取り組むときは構造をざっくり先に作ってから進める。実務では、セッション履歴またはログは必ず保持した上で3・4から始め、曖昧さや指示に従わなくなってきた時にループして1・2に戻る方がスピード感を持って進められる。

## まとめ：取り組む順序

1. **業務目的を言語化**: 何を守る／守らない／人が介入するか
2. **構造で守れないか問う**: 設計／型／依存方向／アーキテクチャ ＞ scaffolding ＞ test ＞ 指示
3. **指示で補う**: 基本はSkill（AGENTS.md）は最小限
4. **削れる状態で運用**: 小さく始め、失敗駆動で足し、目的で削る

判断基準を先に持つとシンプルかつ変更容易のハーネスを作りやすい。

## 要点まとめ

- 「ハーネスがない／作り始めた／運用中」のいずれの段階でも、根本原因は「判断基準が明文化されていない」ことにある。
- 足すだけで削れないハーネスは肥大化し、トークンコスト増と精度低下の両方を招く。
- ハーネスを足す前に「業務目的の言語化→構造で守れないか→指示で補う→削れる状態で運用」の4ステップを踏むべき。
- 優先順位は「設計/型/依存方向・アーキテクチャ ＞ scaffolding ＞ test ＞ 指示」であり、指示（CLAUDE.md/Skill/agent）は最後の手段かつ最小限に留める。
- 既存ハーネスの棚卸しは「観点を絞る／層を移す／目的次第で採否を変える」の3パターンで判定する。
- 実務では理想順序どおりに進めず、セッション履歴/ログを残しつつ小さく始め、失敗駆動で足し、目的に照らして削るループを回すのが現実的。
