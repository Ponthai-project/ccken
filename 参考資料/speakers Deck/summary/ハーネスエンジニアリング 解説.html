ハーネスエンジニアリング 解説 — モデルを「賢い馬」として走らせるための、鞍・手綱・ルートの設計論 —

「ハーネスエンジニアリング」とは、AIモデル単体ではできないことを、モデルの外側の仕組み（コンテキスト・制約・検証・観測性）で補い、AIを安全かつ自律的に走らせるための環境設計である。 本ページは、27本の登壇資料を横断統合した「ハーネスエンジニアリング総まとめ」を土台に、比喩イラストと図表でその要点を一望できるようにまとめたものである。

1. ハーネスとは何なのか

結論を先に言い切る。ハーネスとは「AIモデル単体ではできないことを、モデルの外側の仕組み（コンテキスト・制約・検証・観測性）で補い、AIを安全かつ自律的に走らせるための環境設計」である。そしてハーネスエンジニアリングとは、その環境を意図的に設計・運用・改善し続ける営みを指す。

この結論は、資料横断でほぼ共通して現れる「馬と馬具」の比喩によって裏づけられる。Claude Codeの開発者Boris Chernyの言葉として紹介される「モデルを直接使うことはできない。ハーネスを使う必要がある。Claudeは馬で、鞍のようなものが要る」（Claude Codeとハーネスについて考えてみる／Oikon）は象徴的であり、西岡賢一郎も「モデルを支え、暴走させずに走らせる周辺装備」と同じイメージで説明する（ハーネスエンジニアリング入門／西岡賢一郎）。非エンジニア向けの超入門資料でも「野生の馬（＝賢いが暴れるAI）に、鞍・手綱・ルートを与えて目的地まで安全に走らせる仕事」として同じ比喩が使われている（ハーネスエンジニアリング解説／clshinji）。

より技術的な定義も複数の登壇者から示されている。西岡はハーネスを「Claudeを呼ぶループ＋ツール呼び出しを実インフラに繋ぐ仕組み」であり、本質は「モデル単独ではできないことの“仮定”を符号化したもの」と定義する（ハーネスエンジニアリング入門）。梶川琢馬は「自律的なAI Agentの実現のために、LLMモデルの性能を最大限引き出すための環境を設計すること」（実践ハーネスエンジニアリング）、神谷亮平は語源に立ち返り「体や物を安全に固定し、力を効率よく伝えるためのベルト状の装具」から転じて「AIを安全に制御し、能力を効率よく発揮させるための環境整備活動」（ハーネスエンジニアリング×AI適応開発）とする。そして多くの資料が共通して引用するのが、LangChain（Vivek Trivedy）の定式である。

Agent = Model + Harness
"If you're not the model, you're the harness."
（ハーネスエンジニアリングとは？／KINOPEE、ハーネスエンジニアリングとは？を巡る各資料 ほか多数）

非エンジニア向け資料が結びとして掲げる定式も分かりやすい。

Oikonは、この「ハーネス」がどこに位置するかを実務的に「4層構造」で整理している。Prompt Eng.（指示を作り込む）→ Context Eng.（見せる情報を管理する）→ Harness Eng.（モデルを動かす仕組み全体）→ Loop Eng.（ループ自体を書く）という段階であり、この整理では「Claude Code自体がハーネス」だとされる（Claude Codeとハーネスについて考えてみる／Oikon）。西岡も類似の進化アークを示している。

なお「どこまでをハーネスに含めるか」の力点は論者によって割れており、これは無理に一致させるべきではない資料横断の最大の論点である。この対立の詳細は4章で扱う。まずは、定義の揺れを前提としつつ、実務上ほぼ共通して挙がる「構成要素」を見ていく。

2. ハーネスを構成する要素

定義は割れていても、「ハーネスが何でできているか」を挙げると、資料横断でかなりの部分が一致する。以下、静的な構成要素（レイヤー）と、それらを動かす動的な仕組み（ループ・統治構造）に分けて整理する。

2.2 tameの「3本柱→5層アーキ」

この整理は資料ごとに呼び方が違うだけで骨格は近い。ため（為安圭介）は前作で「コンテキスト管理／アーキテクチャ制約とガードレール／品質評価と多角的フィードバック」の3本柱を提示し（AIエージェント時代のハーネスエンジニアリングとは）、続編ではそれを5層アーキテクチャ（ルール層／コンテキスト層／フック層／コマンド・スキル層／エージェント・MCP層）というファイル群にまで具体化した（AIエージェント時代のハーネスエンジニアリング Claude Code実装編）。梶川琢馬も「コンテキスト設計／アーキテクチャ制約とガードレール／品質評価ループ」という同型の3本柱を示している（実践ハーネスエンジニアリング）。

池上涼平は同様の骨格を「コンテキスト／アクション／フィードバック／運用」の4設計軸として整理している（新規プロダクトを高速で生み出すハーネスエンジニアリング）。

2.3 動的な仕組み：ステアリングループ

構成要素を並べるだけでは「ハーネスが生きた仕組みであること」が見えない。nrs（成瀬允宣）とBirgitta Böckelerの整理は、この動的な側面を最もよく捉えている。

  • フィードフォワード（動く前に方向づける）：誰として何を目的に動くか明示する／背景知識と完了条件を事前に渡す／使える手段（ツール・権限）を必要な範囲に絞る。
  • フィードバック（動いた後に自己修正を促す）：出力を検査しやすい形で受け取る／異なる観点から複数回チェックする／改善が停滞したら別の手立てに切り替える。

gota（beyond-spec-usable）が実装した4subagentハーネス（Planner→Builder→Evaluator→UX Reviewer）も、この構造を体現した実例である。Plannerが検証シナリオ（WHAT）を先に切り、Builderが縦に一本コアフローを通し、Evaluator・UX Reviewerが多軸スコアで合否判定する――これは「フィードフォワードで方向づけ、フィードバックで検証し直す」ループそのものである。

2.4 統治の視点：立法・司法・行政

外山英幸（ビズリーチ）は、ハーネスを構成する仕組みを「誰が何を決めるか」という統治の視点で捉え直す独自のモデルを提示している（AI駆動開発が変える、大規模開発の前提）。

3. ハーネスを考える上でのポイント

複数資料に共通して現れる設計原則を統合すると、大きく4つにまとめられる。

3.1 モデル進化への追従：足すより剥がす

西岡賢一郎は「ハーネスはモデル進化で陳腐化する」（"Harnesses encode assumptions that go stale as models improve."）ことを明確な原則として掲げ、次の4原則を提示する（ハーネスエンジニアリング入門）。

1最小権限

危険なツール呼び出しを事前に止める。本番DB事故を技術的にゼロにする。

2可観測

何が起きたか後から再現できる。金融・医療・公共では監査要件そのもの。

3失敗前提

1つ落ちても全体は死なない（cattle-not-pets）。緑でなければ終われない仕組みにする。

4足し算ではなく引き算

半年ごとにhooksを読み直し、不要なものを消す（剥がす運用）。

個別のhookは陳腐化するが、「外側で支える」設計スキル自体は陳腐化しない――「剥がす目」を持ち続けることが西岡の一番のメッセージである。

3.2 目的ファースト・優先順位づけ

Gotaは、ハーネスを足す前に踏むべき4ステップと、その優先順位を示す（目的ファーストのハーネス設計）。

  1. 業務目的を言語化する（何を守る／守らない／人が介入するか）
  2. 構造で守れないか問う（設計／型／依存方向 ＞ scaffolding ＞ test ＞ 指示、の優先順位）
  3. 指示で補う（CLAUDE.md／skill／review agentは最小限に）
  4. 削れる状態で運用する（小さく始め、失敗駆動で足し、目的で削る）

渋谷優介の設計思想（Principles・Verification・Hooksの設定＝自由度を下げる施策から着手し、Skillsの設定・パラメータチューニングは後回しにする方が難易度が低い）もこの優先順位と整合する（ハーネスエンジニアリングの概要と設計思想）。Oikonの「まず内部ハーネス＋モデルの素の実力を使い倒し、足りない分だけ外部ハーネスを小さくポータブルに足す」という原則も同じ方向を向く。

3.3 汎用化・密結合の罠（失敗事例からの教訓）

Gotaは自身の失敗を率直に共有している（ハーネスエンジニアリングをやりすぎた話）。「様々なソフトウェア開発に対応できるハーネス」を目指して範囲を拡張した結果、テンプレート・hook・verify・tests・policyが同時に絡み合う密結合に陥り、Script 25,000行・Markdown 10,000行・Hooks 1,500行にまで肥大化。1回の全体テストに1時間かかり、修正の成否評価すら不能になった。並列実装が生む「境界のドリフト」「仕様のドリフト」「依存関係のドリフト」という3つの異常も報告されている。教訓は「ハーネスでなんでもやろうとすると、エージェントを制御するための仕組み自体が制御不能になる」ことであり、目的ごとに小さく分ける（Planning／Execution／Verification／Adaptation Agentの分離）ことが再発防止策として示される。

3.4 統治・責任構造としての設計

外山は、ハーネス設計の議論を「AI実行の統治（Governance）」の問題として捉え直し、ADR・Wiki等の「書かれている」だけの状態と、Agentが参照し違反すれば止まる「効いている」状態を峻別すべきだと主張する（AI駆動開発が変える、大規模開発の前提）。神谷亮平も、OpenAIの短時間マージ運用は「ハーネスの作り込み」だけでなく「適切な妥協点を許容する組織的判断」があってはじめて成立するとし、自社で決めるべき3つの問い――品質の保証範囲・人間の関与の線引き・実行可能な環境づくり――を、仕組み・プロセス・体制を一体で見直す「責任構造の再設計」として提示する（ハーネスエンジニアリング×AI適応開発）。

4. その他の論点

4.1 定義の割れ：広義派 vs 狭義派

論者によって「どこまでをハーネスに含めるか」の力点は明確に割れている。これは無理に一致させるべきではなく、資料横断で見えてきた最大の論点でもある。

  • 広義派（Chase＝LangChain、Chawla）：モデル以外の全てがハーネス。ツール実行・メモリ・サブエージェント・ガードレール・オーケストレーションまで包括し、「コンテキスト設計はハーネスの一構成要素」と位置づける。
  • 狭義派（Martin Fowler／Birgitta Böckeler）：逆に「コンテキストの中の特殊形」としてハーネスを位置づけ、feedforward（guides）とfeedback（sensors）の制御理論として扱う。すなわちコンテキスト⊃ハーネスであり、包含関係が逆転する。
  • 運用哲学派（Mitchell Hashimoto）：ハーネスは設計物というより「失敗するたびにAGENTS.mdへルールを1行足し、同じミスを二度とさせない」累積的な運用プロセスそのもの。
  • スケール駆動派（OpenAI／Ryan Lopopolo）：3人・5ヶ月・100万行規模、大部分をAIが生成したという事例を起点に、「プロンプトでもコンテキストでも解けない、多段ステップ・自律実行・並列特有の故障モードを解くもの」という動機づけで語る。

KINOPEEは、日本語圏の議論を読み解くための5つの流派を提示している（ハーネスエンジニアリングとは？）。

4.2 なぜ今ハーネスが必要なのか

(1) 人間の確認がボトルネック化

AIエージェントはコードを大量かつ高速に生成できるが、人間の理解・判断・QAの速度は急には増えない。「これからのSDLCの希少資源は人間の時間と注意力」であり、ボトルネックは「書くこと」から「読むこと・確かめること」へ移る。外山はこれを「HITL（Human in the Loop）」構造の限界として整理し、ハーネスは開発速度を上げても、それだけではHITLのまま速くなるだけだと指摘する。
（ハーネスエンジニアリング×AI適応開発／神谷亮平、AI駆動開発が変える、大規模開発の前提／外山英幸）

(2) 「動く」だけでは価値に到達しない

仕様通り動くコードを書かせる技術は急速に進歩したが、「動く」と「使える」は別問題。gotaは「動くけど使えない」の4類型（伝わらない・ぼやける・見つからない・始まらない）を挙げ、Anthropic社内でも出荷物の7割以上がPRDなしで「使えるかは触るまで分からない」現場感覚を紹介する。
（beyond-spec-usable／gota）

(3) 属人性・再現性の欠如

「The 70% problem」（非エンジニアは70%まで速く到達できるが残り30%が進まない）や、ハーバード大学・BCGの調査による「居眠り運転」現象（高精度AIへの過信でチェックが手抜きになり成績が下がる）が紹介される。神谷亮平は「7万行規模のSaaSを2人月・従来比50〜56倍」の実績を「あなただからできたのでは」と問われたことで属人性に気づき、「誰でも・どんな組織でも再現できる仕組み」への転換を迫られたと述べる。
（AI駆動で進化する開発プロセス／佐藤智樹、ハーネスエンジニアリング×AI適応開発／神谷亮平）

r-kagayaも、同じデザインシステムを参照していてもコーディングエージェントや人によって選ぶコンポーネントがずれる「人ガチャ・AIガチャ」現象を課題として挙げている（エンジニアにデザインハーネスを）。これらの背景により、ハーネスエンジニアリングは「AIをどう使うか」の技術論を超えて、「人間の介入をどこに、どれだけ残すか」を設計する組織的な問いへと接続していく。

平野は、ハーネスを「初期コンテキスト型」（CLAUDE.md・skills・agentsで最初に方向を示す）と「遅延ロード型」（rules・skills・hooksでズレを事後修正する）の2種に分け、共通化・汎用化を狙うなら遅延ロード型の方が難易度が低いと結論づける（Copilot／Claude Code両方で使えるスキル・ハーネスの紹介）。

多層防御によるレビュー負荷の低減

複数の実務者が、コンテキスト→型→静的解析→テスト→AIレビュー→人間レビューという多層防御の構図を共有している。梶川琢馬は型システム→静的解析→テスト（ユニット＋E2E＋VRT）→AIレビュー→人間レビューの順に問題を自動検出し、CIが通っただけで「完了」と宣言させずChrome DevTools MCPで実ブラウザ動作まで確認させる（実践ハーネスエンジニアリング）。

池上涼平は「事前に書かせない（CLAUDE.md／rules）→書きにくくする（権限分離・型）→動かして検出（実DBテスト）→マージ前に止める（CI）→botに1次レビュー（CodeRabbit）→人間レビュー」の6層を運用し、同じ指摘を2度繰り返さないためレビュー指摘をrulesへ書き戻すskillを運用する。神谷亮平はgenerated/（自動生成・編集禁止）とcustom/（AIがビジネスロジックのみ書く・編集許可）を分離し、宣言的SQLスキーマからのコード生成パイプラインで「単一のsource of truth」を保つ。

Böckelerの整理では、これらの要素は「Computational（決定論的）」と「Inferential（確率論的）」という2つの性質に分けて設計できる（yuuuka経由で紹介）。

ハーネステンプレートとステアリングループの実演／仕様駆動開発／スキルファースト／Python環境

nrsのTAKT（OSS）は、ハーネスを「ワークフロー（ステップの有限状態機械）」と「ファセット（プロンプトの部品：Persona／Knowledge／Instruction／Output Contract／Policy）」の組み合わせとして表現し、workflow.yamlとファセットに改善を書き戻すことでハーネステンプレートとして成熟させていく（2.3節参照）。

Gota（旧「ClaudeCodeで仕様駆動開発を実現するために！」）は、AWS Kiroの仕様駆動開発（Specs／Agent Hooks／Steering）をClaude Codeで再現し、Steering＝CLAUDE.md、Agent Hooks＝Claude Code Hooksという対応関係を示す。渋谷優介は仕様駆動開発を上流工程（要望→要求→要件）にまで拡張し、「What」だけでなく「Why・Why not」までコンテキストに含めることの重要性を提案する（仕様駆動開発を実現する上流工程におけるAIエージェント活用）。吉田真吾は、AWSのAI-DLC（Inception／Construction／Operations）をチーム開発の文脈に接続し、「守破離」モデル（守：基礎トレーニング／破：チームアグリーメント再構築／離：自社ハーネス開発）を示す（チームで実現する高品質・安全なAI駆動開発実践）。

坂上晴信（TOKIUM）は「業務PJ型」に対し、個人プロジェクトでの検証を通じて「スキルファースト」という別解を示す（スキルファーストで作るAIの自走環境）。やりたいことを全部書き出し、それが全て「スキルの組み合わせ」で実行できるようハーネスを構想し、PR1件ごとに振り返りスキルで自己改善させた結果、1,753行のMarkdownと200+のハーネス関連ドキュメント・約70件のPRがほぼ指示なしで生成された。結論は「枯れた技術が相手ならルール・規約ファースト、日々進化し非定型入力もカバーできるAIならスキルファーストが有効」という使い分けである。

yuuukaは、フィードフォワード（uv venvによる環境隔離、uv.lockによる依存固定、AGENTS.md）とフィードバック（ruff・pyright・pytest・AIコードレビュー）を組み合わせた具体的なPython環境ハーネスを提案し、自プロジェクトの棚卸しでは「ガイド側は整備済みだがPermissionsのallow/denyやSkills/Subagentsなどセンサー側が手薄」という結果を共有している（Python開発環境にハーネス適用を検討する）。

4.4 発展的トピック

デザインハーネス

r-kagayaは、デザイナー不在のスタートアップでエンジニア＋コーディングエージェントがUIデザインを担う際の課題（「それっぽさ」止まりの判断力欠如、最低基準の不在、人ガチャ・AIガチャ）に対し、デザインハーネスという概念を提示する（エンジニアにデザインハーネスを）。AIへの委任には「offloading（判断は自分が持ち、作り方だけAIに渡す）」と「surrender（AIの出力がそのまま出力になる）」の分岐があり、デザインハーネスはoffloadingを実現するものだと位置づける。「“Done” is a claim, not proof（完了しましたは自己申告であり証拠ではない）」という指摘は、ハーネスが成果物を後から採点するのではなく、そこに至る手順・観察・判定・検証のループを事前に規定するものだ、という点を象徴する。

Agent Skillsによるポータブル化

Gotaは、Agent Skillsが2025年10月にAnthropicから発表され、2025年12月18日にOpen Standardとして公開されたことで、Codex・GitHub Copilot Agent・Cursor等が続々対応した経緯を紹介する（Agent Skillsがハーネスの垣根を超える日）。SKILL.md1枚（YAML Frontmatterにname・description）だけで「手順」と「コンテキスト」をエージェント非依存でポータブルに渡せる点が核心であり、Progressive Disclosure（検出→アクティベーション→実行）によりコンテキストを段階的に開示する。特定エージェントへのロックインを避けるため「ハーネスのカスタマイズより先にSkillsで構築する」というリセット提案がなされる一方、Supply Chain Attack・Indirect Prompt Injection・Shadow AIといった特有のセキュリティリスクにも注意が促されている。

マネージド実行基盤（Managed Agents）との違い

西岡は、ハーネス設計の次の段階として「ハーネスを載せるホステッドサービス」＝Managed Agents（meta-harness）を位置づける（ハーネスエンジニアリング入門）。r-kagayaも、モデル・モデルプロバイダーの進化がハーネスの一部を陳腐化させうる可能性に言及し、「モデルやマネージドサービスの進化によって消えづらいレイヤーに投資する」ことを今後の方針として掲げる（ハーネスエンジニアリングにどう向き合うか）。外山は、汎用harnessは構造的に「行政（実行）」までしか担えず、統治（立法・司法）は自社で作るしかないと明確に線引きする（2.4節参照）。

ハーネス自体がプロダクトになる：HaaS

池上涼平は、「AIエージェント＝Model（賢さ）＋Harness（環境）」という等式のもと、モデル自体は汎用LLMを使いながら、周辺のコンテキスト・ツール・UI・安全性を売るHaaS（Harness as a Service）という発展的視点を提示する（新規プロダクトを高速で生み出すハーネスエンジニアリング）。

Self-improve／メタハーネスへの萌芽

r-kagayaは、決定論的ステップとエージェント的自由を組み合わせたハイブリッドオーケストレーション（Claude Agent SDK＋TypeScript）を構築し、Braintrustへのログ送信による定量評価基盤の整備、エラー履歴のメモリ化など、ハーネス自体を自動改善する「Self-improve」に挑戦している（ハーネスエンジニアリングにどう向き合うか）。将来的な方向性として、LLM自身が制約を発見しハーネスを生成・改善するMeta-Harness（Stanford）やAutoHarness（Google DeepMind）にも言及されている。

今後の展望として押さえておきたい論点（総まとめ8章より）
  • ハーネスだけでは「人間が律速点である」という構造（HITL）は変わらない。真に問うべきは「AI実行をどう統治するか」（外山）。
  • 評価・計測（Eval）への投資不足。「ハーネスは効いているかわからない」という難しさ（r-kagaya）。Four Keys／SPACE指標の改善とは別問題（渋谷優介）。
  • ハーネスの対象範囲は開発・テスト工程に偏りがちで、要件定義・運用は一般にスコープ外（渋谷優介）。
  • 「作る力」から「見極める力」への価値シフト。ハーネスで開発が速くなった先に、何を作るべきかの判断が新たなボトルネックになる（鈴木和真、tame）。
  • 汎用ハーネスと自社ハーネスの境界線。許容できる失敗の幅が狭い領域では、他社の運用をそのまま輸入できない（神谷亮平、吉田真吾の「守破離」）。
  • 定義の対立（4.1節）は今後も解消されない可能性が高い（KINOPEE）。
