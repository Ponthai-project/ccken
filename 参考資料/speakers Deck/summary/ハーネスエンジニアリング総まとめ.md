## 1. 結論：ハーネスとは何なのか

**結論を一言で言えば、ハーネスとは「AIモデル単体ではできないことを、モデルの外側の仕組み（コンテキスト・制約・検証・観測性）で補い、AIを安全かつ自律的に走らせるための環境設計」である。** そしてハーネスエンジニアリングとは、その環境を意図的に設計・運用・改善し続ける営みを指す。

この結論は、ほぼ全ての資料に共通して現れる比喩によって裏づけられる。Claude Codeの開発者Boris Chernyの言葉として紹介される「モデルを直接使うことはできない。ハーネスを使う必要がある。Claudeは馬で、鞍のようなものが要る」（Claude Codeとハーネスについて考えてみる／Oikon）は象徴的で、西岡賢一郎も同じ「馬具」のイメージで「モデルを支え、暴走させずに走らせる周辺装備」と説明する（ハーネスエンジニアリング入門／西岡賢一郎）。非エンジニア向けの超入門資料でも「野生の馬（＝賢いが暴れるAI）に、鞍・手綱・ルートを与えて目的地まで安全に走らせる仕事」として同じ比喩が採用されており（ハーネスエンジニアリング解説／clshinji）、「AIの性能×ハーネス＝業務インパクト」という定式で結ばれている。馬具の比喩は、ハーネスが単なる「抑制装置」ではなく「能力を最大限引き出すための装具」であるという含意を共有している点が重要である（Python開発環境にハーネス適用を検討する／yuuuka、ハーネスエンジニアリング入門／西岡賢一郎）。

より技術的な定義としては、次のようなものが挙げられる。

- 西岡：ハーネス＝「Claudeを呼ぶループ＋ツール呼び出しを実インフラに繋ぐ仕組み」であり、本質は「モデル単独ではできないことの“仮定”を符号化したもの」（ハーネスエンジニアリング入門）。
- 渋谷優介：複数文献（Biderman、Xu、Zhang、Bui、Anthropic、LangChain）を横断整理した上で「LLMを効果的に動作させるための仕組み全般（ハーネス）を設計・開発する行為」（ハーネスエンジニアリングの概要と設計思想）。
- 梶川琢馬：「自律的なAI Agentの実現のために、LLMモデルの性能を最大限引き出すための環境を設計すること」（実践ハーネスエンジニアリング）。
- 神谷亮平：語源どおり「体や物を安全に固定し、力を効率よく伝えるためのベルト状の装具」から転じて「AIを安全に制御し、能力を効率よく発揮させるための環境整備活動」（ハーネスエンジニアリング×AI適応開発）。
- LangChain（Vivek Trivedy）／多数の資料が引用：**Agent = Model + Harness**、あるいは "If you're not the model, you're the harness."（ハーネスエンジニアリングとは？／KINOPEE、ほか多数）。

一方で、**論者によって「どこまでをハーネスに含めるか」の力点は明確に割れている**。これは無理に一致させるべきではなく、資料横断で見えてきた最大の論点でもある。

- **広義派**（Chase＝LangChain、Chawla）：モデル以外の全てがハーネス。ツール実行・メモリ・サブエージェント・ガードレール・オーケストレーションまで包括し、"コンテキスト設計はハーネスの一構成要素" と位置づける（ハーネスエンジニアリングとは？／KINOPEE）。Gamoも「プロンプトエンジニアリング⊂コンテキストエンジニアリング⊂ハーネスエンジニアリング」という個人的整理を示す（コンテキスト/ハーネスエンジニアリングの現在）。
- **狭義派**（Martin Fowler／Birgitta Böckeler）：逆に「コンテキストの中の特殊形」としてハーネスを位置づけ、feedforward（guides）とfeedback（sensors）の制御理論としてハーネスを扱う（ハーネスエンジニアリングとは？／KINOPEE、Python開発環境にハーネス適用を検討する／yuuuka）。すなわち **コンテキスト⊃ハーネス** であり、広義派とは包含関係が逆転する。
- **運用哲学派**（Mitchell Hashimoto）：ハーネスは設計物というより「失敗するたびにAGENTS.mdへルールを1行足し、同じミスを二度とさせない」累積的な運用プロセスそのもの（ハーネスエンジニアリングとは？／KINOPEE、Python開発環境にハーネス適用を検討する／yuuuka）。
- **スケール駆動派**（OpenAI／Ryan Lopopolo）：3人・5ヶ月・100万行・ゼロ手書きという規模の事例を起点に、「プロンプトでもコンテキストでも解けない、多段ステップ・自律実行・並列特有の故障モードを解くもの」という動機づけで語る（ハーネスエンジニアリングとは？／KINOPEE、AI駆動開発が変える、大規模開発の前提／外山英幸）。

Oikonは、この揺れを実務的に「4層構造」に整理している：Prompt Eng.（指示を作り込む）→ Context Eng.（見せる情報を管理）→ **Harness Eng.（モデルを動かす仕組み全体）** → Loop Eng.（ループ自体を書く）。この整理では「Claude Code自体がハーネス」であり、その内部に①モデル ②内部ハーネス（エージェントループ・ツール・サンドボックス）③外部ハーネス（CLAUDE.md・スキル・hooks・スラッシュコマンド・MCP・サブエージェント）の3層があるとする（Claude Codeとハーネスについて考えてみる／Oikon）。西岡も類似の進化アークとして「Prompt Engineering（〜2024）→ Context Engineering（2025/9）→ Harness設計（2025/11）→ Managed Agents／meta-harness（2026/4）」という段階整理を示している（ハーネスエンジニアリング入門）。

**要するに、定義自体に唯一の正解はない。** KINOPEEが結論づける通り「言葉の定義は重要ではない。大切なのは期待した結果を得ること」（ハーネスエンジニアリングとは？）であり、yuuukaも各定義を「狭義⇔広義」「ユーザーハーネス⇔ビルダーハーネス」の2軸整理に落とし込んでいる（Python開発環境にハーネス適用を検討する）。本ドキュメントも以降、この揺れを前提に、実務上ほぼ共通して挙がる構成要素・原則・実践パターンを軸に統合する。

## 2. ハーネスを構成するもの

定義は割れていても、「ハーネスが何でできているか」を挙げると、資料横断でかなりの部分が一致する。以下、静的な構成要素（レイヤー）と、それらを動かす動的な仕組み（ループ・統治構造）に分けて整理する。

### 2.1 レイヤー別の構成要素

| レイヤー | 目的（何のためにあるか） | 代表的な実装・ファイル | 主な言及資料 |
|---|---|---|---|
| **ルール／原則層** | AIの行動原則・役割・禁止事項を事前に固定する（feedforward） | CLAUDE.md / AGENTS.md / copilot-instructions.md / Kiro Steering | 多数（Oikon、tame、平野孝明、gota、yuuuka 他） |
| **コンテキスト・メモリ層** | セッションを跨いだ記憶、局所文脈の分離によりトークンを節約し推論精度を保つ | MEMORY.md、agents/*.md（局所文脈）、段階的開示 | AIエージェント時代のハーネスエンジニアリング Claude Code実装編／ため、コンテキスト/ハーネスエンジニアリングの現在／Gamo |
| **スキル・コマンド層** | 手順とドメイン知識をポータブルにパッケージ化し、必要な時だけ段階的にロードする | Agent Skills（SKILL.md）、Slash Commands、Kiro Agent Hooks | Agent Skillsがハーネスの垣根を超える日／Gota、スキルファーストで作るAIの自走環境／坂上晴信 |
| **エージェント・MCP層** | 責務を分離し、外部ツール・データソースに安全に接続する | Subagents、MCP、Planner/Builder/Evaluator等の役割分担 | 実践ハーネスエンジニアリング／梶川琢馬、beyond-spec-usable／gota |
| **フック・ガードレール層** | ツール実行の前後を決定論的にブロック／検証する | PreToolUse / PostToolUse / Stop hook、permissions allow/deny | ハーネスエンジニアリング入門／西岡賢一郎、ため、平野孝明 |
| **アーキテクチャ制約層** | 「そもそも間違った書き方ができない」設計にする（構造で守る） | 型システム、依存方向の強制（deptrac等）、レイヤードアーキテクチャ | 実践ハーネスエンジニアリング／梶川琢馬、ハーネスエンジニアリング×AI適応開発／神谷亮平 |
| **検証・評価（フィードバック）層** | 出力の正しさ・使えるかを機械的／多角的に確認する | lint／型チェック／テスト／E2E／LLM judge／UXレビュー・uxaudit | beyond-spec-usable／gota、AIエージェント時代のハーネスエンジニアリングとは／ため |
| **観測性・監査層** | 何が起きたかを後から追跡・再現できるようにする | audit.log、report.md、events.jsonl、Braintrust連携 | ハーネスエンジニアリング入門／西岡賢一郎、実践ハーネスエンジニアリング～ステアリングループ／nrs、ハーネスエンジニアリングにどう向き合うか／r-kagaya |
| **学習・自己改善層** | 失敗や気づきを仕組みに書き戻し、ハーネス自体を成長させる | rules更新、feedback_log.md、パターンDB、Self-Refinement | 新規プロダクトを高速で生み出すハーネスエンジニアリング／池上涼平、ハーネスエンジニアリングにどう向き合うか／r-kagaya |

この整理は資料ごとに呼び方が異なるだけで骨格は近い。たとえばtame（為安圭介）は「コンテキスト管理／アーキテクチャ制約とガードレール／品質評価と多角的フィードバック」の**3本柱**でハーネスエンジニアリングを説明し（AIエージェント時代のハーネスエンジニアリングとは）、続編ではそれを**5層アーキテクチャ**（ルール層／コンテキスト層／フック層／コマンド・スキル層／エージェント・MCP層）というファイル群にまで具体化している（AIエージェント時代のハーネスエンジニアリング Claude Code実装編）。梶川琢馬も「コンテキスト設計／アーキテクチャ制約とガードレール／品質評価ループ」という同型の3本柱を示す（実践ハーネスエンジニアリング）。池上涼平は「コンテキスト／アクション／フィードバック／運用」の4設計軸として整理する（新規プロダクトを高速で生み出すハーネスエンジニアリング）。

### 2.2 動的な仕組み：フィードフォワードとフィードバックのループ

構成要素を並べるだけでは「ハーネスが生きた仕組みであること」が見えない。nrs（成瀬允宣）とBirgitta Böckelerの整理は、この動的側面を最もよく捉えている（実践ハーネスエンジニアリング～ステアリングループを実例から読み解く／nrs、Python開発環境にハーネス適用を検討する／yuuuka経由でBöckeler紹介）。

- **フィードフォワード（動く前に方向づける）**：誰として何を目的に動くか明示する／背景知識と完了条件を事前に渡す／使える手段（ツール・権限）を必要な範囲に絞る。
- **フィードバック（動いた後に自己修正を促す）**：出力を検査しやすい形で受け取る／異なる観点から複数回チェックする／改善が停滞したら別の手立てに切り替える。

この両輪を回し続けるのが**ステアリングループ**である。「フィードフォワード（事前の方向づけ）→エージェント実行→フィードバック（事後の検査）→結果→ハーネスを調整」というループを継続的に磨き続けることこそがハーネスエンジニアリングの本体だとnrsは述べ、TAKT（OSS）での4つの実例（ファセット更新／ワークフロー更新／ループモニター追加／タスク分解の導入）を通じて、気づきをログとレポートで裏づけながら`workflow.yaml`に書き戻すプロセスを実演している。同じ枠組みはBöckelerの提唱する**Computational（決定論的：lint・型チェック・テスト）vs Inferential（確率論的：LLMレビュー・AIジャッジ）**という軸とも対応し、フィードフォワード／フィードバックの各要素がどちらの性質を持つかを整理できる（yuuuka）。

gota（beyond-spec-usable）が実装した4subagentハーネス（Planner→Builder→Evaluator→UX Reviewer）も、この構造を体現した実例である。Plannerが検証シナリオ（WHAT）を先に切り、Builderが縦に一本コアフローを通し、Evaluator・UX Reviewerが多軸スコアで合否判定する——これは「フィードフォワードで方向づけ、フィードバックで検証し直す」ループそのものである。

### 2.3 統治の視点：立法・司法・行政（三権分立モデル）

外山英幸（ビズリーチ）は、ハーネスを構成する仕組みを「誰が何を決めるか」という統治の視点で捉え直す独自のモデルを提示している（AI駆動開発が変える、大規模開発の前提）。

1. **立法AI（ルール）**：何が正しいかを定義する（ルールのSSOT）。
2. **司法AI（検証）**：適合しているかを裁く（contracts / guards）。
3. **行政AI（実行）**：実際に実行する（workflow / agent）。

外山の重要な指摘は、**「汎用ハーネスが担えるのは行政（実行環境・文脈管理・検証ループ）までであり、立法（プロダクトドメイン依存のルール）と司法（フェーズ依存のチェック）は自社で作るしかない」**という点である。この視点は、次章「なぜ必要か」「ハーネスだけで足りるか」の議論に直結する。

## 3. なぜハーネスが必要になったのか

背景にある課題意識は、複数資料でほぼ共通して次の3つに収斂する。

**(1) 人間の確認・レビューがボトルネック化する。** AIエージェントはコードを大量かつ高速に生成できるようになったが、人間の理解・判断・QAの速度は急には増えない。「これからのSDLCの希少資源は人間の時間と注意力」であり、ボトルネックは「書くこと」から「読むこと・確かめること」へ移る（ハーネスエンジニアリング×AI適応開発／神谷亮平、OpenAI「Harness engineering」の紹介経由）。tameも同様に「AIの出力量の急増」に対し「人間の確認コストがそれを上回る恐れ」を指摘し、これに対処する手法としてハーネスエンジニアリングを位置づける（AIエージェント時代のハーネスエンジニアリングとは）。外山はこれを「HITL（Human in the Loop：人間がフロー内部にいて律速点になる）」構造の限界として整理し、ハーネスは開発速度を上げても、それだけでは**HITLのまま速くなるだけ**で、「AI実行をどう統治するか」という問いには答えられないと指摘する（AI駆動開発が変える、大規模開発の前提）。

**(2) 「動く」だけでは価値に到達しない。** 仕様通りに動くコードを書かせる技術は急速に進歩したが、「動く」と「使える」は別問題である。gotaは「触って目的が達成できる／コア体験がはっきり／初見で始められる」という基準を満たさない「動くけど使えない」の4類型（伝わらない・ぼやける・見つからない・始まらない）を挙げ、Anthropic社内でも出荷物の7割以上がPRDなしで、「使えるかは触るまで分からない」という現場感覚を紹介する（beyond-spec-usable／gota）。速度・品質・量の全てを人手で担保しようとすると必ずどれかが落ちるため、人手を極限まで減らして使えるものを速く出す自律ハーネスが要請される。

**(3) 属人性・再現性の欠如。** クラスメソッドの事例では、Vibe Codingにおける「The 70% problem」（非エンジニアは70%まで速く到達できるが残り30%が進まない）や、ハーバード大学・BCGの調査による「居眠り運転」現象（高精度AIへの過信でチェックが手抜きになり成績が下がる）が紹介され、AIの出力をそのまま信じることのリスクが指摘される（AI駆動で進化する開発プロセス／佐藤智樹）。神谷亮平の「7万行規模のSaaSを2人月・従来比50〜56倍」という実績も、発表後に「あなただからできたのでは」と問われたことで属人性に気づき、「誰でも・どんな組織でも再現できる仕組み」への転換を迫られたと述べている（ハーネスエンジニアリング×AI適応開発）。r-kagayaも、同じデザインシステムを参照していてもコーディングエージェントや人によって選ぶコンポーネントがずれる「人ガチャ・AIガチャ」現象を課題として挙げる（エンジニアにデザインハーネスを）。

これらの背景により、ハーネスエンジニアリングは「AIをどう使うか」の技術論を超えて、**「人間の介入をどこに、どれだけ残すか」を設計する組織的な問い**へと接続していく（後述8章）。

## 4. 論者による定義の違い

1章で述べた広義／狭義の対立に加え、KINOPEEは「同じ『ハーネス』を語っても、論者が原器にしている実装で重心が違う」として、日本語圏の議論を読み解くための**5つの流派**を提示している（ハーネスエンジニアリングとは？）。

| 流派 | 代表者 | 中核主張 |
|---|---|---|
| Chase派（広義・分類論） | Harrison Chase（LangChain） | モデル以外のすべてがハーネス。"If you're not the model, you're the harness." |
| Hashimoto派（運用哲学） | Mitchell Hashimoto（HashiCorp） | 失敗したら環境を再設計せよ。AGENTS.mdへの累積的なルール追加そのものがハーネスエンジニアリング |
| Fowler派（制御理論） | Martin Fowler／Böckeler | コンテキスト設計の特殊形（コンテキスト⊃ハーネス）。guides（feedforward）とsensors（feedback）の両輪 |
| Chawla派（同心円モデル） | Avi Chawla | プロンプト⊂コンテキスト⊂ハーネスの三重同心円。LLM=CPU／コンテキスト=RAM／ハーネス=OSという比喩 |
| Codex派（スケール駆動） | Ryan Lopopolo（OpenAI Codex） | "Agents aren't hard; the Harness is hard." スケールで壊れる問題への回答という動機づけ |

yuuukaも独立に「①Hashimoto（ユーザー・フィードバックループ）②Ryan Lopopolo（ビルダー・環境全体設計）③Prithvi Rajasekaran＝Anthropic（ビルダー・Planner-Generator-Evaluator）④Vivek Trivedy＝LangChain（ビルダー・Agent=Model+Harness）⑤Birgitta Böckeler（ユーザー・Guides/Sensors）⑥nrs（ユーザー／ビルダー）」という6提唱者を整理し、Böckelerの「Guides/Sensors」の枠組みを最も体系的で入門に適すると評価している（Python開発環境にハーネス適用を検討する）。

この分類の実務的な意味は、**「ユーザーハーネス」（コーディングエージェントを使う側が組む文脈・制約・レビュー）と「ビルダーハーネス」（エージェント基盤そのものを作る側の設計）を区別すること**にある。r-kagayaやGota、tame、平野孝明など、本総まとめが多く参照する日本の実践知はほぼ全てユーザーハーネス側の議論であり、Anthropic・OpenAI・LangChainの発信はビルダー側の議論であることが多い。この2つを混同すると「なぜ自社でOpenAIと同じ運用ができないのか」という誤った比較に陥りやすい（後述8章、外山・神谷亮平の指摘とも符合）。

## 5. 設計思想・原則

複数資料に共通して現れる設計原則を統合すると、大きく4つにまとめられる。

### 5.1 モデル進化への追従：足すより剥がす

西岡賢一郎は「ハーネスはモデル進化で陳腐化する」（"Harnesses encode assumptions that go stale as models improve."）ことを明確な原則として掲げ、**最小権限・可観測・失敗前提・引き算（剥がす運用）**の4原則を提示する（ハーネスエンジニアリング入門）。半年ごとにhooksを読み直し不要なものを消す運用を推奨し、「個別のhookは陳腐化するが『外側で支える』設計スキル自体は陳腐化しない」とまとめる。Gotaの「目的ファーストのハーネス設計」も同じ問題意識から出発しており、「足せても削れず肥大化する」ことが実務のあらゆる段階（ハーネスがない／作り始めた／運用中）で共通の症状として現れると指摘する。

### 5.2 目的ファースト・優先順位づけ

Gotaは、ハーネスを足す前に踏むべき4ステップを提示する（目的ファーストのハーネス設計）。

1. 業務目的を言語化する（何を守る／守らない／人が介入するか）
2. 構造で守れないか問う（設計／型／依存方向 ＞ scaffolding ＞ test ＞ 指示、の優先順位）
3. 指示で補う（CLAUDE.md／skill／review agentは最小限に）
4. 削れる状態で運用する（小さく始め、失敗駆動で足し、目的で削る）

この「構造＞テスト＞指示」という優先順位は、渋谷優介の設計思想（Principles・Verification・Hooksの設定＝自由度を下げる施策から着手し、Skillsの設定・パラメータチューニングは後回しにする方が難易度が低い）とも整合する（ハーネスエンジニアリングの概要と設計思想）。Oikonの「まず内部ハーネス＋モデルの素の実力を使い倒し、足りない分だけ外部ハーネスを小さくポータブルに足す」という原則も同じ方向を向く（Claude Codeとハーネスについて考えてみる）。

### 5.3 汎用化・密結合の罠（失敗事例からの教訓）

Gotaは自身の失敗を赤裸々に共有している（ハーネスエンジニアリングをやりすぎた話）。「様々なソフトウェア開発に対応できるハーネス」を目指して範囲を拡張した結果、テンプレート・hook・verify・tests・policyが同時に絡み合う密結合に陥り、Script 25,000行・Markdown 10,000行・Hooks 1,500行にまで肥大化。1回の全体テストに1時間かかり、修正の成否評価すら不能になった。並列実装が生む「境界のドリフト」「仕様のドリフト」「依存関係のドリフト」という3つの異常も指摘されている。教訓は「ハーネスでなんでもやろうとすると、エージェントを制御するための仕組み自体が制御不能になる」ことであり、目的ごとに小さく分ける（Planning／Execution／Verification／Adaptation Agentの分離）ことが再発防止策として示される。

### 5.4 統治・責任構造としての設計

外山は、ハーネス設計の議論を「AI実行の統治（Governance）」の問題として捉え直し、ADR・Wiki等の「書かれている」だけの状態と、Agentが参照し違反すれば止まる「効いている」状態を峻別すべきだと主張する（AI駆動開発が変える、大規模開発の前提）。神谷亮平も同様に、OpenAIの短時間マージ運用は「ハーネスの作り込み」だけでなく「適切な妥協点を許容する組織的判断」があってはじめて成立するとし、**自社で決めるべき3つの問い**（品質の保証範囲・人間の関与の線引き・実行可能な環境づくり）を、仕組み・プロセス・体制を一体で見直す「責任構造の再設計」として提示する（ハーネスエンジニアリング×AI適応開発）。

## 6. 実践パターン・具体的な進め方

### 6.1 ファイル・ツール横断の実装対応表

tameは、ハーネスの成果物が具体的な「ファイル」であることを強調し、5層をそれぞれ対話で生成できると説明する（AIエージェント時代のハーネスエンジニアリング Claude Code実装編）。平野孝明は、これをClaude CodeとGitHub Copilotの双方に対応づけている（Copilot／Claude Code両方で使えるスキル・ハーネスの紹介）。

| 種別 | Claude Code | GitHub Copilot |
|---|---|---|
| ルール定義 | CLAUDE.md | copilot-instructions.md |
| 追加ルール | .claude/rules | .github/instructions |
| スキル | .claude/skills | .github/skills |
| 設定・フック | .claude/settings.json | .github/hooks |
| コマンド | .claude/commands | .github/prompts |

平野は、ハーネスを「初期コンテキスト型（CLAUDE.md・skills・agentsで最初に方向を示す）」と「遅延ロード型（rules・skills・hooksでズレを事後修正する）」の2種に分け、**共通化・汎用化を狙うなら遅延ロード型の方が難易度が低い**と結論づける。汎用的な共通ハーネスの例として、初期コンテキストを改善する「grill-me / grill-with-docs」（意思決定ツリーの各分岐を人にヒアリングして解決するスキル）と、遅延ロードコンテキストを改善する「PreToolUse hooks + additional context」（危険コマンドのdenyと同時にコンテキストを注入する仕組み）が紹介されている。

### 6.2 多層防御によるレビュー負荷の低減

複数の実務者が、コンテキスト→型→静的解析→テスト→AIレビュー→人間レビューという**多層防御**の構図を共有している。

- 梶川琢馬：型システム（ValueObject／不変条件）→静的解析（deptrac／phpstan／ESLint）→テスト（ユニット＋E2E＋VRT）→AIレビュー→人間レビューの順に問題を自動検出し、CIが通っただけで「完了」と宣言させずChrome DevTools MCPで実際のブラウザ動作まで確認させる（実践ハーネスエンジニアリング）。
- 池上涼平：事前に書かせない（CLAUDE.md／rules）→書きにくくする（権限分離・型）→動かして検出（実DBテスト）→マージ前に止める（CI）→botに1次レビュー（CodeRabbit）→人間レビュー、という6層。「同じ指摘を2度繰り返さない」ためレビュー指摘をrulesへ随時書き戻すskillを運用し、Row-Level Securityでテナント分離をDBレベルで宣言的に管理する（新規プロダクトを高速で生み出すハーネスエンジニアリング）。
- 神谷亮平：generated/（自動生成・編集禁止）とcustom/（AIがビジネスロジックのみ書く・編集許可）を分離し、宣言的SQLスキーマからのコード生成パイプラインで「単一のsource of truth」を保つ（ハーネスエンジニアリング×AI適応開発）。

### 6.3 ハーネステンプレートとステアリングループの実演

nrsのTAKT（OSS）は、ハーネスを「ワークフロー（ステップの有限状態機械）」と「ファセット（プロンプトの部品：Persona／Knowledge／Instruction／Output Contract／Policy）」の組み合わせとして表現し、`workflow.yaml`とファセットに改善を書き戻すことでハーネステンプレートとして成熟させていく実例を示す。ログ（events.jsonl）とレポート（各ステップのreport.md）が原因追跡を支え、「追えるから改善に戻せる。改善に戻せるからループが回る」（実践ハーネスエンジニアリング～ステアリングループを実例から読み解く）。

gotaのbeyond-spec-usableでは、Credo（5原則：Core First・Wire Before You Decorate・No Dead Code・The Spec Is Law・Built to Grow）をSkillとして全subagentに注入することで、複数エージェントに一貫した振る舞い方針を持たせている。さらに`uxaudit`（Claude Code Plugin）により、静的スキャン→ブラウザ計測→画像判定LLM→ジャーニー判定という段階的な検証で「触らないと分からない層」（UX）まで自動評価を広げている。

### 6.4 仕様駆動開発とハーネスの接続

Gota（旧「ClaudeCodeで仕様駆動開発を実現するために！」）は、AWS Kiroの仕様駆動開発（Specs／Agent Hooks／Steering）をClaude Codeで再現する取り組みを紹介し、Steering＝CLAUDE.md、Agent Hooks＝Claude Code Hooksという対応関係を示した上で、「Specs/Steeringをしっかり再現できればKiroの仕様駆動開発を実現できる」として`claude-code-spec`を公開している。渋谷優介は、仕様駆動開発を上流工程（要望→要求→要件）にまで拡張し、「What」だけでなく「Why・Why not」までコンテキストに含めることの重要性、および「聞くことリスト.md」による質問型エージェント設計を提案する（仕様駆動開発を実現する上流工程におけるAIエージェント活用）。吉田真吾は、AWSのAI-DLC（Inception／Construction／Operations）をチーム開発の文脈に接続し、`aidlc-workflows`のユーザーハーネスが不明点を質問表化して曖昧なまま進むのを防ぐと述べる（チームで実現する高品質・安全なAI駆動開発実践）。

### 6.5 スキルファーストという別解

坂上晴信（TOKIUM）は、ルール・規約を先に整備する「業務PJ型」に対し、個人プロジェクトでの検証を通じて「スキルファースト」という別のアプローチを示す（スキルファーストで作るAIの自走環境）。やりたいことを全部書き出し、それが全て「スキルの組み合わせ」で実行できるようにハーネスを構想し、PR1件ごとに振り返りスキルでスキル・規約を自己改善させた結果、1,753行のMarkdownと200+のハーネス関連ドキュメント・約70件のPRがほぼ指示なしで生成された。結論として「枯れた技術が相手ならルール・規約ファースト、日々進化し非定型入力もカバーできるAIならスキルファーストが有効」という使い分けが提示されている。

### 6.6 Python環境への具体適用

yuuukaは、フィードフォワード（`uv venv`による環境隔離、`uv.lock`による依存固定、AGENTS.md）とフィードバック（`ruff`・`pyright`・`pytest`・AIコードレビュー）を組み合わせた具体的なPython環境ハーネスを提案し、自プロジェクトの棚卸しでは「ガイド側は整備済みだがPermissionsのallow/denyやSkills/Subagentsなどセンサー側が手薄」という結果を共有している（Python開発環境にハーネス適用を検討する）。

## 7. 発展的トピック

### 7.1 デザインハーネス

r-kagayaは、デザイナー不在のスタートアップでエンジニア＋コーディングエージェントがUIデザインを担う際の課題（「それっぽさ」止まりの判断力欠如、最低基準の不在、人ガチャ・AIガチャ）に対し、**デザインハーネス**という概念を提示する（エンジニアにデザインハーネスを）。AIへの委任には「offloading（判断は自分が持ち、作り方だけAIに渡す）」と「surrender（AIの出力がそのまま出力になる）」の分岐があり、デザインハーネスはoffloadingを実現するものだと位置づける。「“Done” is a claim, not proof（完了しましたは自己申告であり証拠ではない）」という指摘は特に重要で、ハーネスは成果物を後から採点するのではなく、そこに至る手順・観察・判定・検証のループを事前に規定するものだと強調される。ユーザーハーネス同様「導く／検知する」×「推論的／計算的」の4象限で整理でき、計算的センサー（型エラー・lintでの契約違反ブロック）が有効だとする。

### 7.2 Agent Skillsによるポータブル化

Gotaは、Agent Skillsが2025年10月にAnthropicから発表され、2025年12月18日にOpen Standardとして公開されたことで、Codex・GitHub Copilot Agent・Cursor等が続々対応した経緯を紹介する（Agent Skillsがハーネスの垣根を超える日）。SKILL.md1枚（YAML Frontmatterに`name`・`description`）だけで「手順」と「コンテキスト」をエージェント非依存でポータブルに渡せる点が核心であり、Progressive Disclosure（検出→アクティベーション→実行）によりコンテキストを段階的に開示する。特定エージェントへのロックインを避けるため「ハーネスのカスタマイズより先にSkillsで構築する」というリセット提案がなされる一方、Supply Chain Attack・Indirect Prompt Injection・Shadow AIといった特有のセキュリティリスクにも注意が促されている。r-kagayaのデザインハーネスも「デザインシステムをスキルに組み込むことで、コーディングエージェント非依存で同じループを辿れる」ことを狙っており、Agent Skillsのポータブル性という思想を共有する。

### 7.3 マネージド実行基盤（Managed Agents）との違い

西岡は、ハーネス設計の次の段階として「ハーネスを載せるホステッドサービス」＝Managed Agents（meta-harness）を位置づけている（ハーネスエンジニアリング入門）。r-kagayaも、モデル・モデルプロバイダーの進化により「Claude Managed Agentsのような登場」がハーネスの一部を陳腐化させうる可能性に言及し（ハーネスエンジニアリングにどう向き合うか）、「モデルやマネージドサービスの進化によって消えづらいレイヤーに投資する」ことを今後の方針として掲げる。外山は、Anthropic・OpenAI・Cursor等が提供する汎用harnessは構造的に「行政（実行）」までしか担えず、統治（立法・司法）は自社で作るしかないと明確に線引きする（AI駆動開発が変える、大規模開発の前提）。

### 7.4 ハーネス自体がプロダクトになる：HaaS

池上涼平は、「AIエージェント＝Model（賢さ）＋Harness（環境）」という等式のもと、モデル自体は汎用LLMを使いながら、周辺のコンテキスト・ツール・UI・安全性を売る**HaaS（Harness as a Service）**という発展的視点を提示する（新規プロダクトを高速で生み出すハーネスエンジニアリング）。Cursor／Copilot／Devin（コーディング）、CodeRabbit（レビュー）、v0／Bolt.new／Lovable（デザイン/プロト）、Harvey／Hebbia（業界特化）、Sierra／Salesforce Agentforce（業務自動化）が例に挙がり、「これからのAIプロダクトの正体は、ハーネス（かも？）」と結ばれる。

### 7.5 Self-improve／メタハーネスへの萌芽

r-kagayaは、決定論的ステップとエージェント的自由を組み合わせたハイブリッドオーケストレーション（Claude Agent SDK＋TypeScript）を構築し、Braintrustへのログ送信による定量評価基盤の整備、エラー履歴のメモリ化など、ハーネス自体を自動改善する「Self-improve」に挑戦している（ハーネスエンジニアリングにどう向き合うか）。将来的な方向性として、LLM自身が制約を発見しハーネスを生成・改善するMeta-Harness（Stanford）やAutoHarness（Google DeepMind）にも言及されている。

## 8. 論点・今後の展望

**(1) ハーネスだけでは「人間が律速点である」という構造は変わらない。** 外山は、ハーネスが開発速度を劇的に上げても、それは「HITL（Human In The Loop）のまま速くなっただけ」であり、真に問うべきは「AI実行をどう統治するか（Governance of AI Execution）」だと主張する。統治には立法（ルール）・司法（検証）・行政（実行）の三権分立と、それらを機械可読に接続するAuthority Provenance Graph／Specification Provenance Graphが必要であり、これは汎用ハーネスの範囲外（プロダクト・組織依存）だとされる（AI駆動開発が変える、大規模開発の前提）。神谷亮平の「責任構造の再設計」（品質の保証範囲・人間の関与の線引き・実行可能な環境づくりという3つの問い）も同じ論点への異なるアプローチであり、両者は「ハーネス＝必要条件、統治／責任構造＝十分条件」という点で符合する。

**(2) 評価・計測（Eval）への投資不足。** r-kagayaは「ハーネスは効いているかわからない」という3つの難しさの一つを挙げ、ルールファイル・スキル整備だけに閉じた「個人のうまい使い方」に留まらず、評価・計測基盤（Braintrust等）まで含めたプロセス設計が必要だと指摘する（ハーネスエンジニアリングにどう向き合うか）。渋谷優介も、ハーネスエンジニアリングだけでは「Four Keys」（デプロイ頻度・変更リードタイム・変更失敗率・復旧時間）や「SPACE指標」（人の働き方）を改善できるとは限らず、開発・テストが速くなってもデプロイ頻度が上がらなければ価値提供は加速しないと注意を促す（ハーネスエンジニアリングの概要と設計思想）。

**(3) ハーネスの対象範囲は開発・テスト工程に偏っている。** 渋谷優介は、既存のハーネスがSDLC/V字モデルの中でも開発・テスト領域に集中し、要件定義・運用は一般的なスコープ外になりがちだと指摘する。渋谷の別発表（仕様駆動開発を実現する上流工程における〜）は、この隙間を埋めるべく上流工程（要望→要求→要件）への拡張を試みたものであり、外山の「機能以外（セキュリティ・堅牢性・保守品質・インフラ・運用・DevOps）への統治の拡張」という今後の論点とも重なる。

**(4) 「作る力」から「見極める力」への価値シフト。** 鈴木和真（マイベスト）は、AIによって「作る」速度が上がった一方でアウトカムが同じ速度で上がるとは限らない「AI時代のビルドトラップ」を指摘し、エンジニアの価値が「作る力」から「何を作るべきかを見極め、学習速度を上げる力」へ広がっていると述べる（作る力から、見極める力へ）。これはハーネスエンジニアリング自体の議論ではないが、「ハーネスで開発が速くなった先に何を作るべきかという判断が新たなボトルネックになる」という点で、外山・神谷亮平の議論と地続きの問題意識である。tameも同様に、エンジニアの役割が「コードの記述者」から「インヴァイロメントアーキテクト」へ、企業独自の暗黙知をAI可読な環境仕様に変換する「Translation Bridge Skill」がコアスキルになると述べる（AIエージェント時代のハーネスエンジニアリングとは）。

**(5) 汎用ハーネスと自社ハーネスの境界線。** 神谷亮平は、OpenAIの短時間マージ運用のような「見えにくい前提」（組織的な意思決定）を自社にそのまま輸入することの危うさを指摘し、大規模SI・医療・金融・インフラなど「許容できる失敗の幅が狭い」領域では同じ判断が採用しにくいとする。吉田真吾のAI-DLC「守破離」モデル（守：基礎トレーニング／破：チームアグリーメント再構築／離：自社ハーネス開発）も、汎用ハーネスの利用から自社固有のハーネス開発へと段階的に進む道筋を示している。

**(6) 定義の対立は今後も解消されない可能性が高い。** 4章で見た「コンテキスト⊃ハーネス」（Fowler派）と「ハーネス⊃コンテキスト」（Chase／Chawla派）の包含関係の逆転は、概念の成熟が追いついていないことの表れであり、KINOPEEが言うように「日本に紹介する際は、どちらの意味で語られているかを毎回明示する必要がある」状況は当面続くと見られる。
